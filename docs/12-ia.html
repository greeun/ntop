12 · 정보구조(IA) — TUI 레이아웃 계층 + CLI 명령 트리 (ntop)

이 문서는 ntop 인터페이스 정보구조(IA)의 단일 출처다. ntop의 모든 화면(=TUI 뷰/패널/탭/다이얼로그 + CLI 명령 출력)이 어떻게 묶이고 어디서 도달되는지를 화면 ID와 함께 트리로 정리한다.

ntop은 터미널 도구다. 여기서 "화면(screen)"은 웹/GUI 페이지가 아니라 TUI 뷰·패널·탭·모달 다이얼로그 + CLI 서브커맨드 출력을 뜻한다. 따라서 ASCII 레이아웃이 곧 실제 충실도다(spec §4-0).

참조(재정의 0): 기능 ID(F1~F21)·우선순위는 10-prd, 문제(PRB-1~8)는 research-s2(→03 §2), 플로우는 13-user-flows, 키 바인딩의 정규 출처는 21-screen-spec(현재는 research-s3 §7), Config 기본값은 31-erd 가 단일 출처다. 본 문서는 화면 IA를 소유하고 나머지는 참조만 한다(spec §4-3). 화면 ID(V-/TAB-/DLG-/BAR-/CLI-)는 research-s3를 정규로 승계한다.

전제: ntop은 이미 v0.2.0까지 구현된 MIT 오픈소스 도구다(codebase-facts §0). 본 IA는 신규 화면 설계가 아니라 이미 구현된 TUI/CLI를 역설계해 정리한 것이다(창작 0). 모든 화면/색/진입은 codebase-facts(이하 "도시에") §6·§7 또는 실제 소스코드(src/tui/**, src/cli.rs, src/main.rs)에 역추적된다.

용어 풀이(첫 등장): IA(Information Architecture, 정보구조) = 화면·기능이 어떻게 묶이고 이동하는지의 뼈대 / TUI(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI / CLI(Command-Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식 / 패널(panel) = 화면을 나눈 영역 / 탭(tab) = 한 패널 안에서 전환되는 하위 화면 / 모달 다이얼로그(modal dialog) = 떠서 다른 입력을 막고 결정을 받는 작은 창 / 포커스(focus) = 지금 키 입력을 받는 패널 / 크롬(chrome) = 콘텐츠가 아니라 항상 떠 있는 테두리 막대(상단/하단 바) / 플래그(flag) = 명령 뒤에 붙이는 --옵션.

0. 사실 정정 (코드-진실) — 반드시 준수#

일부 선행 표현이 "좌/우 분할" 또는 "목록 NAME/FRAMEWORK 컬럼"으로 적었으나 실제 코드와 불일치다. 본 IA는 아래 코드-진실(src/tui/ui.rs L37–44, process_list.rs + 도시에 §6 정정값)을 정규로 쓴다.

① 레이아웃 = 세로(상/하) 분할 — 좌/우 아님. ui.rs L37–44 Layout::vertical([55%, 45%]):

  • BAR-TOP 상단 상태바(1줄, 크롬)
  • V-LIST 프로세스 목록 위 55%(패널1)
  • V-DETAIL 상세 패널 아래 45%(패널2) — 탭: Info · Log · Net · Env
  • BAR-BOTTOM 하단 키힌트(1줄, 크롬)

② TUI 목록에는 별도 FRAMEWORK 컬럼이 없다. 목록 컬럼(코드 권위, process_list.rs): [✓/health] PID·NAME·PORT·THR·CPU·MEM·USER·STS·UPTIME — FRAMEWORK 없음. 프레임워크 정체는 ⓐ 상세 Info 탭(TAB-INFO), ⓑ CLI ntop list 테이블 FRAMEWORK 컬럼에서만 별도 열로 노출.

본 문서의 모든 레이아웃·노출 경로는 이 코드-진실을 따른다. (도시에 §6의 "좌/우" 표기는 구식 — research-s3 §0·§10에서 메인테이너 정정 권고로 회부됨.)

정정(NAME 인라인 배지): 위 ②의 "프레임워크는 Info 탭/CLI 테이블에서만"은 별도 FRAMEWORK 컬럼에 한정된 말이다. 실제 목록 NAME 컬럼은 인라인 배지로 프레임워크를 함께 노출한다(next-server (Next.js) / uvicorn [Python], process_list.rs L121·L127). 즉 목록에 별도 FRAMEWORK 컬럼이 없을 뿐, 프레임워크명 자체는 NAME에서도 보인다. (로드맵 40 M2)

1. 이 문서가 답하는 것 (Scope)#

ntop의 인터페이스는 두 진입점으로 갈린다 — 서브커맨드 없이 ntop이면 TUI 기동(run_tui), 서브커맨드가 있으면 CLI 일회성 출력(scan_blocking). 둘 다 동일 코어(ProcessScanner + NetworkInspector)를 공유한다(도시에 §7). 본 문서는:

  • TUI 레이아웃 계층(§3)과 두 진입점을 합친 IA 트리(§4)를 그린다.
  • CLI 명령 트리(§5)를 플래그·출력까지 정리한다.
  • 모든 화면에 ID를 배정하고 노출 경로(어느 키/모드/명령에서 도달) 를 표로 확정한다(§6).
  • IA의 구조적 특성(평평함·얕은 깊이)이 문제(PRB)를 어떻게 구조적으로 줄이는지 분석한다(§7).
추적 사슬에서의 위치(spec §4-2): 문제(research-s2) → 기능 ID(10) → 화면·CLI(12·13) → 데이터(31) → 계약/정책(32·33). 본 문서가 "화면 IA" 마디를 소유한다.

3. TUI 레이아웃 계층 (★ 세로 분할) — I-1#

ntop TUI는 세로 3단이다: 상단바(1줄) / 메인 콘텐츠(가변) / 하단바(1줄). 메인 콘텐츠는 다시 세로(상/하)로 55%/45% 분할된다(ui.rs L37–44 Layout::vertical). 프로세스가 0건이면 메인 전체가 빈/로딩 위젯(V-EMPTY)으로 대체된다. 필터 모드·다이얼로그는 이 위에 겹친다.

계층이 말하는 것: 깊이 2~3단으로 끝난다. 상단바·하단바는 콘텐츠가 아닌 크롬이라 상시 떠 있고, 실제 작업은 V-LIST(식별)→V-DETAIL 4탭(점검)에서 끝난다. 위험한 동작만 모달 한 겹(DLG-*)으로 감싼다. 별도 라우팅·페이지 전환이 없어 PRB-8(GUI↔터미널 왕복)을 구조적으로 제거한다(F20).

4. IA 트리 (TUI + CLI 통합) — I-2#

두 진입점(TUI/CLI)을 합친 전체 IA 트리. 각 노드에 [역할] 색 태그 + 화면 ID + 진입 키 + 연결 기능 ID(10-prd) + PRB(research-s2)를 부기한다. 분기 진입점은 src/main.rs의 clap 파싱.

IA가 말하는 것: 깊이 2~3단의 얕은 트리다. TUI는 메인 한 화면(V-MAIN)에 정보를 모으고 파괴적 동작만 모달 한 겹으로 감싼다. CLI는 TUI와 동일 코어를 재사용하는 평면 5개 명령이다. 별도 라우팅·페이지 전환·설정 마법사·로그인이 없다(§7).

5. CLI 명령 트리 (플래그·필수성·출력) — I-3#

ntop(서브커맨드 없음)은 TUI를 기동하고, 5개 서브커맨드는 일회성 출력을 낸다(도시에 §7, cli.rs clap derive). CLI 출력 스키마(table 컬럼·JSON 필드·CSV 컬럼)의 정규 출처는 32-api-spec이며 본 트리는 요지만 표기한다.

CLI 설계 요지: 평면 5명령 + 단순 플래그. --json/--format csv가 헤드리스 자동화(PRB-7)의 연결점이고, kill의 --tree/--all/--no-confirm이 스크립트 일괄 종료를 가능케 한다. TUI와 같은 코어라 동작 의미가 일치한다(예: graceful→force 에스컬레이션은 둘 다 적용, 트리거 방식만 다름 — §13 참조).

7. IA 특성 분석 — 왜 이렇게 평평한가#

IA의 구조 자체가 ntop의 가치(저마찰·안전)를 떠받친다. 깊이·분기 수를 시각화한다.

 depth 2    주 화면        V-MAIN(=BAR-TOP/BOTTOM·V-LIST·V-DETAIL·V-EMPTY) · CLI-LIST/KILL/INFO/LOG/CONFIG

 ⇒ 최대 깊이 3. 페이지 전환·라우팅 없음. "위험 동작(kill/signal)"만 모달 한 겹 추가.
