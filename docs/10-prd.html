10 · PRD — 기능 목록 / 우선순위 / 추적표 / 비기능 요구 (ntop)

이 문서는 단일 출처(single source)다. ntop의 기능 목록·정규 기능 ID(F1~F21)·우선순위(P0/P1/P2)· 기능↔문제 추적표는 여기서만 확정한다. 하위 문서(11-user-stories·12-ia·13-user-flows· 21-screen-spec·30-functional-spec·40-backlog·41-qa-testcases)는 이 기능 ID를 참조할 뿐 재정의하지 않는다(spec §4-3).

반대로 본 문서는 핵심 가치·MVP 범위·성공지표·차별화는 01-service-plan, 페르소나(P1·P2·P3)는 03-personas, 문제 정의(PRB-1~8)는 research-s2(→03), Config 기본값·구조체 정의는 31-erd 를 단일 출처로 참조하며 재정의하지 않는다. 같은 수치(refresh_interval=3·graceful_timeout=10· MAX_BUFFER_LINES=1000)는 31-erd가 정규값이고 본 문서는 측정 임계값으로만 인용한다.

전제: ntop은 이미 v0.2.0까지 구현된 MIT 오픈소스 도구다(codebase-facts §0). 따라서 본 PRD는 "앞으로 만들 기능 명세"가 아니라 이미 구현된 동작을 역설계해 기능 ID·우선순위·추적으로 정리한 것이다. 모든 기능은 codebase-facts(이하 "도시에")의 실제 코드 동작에서 도출했다(창작 0).

용어 풀이(첫 등장): TUI(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI / CLI(Command-Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식 / 런타임(runtime) = 코드를 실행하는 엔진(Node·Python·Java 등) / 프레임워크(framework) = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등) / graceful 종료(정상 종료) = "정리하고 꺼져라"라고 요청 (SIGTERM), 강제 종료(SIGKILL)와 다름 / 에스컬레이션 = 정상 요청이 안 통하면 강제로 단계 올림 / 트리 종료 = 부모-자식 프로세스를 묶어 함께 종료 / PRB-n = research-s2가 정의한 문제(problem) 번호 / PID = 프로세스 고유 번호 / NFR(Non-Functional Requirement) = 기능 자체가 아니라 성능· 안전·정확성 같은 "품질" 요구 / MVP(Minimum Viable Product) = 가치를 전달하는 최소 제품 범위.

1. PRD가 답하는 것 (Scope of this doc)§

ntop은 "여러 런타임의 서버 프로세스를 의미 인지(semantic awareness) 수준으로 식별·점검·안전 종료" 하는 단일 바이너리 TUI+CLI 도구다(가치·차별화 단일 출처는 01-service-plan). 본 PRD는 그 가치를 떠받치는 구현된 기능 21개에 정규 ID(F1~F21)를 부여하고, 각 기능을 research-s2의 문제(PRB-1~8)로 역추적하며, 우선순위와 비기능 품질 기준을 확정한다.

  • 기능 ID는 추적의 척추다. 문제(01·03) → 기능 ID(본 문서) → 화면·키(12·13·21) → 데이터(31) → 계약/정책(32·33) 사슬에서 본 문서가 "기능 ID" 마디를 소유한다(spec §4-2).
  • 기능은 식별 / 포트·네트워크 / 안전 종료 / 뷰·탐색 / 점검 / 인터페이스·자동화 6개 군으로 묶인다.
  • 우선순위는 가치 루프(스캔→식별→점검→안전 종료)의 척추 여부로 가른다(01 §4·§5 근거).

2. 기능 카탈로그 — 정규 기능 ID (F1~F21)§

이 표가 기능 ID·우선순위의 단일 출처다. 각 행: 기능 ID · 기능명 · 한 줄 동작 · 우선순위 · 연결 PRB · 화면/키 연결(코드-진실) · 도시에 근거 · 잠정 핸들(01/research-s2 추적용). 우선순위 표기: P0=코어(없으면 ntop이 아님) · P1=핵심 보조 · P2=편의. 근거는 §3.

3. 우선순위 근거 (P0/P1/P2)§

가르는 기준 = 가치 루프(스캔→식별→점검→안전 종료, 01 §4)에서 그 기능이 차지하는 위치. "없으면 ntop이 아닌가?"(P0) · "코어를 실사용 가능하게 하는가?"(P1) · "발견성·편의인가?"(P2).

3-1. 티어별 근거§

티어기능근거(왜 이 티어인가)
P0 코어F1·F2·F3·F4·F5·F7·F8의미 격차(PRB-1~5)를 직접 메우는 가치 루프의 척추. 식별(F1·F2·F3·F4)·포트(F5)·안전 종료(F7·F8)가 빠지면 "익명 PID 더미"를 넘는 ntop의 정체성이 성립 안 함(01 §5-1, research-s2 §1).

3-3. MVP / Out-of-scope (참조 — 재정의 0)§

  • 향후 로드맵 근거: 도시에 §12뿐(Express/Fastify/Koa/Hapi 규칙 미등록, include_bun deprecated, Windows 시그널 한계, 외부 명령 의존, 로그 stdout 캡처 한계). → 단일 출처는 40-backlog.
