ntop — 기획 패키지 안내 허브

여러 런타임(Node·Python·Java·Deno·Bun·Ruby·PHP·.NET)의 서버 프로세스를 한 화면에서 모니터링·종료하는 Rust TUI + CLI 단일 바이너리. 이 허브는 이미 동작하는 코드를 역설계해 정리한 16종 기획 문서를 중요도 티어·읽기 순서·역할별 번들·최소 시작 세트로 안내합니다.

📦 MIT OSS · v0.2.0 📑 문서 16종 🔄 역설계 기획 패키지 🖥 macOS · Linux · Windows

② 작업(읽기) 순서 — 의존 기반 #

정체성 → 문제·맥락 → 구조 → 구현 정본 → 실행. 의존 흐름: 01→03→02→11→12→13→20→21→30→31→32→33→40→41 (00은 01과 함께).

  1. 정체성무엇을 왜 만드는가 (가치·MVP·지표·차별화)

    01 서비스 기획 00 비즈니스 모델 (필요 시 동반)

  2. 문제·맥락누구의 어떤 문제를, 경쟁 대비 어떻게

    03 페르소나→ 02 시장·경쟁→ 11 유저 스토리

  3. 구조구조·흐름·화면 (IA → 플로우 → 시안 → 명세)

    12 IA→ 13 유저 플로우→ 20 와이어프레임→ 21 화면 명세

  4. 구현 정본동작 정본 → 데이터 모델 → 인터페이스 계약 → 안전/정책

    30 기능 명세→ 31 ERD→ 32 API 명세→ 33 정책

  5. 실행무엇이 남았나(로드맵·정직 발견) → 어떻게 검증하나(QA)

    40 백로그→ 41 QA 테스트케이스

④ 최소 시작 세트 — 처음 6개만

ntop을 빠르게 파악하려면 이 6개로 충분하다. 나머지는 필요할 때 펼친다.

1 서비스 기획 01-service-plan
무엇을 왜 (가치·MVP·지표·차별화)
 2 PRD 10-prd
기능 목록·우선순위·문제 추적
 3 기능 명세 30-functional-spec
실제 동작·알고리즘 정본
 4 ERD 31-erd
데이터 모델·Config 기본값
 5 화면 명세 21-screen-spec
화면·키 바인딩(상호작용 계약)
 6 백로그 40-backlog
현 상태·다음 작업·정직 발견
핵심 안전 1줄 — TUI 종료 = SIGTERM 1회 직접 전송(폴링·자동 강제승격 없음), 자동 graceful→force 에스컬레이션은 CLI ntop kill 단건 전용(정본 30 · 정책 33). 이 구분을 먼저 이해하면 종료 관련 문서가 모두 일관되게 읽힌다.
