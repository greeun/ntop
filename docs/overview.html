ntop — 기획 패키지 종합 대시보드

익명 PID·CPU·메모리를 넘어 프레임워크·포트·역할·안전여부까지 — 여러 런타임의 서버 프로세스를 한 화면에서 모니터링하고 안전하게 종료하는 Rust TUI+CLI 단일 바이너리.

이 대시보드는 이미 동작하는 코드를 역설계한 16종 기획 문서의 구조·수치·관계만 시각화합니다(신규 내용 없음). 모든 다이어그램은 인라인 SVG/CSS — 외부 의존 0.

① 핵심 가치 루프 / 메커니즘 #

ntop의 모든 동작은 한 루프 위에 있다 — 스캔 → 식별 → 점검 → 안전 종료 → (반복). 각 단계가 특정 문제(PRB)를 직접 해소한다. 출처: 01-service-plan §4-1.

② 생태계 맵 #

ntop을 둘러싼 참여자와 가치 흐름. 코드 → 배포 → 사용 → 피드백·후원이 무료 OSS 순환을 이루고, ntop은 크레이트·시스템 명령에 의존하며 런타임/프레임워크를 모니터링한다. 결제·정산 화살표는 없다(후원만). 출처: 02-market-competition §1~2.

⑥ IA — 화면·명령 트리 #

TUI는 16 고유 화면(세로 3단 + 탭 4 + 모달 5 …), CLI는 5종 명령. 최대 깊이 3. 출처: 12-ia §3~6 · 21-screen-spec §1.

⑦ ERD · Config 기본값 #

인메모리 도메인 모델. ProcessInfo가 코어(18필드, 자기참조 트리), App이 집합 루트, NetworkConnection은 pid로 논리 조인된다. 출처: 31-erd.

Config 기본값 (단일 출처 · ~/.config/ntop/config.toml)

⑧ 시스템 아키텍처 #

한 스캔 틱의 데이터 흐름: 지속 System → 2-pass 스캔(분류 포함) → 포트 별도 채움 → 트리 빌드 → TUI/CLI 출력. 분류는 파일시스템을 절대 읽지 않는다(이름+커맨드만). 출처: 30-functional-spec · 32-api-spec.

⑪ MVP vs 비범위 #

v0.2.0 = MVP. 코어 가치 루프의 척추(P0)는 모두 포함되었고, 영구 제외 항목은 설계 정체성·해자(moat)와 직결된다. 출처: 01-service-plan §5 · 40-backlog §3.

✓ MVP 포함 (v0.2.0 · 21/21 구현)

  • P0 코어: 식별·정확분류(fs 미독)·포트 매핑·안전 종료·트리 종료·멀티런타임(8종)
  • P1 보조: 시그널 선택·필터/정렬/Node-only·네트워크 탭·Env 마스킹·정확 메모리·CLI 5종
  • P2 편의: 다중 선택·로그 자동 tail·TUI 일원화·설정(TOML)
  • TUI 16화면 + CLI 5명령 · 단일 바이너리 · MIT

✗ 비범위 (영구 제외 · 로드맵 아님)

  • 파일시스템(package.json) 기반 탐지 — npx/전역 프로세스 오분류 유발. 안 읽는 것이 해자(★영구)
  • 웹 GUI / 데스크톱 앱 — 터미널 도구 정체성 밖(PRB-8과 상충)
  • 프로세스 감독 / 자동 재시작 / 데몬화 — "죽이는 도구"이지 supervisor 아님(pm2·systemd 영역)
  • 시스템 전역 그래프 · 실시간 이력 차트 — 전역 모니터(btop·glances) 영역

⑫ 로드맵 · 정직 발견 #

v0.2.0(MVP)에서 21/21 완료. 향후는 권장 순서 3단계(약속된 릴리스 아님 — 가설)로 정렬했고, 같은 문서가 버그·죽은 코드를 정직하게 발견해 로드맵에 올렸다. 출처: 40-backlog.
