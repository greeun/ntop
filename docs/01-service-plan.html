01 · 서비스 기획 — 핵심 가치 / MVP / 성공지표 / 차별화 (ntop)

이 문서는 단일 출처(single source)다. ntop의 핵심 가치·MVP 범위·성공지표·차별화는 여기서만 확정하고, 하위 문서(10-prd 기능·우선순위, 40-backlog 로드맵)는 이 문서를 참조한다. 반대로 이 문서는 OSS 지속가능성/수익화/배포 채널은 00-business-model, 경쟁/생태계 분석은 02-market-competition, 페르소나 상세는 03-personas 를 단일 출처로 참조하며 재정의하지 않는다.
전제: ntop은 이미 v0.2.0까지 구현된 MIT 오픈소스 도구다(codebase-facts §0). 따라서 이 기획서는 "앞으로 만들 것"이 아니라 이미 만들어진 동작을 역설계(reverse-engineering) 해 가치·범위·지표· 차별점으로 정리한 것이다.
용어 풀이(첫 등장)
  • TUI(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI
  • CLI(Command-Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식
  • 런타임(runtime) = 코드를 실행하는 엔진(Node·Python·Java 등)
  • 프레임워크(framework) = 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등)
  • dev 서버 = 개발 중 코드를 바로 띄워 확인하는 임시 서버
  • MCP(Model Context Protocol) = AI 비서가 외부 도구·데이터에 붙는 표준 연결, 보통 npx로 백그라운드 실행
  • graceful 종료(정상 종료) = "정리하고 꺼져라"라고 요청(SIGTERM), 강제 종료(SIGKILL)와 다름
  • 에스컬레이션 = 정상 요청이 안 통하면 강제로 단계 올림
  • 트리 종료 = 부모-자식 프로세스를 묶어 함께 종료
  • PRB-n = research-s2가 정의한 문제(problem) 번호
  • MVP(Minimum Viable Product) = 가치를 전달하는 최소 제품 범위

1. 문제 정의 (who-what-why) — 의미 격차 #

출처: research-s2 §1~§3(PRB-1~8). 본 절은 요약이며, 문제 목록의 단일 출처는 research-s2(→ 03-personas)다.
  • 누가: 한 머신에서 여러 런타임의 서버 프로세스를 동시에 띄우는 개발자/운영자.
  • 무엇이 문제: 범용 프로세스 모니터(htop·top·Activity Monitor 등)는 프로세스를 익명의 PID·CPU·메모리로만 보여줄 뿐, "이 node가 어느 프레임워크의, 어느 포트의, 어느 프로젝트 서버인가"를 모른다 — 이것이 의미 격차(semantic gap) 다.
  • 왜 문제: 개발자는 그 격차를 lsof -i :3000(포트→PID), ps aux | grep node(이름 검색), Activity Monitor(메모리) 사이를 손으로 오가며 메우고, 종종 엉뚱한 프로세스를 죽이는 사고로 이어진다(research-s2 §1).

4-1. 가치 루프 (스캔 → 식별 → 점검 → 안전 종료 → 반복) #

루프가 말하는 것: ntop의 모든 동작은 식별 → 점검 → 안전 종료라는 한 루프 위에 있다(research-s2 §5의 3대 능력군과 1:1). 위험한 분기(종료·시그널)에는 반드시 모달 확인 또는 graceful 우선이 끼어 있어, 단계가 군더더기가 아니라 안전 게이트다(research-s3 §3·§9). 별도 로그인·모드 선택·설정 마법사가 없어 PRB-8(도구 왕복)을 구조적으로 제거한다.

5. MVP 범위 / Out-of-scope (G-e) — v0.2.0 = MVP #

MVP = 이미 출시된 v0.2.0 범위. ntop은 역설계 기획이므로 "구현 완료된 v0.2.0이 곧 MVP"다. 향후 항목은 40-backlog(로드맵, codebase-facts §12 근거)가 단일 출처이며, 여기서는 포함/제외 경계와 우선순위 근거만 확정한다.
