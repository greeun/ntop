# 00 · 비즈니스 모델 — OSS 채택·지속가능성 (ntop)

> **이 문서는 단일 출처(single source)다.** OSS 지속가능성/수익화/비용 구조/배포 채널/채택 퍼널은
> **여기서만 정의**하고, 다른 문서(`01-service-plan` 가치/MVP/지표, `40-backlog` 로드맵)는 이 문서를
> **참조**한다. 경쟁/생태계의 전체 그림은 `02-market-competition`(생태계 맵·경쟁 매트릭스)가 단일
> 출처이므로, 여기서는 지속가능성에 직접 닿는 흐름만 다룬다.
>
> **필수 면책(disclaimer):** **정산/take-rate/결제 없음 — 후원·배포 흐름으로 해석.** ntop은 **MIT
> 라이선스 오픈소스(OSS)** 도구이며 **정산(settlement)·take-rate(수수료 비율)·결제(payment)가 전혀
> 없다.** 따라서 표준 비즈니스 모델 문서의 "돈의 흐름(money-flow)"은 이 문서에서 **"기여·후원의 가치
> 흐름"으로 치환**해 해석한다. 결제 시퀀스·정산 트리거는 존재하지 않으며, 이를 지어내지 않는다.
>
> **용어 풀이(첫 등장):** **OSS**(Open Source Software) = 소스코드가 공개돼 누구나 쓰고 고칠 수 있는
> 소프트웨어 / **MIT 라이선스** = 가장 자유로운 OSS 약관 중 하나(상업적 사용·수정·포크 허용) /
> **채택 퍼널(adoption funnel)** = "알게 됨 → 설치 → 매일 씀 → 기여/후원"으로 사용자가 좁아지며
> 흘러가는 깔때기 / **이탈(drop-off)** = 다음 단계로 못 넘어가고 떨어져 나가는 것 / **전환 레버
> (conversion lever)** = 이탈을 줄여 다음 단계로 넘기는 장치 / **메인테이너(maintainer)** = 프로젝트를
> 유지·관리하는 사람 / **기여자(contributor)** = 코드·문서·버그 리포트를 보태는 사람 / **bus-factor** =
> "핵심 인원 몇 명이 빠지면 프로젝트가 멈추는가"를 나타내는 위험 지표(1이면 1명에게 전부 의존) /
> **상류(upstream)** = ntop이 의존하는 외부 라이브러리(크레이트)의 원작자/저장소 / **크레이트(crate)** =
> Rust의 패키지(라이브러리) 단위.

## 1. 한눈에 보는 결론 (Executive snapshot)

ntop의 "비즈니스 모델"은 매출이 아니라 **채택(adoption)과 지속가능성(sustainability)** 이다. 돈을
받지 않으므로 성공은 "얼마를 버나"가 아니라 **"얼마나 많은 개발자가 매일 쓰고, 얼마나 쉽게 기여가
들어오며, 메인테이너가 지치지 않고 계속 만들 수 있나"** 로 측정된다.

- **비용은 거의 0에 수렴한다.** ntop은 사용자의 로컬에서 도는 단일 실행파일이라 **서버·DB·결제·
  데이터 저장 인프라가 없다**(codebase-facts §0-1). 실질 비용은 **메인테이너의 시간**과 OSS 무료 한도
  안의 **CI(자동 빌드·테스트)** 뿐이다(§2).
- **배포는 여러 경로로 갈라진다.** 손쉬운 순서로 GitHub Releases 바이너리(무빌드) → Homebrew →
  `cargo install`(소스 빌드)이며, 채널마다 대상 사용자와 마찰이 다르다(§3).
- **수익화는 전부 "가설"이다.** GitHub Sponsors·기업 후원·향후 유료 기능은 **현재 코드에 없고 매출도
  0**이며, 모두 "만약 채택이 쌓인다면" 수준의 가정이다(§5). 숫자 목표는 사실이 아니라 가설로만 다룬다
  (정량 채택 지표·목표는 **01-service-plan §6** 단일 출처).
- **해자는 곧 지속가능성 엔진이다.** 새 런타임/프레임워크를 `Rule` 1개 + enum 값 1개로 추가하는
  **낮은 기여 장벽**(research-s1 §9-③)이 기여를 끌어들이고, 그 기여가 "식별 사전"을 복리로 키운다 —
  지속가능성과 차별화가 같은 메커니즘이다(§6).

## 2. 비용 구조 (B-4) — 로컬 단일 바이너리 = 인프라 비용 0

ntop은 **백엔드 서버·DB·HTTP API·회원 계정이 없는 로컬 도구**다(codebase-facts §0-1). 따라서 SaaS의
"서버·트래픽·스토리지 비용"이 **존재하지 않는다.** 비용은 사실상 두 가지 — 사람의 시간과 무료 CI다.

```
비용 항목            │ 부담 주체        │ 크기 / 성격           │ 근거
─────────────────────┼──────────────────┼───────────────────────┼──────────────────────────────
메인테이너 시간       │ greeun(공급측)   │ ★가장 큰 실질 비용     │ 코드·리뷰·이슈 트리아지·릴리스
CI (GitHub Actions)  │ GitHub(무료 한도) │ OSS 무료 범위 내       │ build·test(10모듈)·릴리스 자동화
패키지 배포           │ crates.io/HB/GH  │ 0 (무료 레지스트리)    │ 호스팅·다운로드 무료
런타임 인프라         │ 없음             │ 0                     │ 사용자 로컬 실행(서버 없음)
데이터 / 저장         │ 없음             │ 0                     │ 인메모리, 영속 저장 없음(§ERD 31)
결제 / 정산           │ 없음             │ 0                     │ ★결제·정산 자체가 없음
```

```
비용 구조 한눈에 — "비용의 거의 전부가 사람의 시간"

   [메인테이너 시간] ███████████████████████████  (실질 비용의 대부분)
   [CI/Actions]      ██                            (OSS 무료 한도 내)
   [배포 레지스트리]  ·                             (≈0, 무료)
   [서버/DB/인프라]   ·                             (0, 로컬 바이너리)
   [결제/정산]        ·                             (0, 존재하지 않음)
   ──────────────────────────────────────────────
   ⇒ 지속가능성의 병목은 "돈"이 아니라 "메인테이너의 시간·동기"다 → §6
```

**함의:** 비용이 거의 0이므로 ntop은 "수익을 내야 살아남는" 구조가 아니다. 위험은 **재무**가 아니라
**메인테이너 번아웃·bus-factor**(§6)에 있다. 따라서 수익화(§5)는 "생존 조건"이 아니라 "메인테이너
시간을 보전하는 선택적 보강"으로 위치한다.

## 3. 배포 채널 비교 (B-1) — ≥3 채널 매트릭스

ntop은 **Rust 단일 바이너리**라 설치 경로가 여러 갈래다. 아래 4개 채널을 **설치 난이도 / 업데이트 방식 /
대상 사용자 / 선결 조건**으로 비교한다. (codebase-facts §11은 `cargo install --git`와 릴리스 바이너리를
**확정 경로**로 명시; crates.io 게시·Homebrew 포뮬러는 Cargo 메타데이터(name/license/keywords/categories)
와 research-s1 §6-3 생태계 채널에 근거한 **가설/예정** 경로로, 사실과 가설을 구분해 표기한다.)

```
배포 채널 비교 매트릭스
 채널                       │ 설치 난이도        │ 업데이트 방식         │ 대상 사용자           │ 선결 조건            │ 상태
 ───────────────────────────┼────────────────────┼───────────────────────┼───────────────────────┼──────────────────────┼────────
 cargo install --git        │ 중 (소스 빌드)     │ 명령 재실행(수동)      │ Rust 보유 개발자       │ Rust 1.70+ 툴체인     │ 확정
 crates.io (cargo install)  │ 중 (소스 빌드)     │ cargo install 재실행   │ Rust 생태계 사용자     │ Rust 툴체인           │ 가설/예정
 Homebrew (brew install)    │ 낮음 (무빌드)      │ brew upgrade           │ macOS/Linux 일반       │ Homebrew 설치         │ 가설/예정
 GitHub Releases 바이너리    │ 매우 낮음 (다운로드)│ 새 바이너리로 교체     │ 비-Rust 사용자 포함     │ 없음(실행파일만)      │ 확정
```

```
채널별 "설치 마찰 ↓ vs 통제력 ↑" 스펙트럼
  마찰 높음(빌드 필요)                                          마찰 낮음(즉시 실행)
  ◀──────────────────────────────────────────────────────────────────────────▶
  cargo --git / crates.io        Homebrew              GitHub Releases 바이너리
  (최신 소스·자유도 ↑,           (무빌드, 자동 업데이트  (무빌드·무툴체인,
   Rust 툴체인 필요)              경로, mac/Linux)        모든 사용자, 수동 교체)
```

**채널 전략 요지:**
- **신규 채택(퍼널 ②, §4)의 핵심 레버는 GitHub Releases 바이너리와 Homebrew**다 — Rust 툴체인이 없는
  사용자(운영자·비-Rust 개발자, research-s1 페르소나 P2)까지 마찰 없이 끌어들인다.
- **기여자·얼리어답터는 `cargo install --git`**로 최신 `develop`/소스를 직접 빌드한다(research-s1 §6-1).
- 같은 단일 바이너리이므로 **채널이 늘어도 추가 인프라 비용은 0**(§2)이며, 채널 다양화는 곧 퍼널 ②의
  이탈을 줄이는 일이다.

## 4. 채택 퍼널 (B-2) — 인지 → 설치 → 상시 사용 → 기여/후원

매출 퍼널이 없으므로 ntop의 퍼널은 **"사용자가 얼마나 깊이 들어오나"** 의 깔때기다. 각 단계마다 **이탈
지점**(여기서 떨어진다)과 **전환 레버**(이탈을 줄이는 장치)를 함께 둔다.

```
채택 퍼널 (Adoption Funnel)                          이탈 지점 →  이를 줄이는 전환 레버
══════════════════════════════════════════════════════════════════════════════════════════
 ┌──────────────────────────────────────────────┐
 │ ① 인지 (Awareness)  "이런 도구가 있네"         │   접점: GitHub 검색·Stars,
 │     넓음 ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓        │   crates.io, 입소문, README/데모
 └───────────────────────┬──────────────────────┘
   이탈: README가 "의미 인지"라는 핵심 가치를 한눈에 못 보여줌(그냥 또 다른 htop으로 오인)
   레버: 한 줄 정의 + "node 더미 → Next.js@:3000" 데모 GIF + keywords(node/process/tui/monitor/nuxt)
                         ▼
 ┌──────────────────────────────────────────────┐
 │ ② 설치 (Install)  "한 번 깔아보자"             │   채널: §3 (Releases/Homebrew/cargo)
 │     ▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓▓                       │
 └───────────────────────┬──────────────────────┘
   이탈: Rust 툴체인·소스 빌드 시간 마찰(cargo 경로), "설치가 번거롭다"
   레버: 사전 빌드 바이너리(무툴체인)·Homebrew(무빌드 자동 경로) → §3 매트릭스의 낮은-마찰 채널
                         ▼
 ┌──────────────────────────────────────────────┐
 │ ③ 상시 사용 (Daily use)  "매일 켠다"            │   가치: 식별→점검→안전 종료
 │     ▓▓▓▓▓▓▓▓▓▓▓▓                               │   (가치 루프는 01-service-plan §4)
 └───────────────────────┬──────────────────────┘
   이탈: 첫 실행에서 가치 체감 실패(범용 모니터와 차이를 못 느낌)
   레버: 첫 화면에서 즉시 의미 식별(익명 node → 프레임워크·포트로 보임) + 안전 종료의 "안심" 경험
         (해소하는 문제 = PRB-1·3·5, 출처 research-s2 / 01-service-plan §1)
                         ▼
 ┌──────────────────────────────────────────────┐
 │ ④ 기여 / 후원 (Contribute / Sponsor)            │   접점: PR(새 Rule)·이슈·
 │     좁음 ▓▓▓▓                                   │   GitHub Sponsors(후원, 결제 아님)
 └──────────────────────────────────────────────┘
   이탈: 기여 방법 불명확·진입 장벽이 높다고 느낌
   레버: 새 런타임/프레임워크 = Rule 1개 + enum 값 1개로 끝(낮은 기여 장벽, research-s1 §9-③) →
         "내가 쓰는 프레임워크를 직접 추가" 동기 → §6 지속가능성 엔진으로 환류
```

| 단계 | 핵심 전환 질문 | 주 이탈 원인 | 레버(이탈↓) | 측정(가설 지표는 01 §6) |
|---|---|---|---|---|
| ① 인지 | "이게 htop과 뭐가 다른가?" | 가치 차별점 불명확 | 한 줄 정의·데모·키워드 | GitHub Stars·트래픽(가설, 01 §6) |
| ② 설치 | "깔기 쉬운가?" | 툴체인/빌드 마찰 | 무빌드 바이너리·Homebrew | 다운로드 수(가설, 01 §6) |
| ③ 상시 사용 | "매일 켤 만큼 가치 있나?" | 첫 인상 실패 | 즉시 의미 식별·안전 종료 | 리텐션은 OSS 특성상 추정만(가설) |
| ④ 기여/후원 | "보탤 만큼 쉽고 의미 있나?" | 기여 장벽 | Rule+enum 저장벽 | 기여자·외부 PR·Sponsors(가설, 01 §6) |

> **정량 채택 지표와 목표치는 모두 "가설"이며 `01-service-plan §6`(성공지표)이 단일 출처다.** 여기
> 00은 퍼널의 **구조와 레버**만 정의하고 숫자를 사실로 제시하지 않는다(중복정의 금지).

## 5. 수익화 옵션 (B-3) — 전부 "가설(hypothesis)", 현재 매출 0

**중요:** 아래 옵션은 **하나도 코드/계약/매출로 존재하지 않는다.** 모두 "만약 채택이 충분히 쌓인다면"
이라는 전제의 **가설**이며, ntop의 생존 조건이 아니라 §2에서 본 "메인테이너 시간"을 보전하는 **선택적
보강**일 뿐이다. **정산·take-rate·결제는 어떤 옵션에도 없다.**

```
수익화 옵션 비교 (★전 행 "가설")
 옵션                  │ 성격            │ 전제              │ 리스크 / 주의               │ 상태
 ─────────────────────┼─────────────────┼───────────────────┼─────────────────────────────┼────────────
 GitHub Sponsors      │ 개인 후원        │ 채택·평판 축적     │ 후원 동기 약하면 사실상 0    │ 가설
 기업 후원            │ 조직 후원        │ 팀 단위 채택       │ B2B 영업·홍보 부재           │ 가설
 향후 Pro/Team 기능    │ 유료 확장(가정)  │ 코어 가치 입증 후  │ ★미구현·OSS 신뢰 훼손 위험   │ 가설(미구현)
```

- **GitHub Sponsors** — 개인 후원 버튼. "후원 흐름"이지 **결제·정산이 아니다**(§7). 가장 OSS-친화적이고
  도입 비용이 낮은 가설.
- **기업 후원** — ntop을 팀 표준 도구로 쓰는 조직의 후원. 검증된 안전성·크로스플랫폼이 전제(research-s1
  §6-2 "팀 리드/도입 검토자"). B2B 영업 인력이 없다는 점이 리스크.
- **향후 Pro/Team 기능(가설·미구현)** — 예: 팀 공유 대시보드·원격 모니터링 등 **현재 코드에 전혀 없는**
  가정적 확장. 코어(식별·안전 종료)는 영원히 무료 MIT로 유지한다는 전제가 깨지면 OSS 신뢰를 훼손할 수
  있어 **신중히 다룬다.** 본 문서는 이를 **방향 가설로만** 적고, 기능을 설계·약속하지 않는다(spec §7
  비목표: 근거 없는 기능 창작 금지).

> **단일 출처 메모:** 어떤 수익화 옵션도 `33-policy`의 결제/환불 규칙으로 이어지지 않는다 — ntop의
> 정책 문서(`33`)는 **안전·프라이버시 정책**이지 결제 정책이 아니다(codebase-facts §0-1).

## 6. 지속가능성 구조 (B-5) — 진짜 병목은 시간·bus-factor

비용이 0에 가깝다는 것은(§2) ntop의 생존을 위협하는 것이 **돈이 아니라 사람**이라는 뜻이다.
지속가능성은 세 축 — **기여 유입 / bus-factor / 상류 의존 리스크** — 으로 본다.

```
지속가능성 구조 (3축)

 ┌─ ① 기여 유입(낮은 장벽) ───────────────────────────────┐
 │  새 런타임/프레임워크 = `Rule` 1개 + enum 값 1개 추가     │  ← research-s1 §9-③ / codebase-facts §2
 │  → 그 외 코드 변경 불필요 → "내가 쓰는 FW 직접 추가" 동기 │     "기여가 쌓일수록 식별 사전이 복리로 커짐"
 │  → 기여가 곧 차별화(해자)를 키우는 엔진                   │
 └────────────────────────────┬───────────────────────────┘
                              │ 유입이 많을수록 ▼ 위험 완화
 ┌─ ② bus-factor(핵심 인원 의존) ─────────────────────────┐
 │  현재 메인테이너 = greeun 1인 → bus-factor ≈ 1 (위험)   │  ← research-s1 §6-1
 │  완화: 낮은 기여 장벽(①) + MIT(자유 포크) → 단일 인원    │
 │        의존을 분산할 구조적 여지                          │
 └────────────────────────────┬───────────────────────────┘
                              │
 ┌─ ③ 상류 의존 리스크 ───────────────────────────────────┐
 │  빌드 의존 크레이트: sysinfo·ratatui·crossterm·clap·    │  ← codebase-facts §1·§12
 │   tokio·nix·windows-sys → 상류 중단/파괴적 변경 위험     │
 │  런타임 의존 시스템 명령: lsof·netstat·taskkill →       │
 │   부재 시 "빈 결과"로 우아하게 저하(graceful degrade)    │     (포트 칸이 비는 정도, 크래시 아님)
 └────────────────────────────────────────────────────────┘
```

| 축 | 현재 상태 | 위험 | 완화책(근거) |
|---|---|---|---|
| 기여 유입 | Rule+enum 저장벽 구조 | 인지 부족 시 기여 0 | 퍼널 ④ 레버(§4)·기여 가이드·good first issue(가설) |
| bus-factor | 메인테이너 1인(≈1) | 1인 이탈 시 정체 | 낮은 장벽으로 기여자 다변화·MIT 포크 자유(research-s1 §6-6) |
| 상류 의존 | 크레이트 7종 + 시스템 명령 3종 | 상류 파손·중단 | 널리 쓰이는 크레이트 선택·버전 고정·명령 부재 시 빈 결과(codebase-facts §12) |

**핵심 통찰:** ntop은 **차별화 메커니즘(낮은 기여 장벽 → 복리로 커지는 식별 사전)이 곧 지속가능성
메커니즘**이다. 즉 "기여를 쉽게 만든 설계"가 해자(02-market-competition / research-s1 §9)와
지속가능성을 동시에 떠받친다. 재무가 아니라 **이 기여 루프가 살아 있는지**가 ntop 생존의 척도다.

## 7. 기여 / 후원 가치 흐름도 (B-6·B-7) — money-flow의 치환

표준 비즈니스 모델의 "돈의 흐름"을 ntop에서는 **"코드·후원·피드백의 흐름"** 으로 치환한다. 아래
다이어그램에는 **결제·정산 화살표가 없다** — 모든 흐름은 무료 OSS 순환이다. (전체 생태계 맵(전 참여자·
대체재 포함)은 `02-market-competition` 단일 출처; 여기서는 **지속가능성에 직접 닿는 가치 흐름**만.)

```
기여 / 후원 가치 흐름  (정산·결제 0 — 화살표는 전부 "가치", 돈 아님)

   ┌────────────┐  코드/PR(새 Rule·런타임)  ┌───────────────────────┐
   │  기여자     │ ─────────────────────────▶ │                       │
   └────────────┘                            │      ntop repo        │
   ┌────────────┐  이슈·Stars·입소문(피드백)  │   (GitHub · greeun)   │
   │  사용자     │ ─────────────────────────▶ │   build·test(CI)      │
   └────────────┘                            │   → 릴리스/머지        │
   ┌────────────┐  GitHub Sponsors(후원 흐름) │                       │
   │  후원자     │ ─────────────────────────▶ │   ※ 결제·정산 아님     │
   └────────────┘                            └───────────┬───────────┘
        ▲                                                │ 새 버전·새 규칙
        │ 가치 환류(무료): 더 정확한 식별·안전 종료        │  (배포: §3 채널)
        │                                                ▼
        │                                    ┌───────────────────────┐
        └────────────────────────────────── │  배포 채널 → 사용자    │
                                             │  (cargo/HB/Releases)   │
                                             └───────────────────────┘

  ⇒ 흐른 가치가 메인테이너의 "시간·동기"를 충전(§2 비용의 핵심)하고,
     충전된 시간이 새 버전을 낳아 사용자에게 무료로 환류한다. 이 순환이 ntop의 "수익 모델"을 대신한다.
```

**흐름이 말하는 것:** ntop에는 "돈이 정산되는 시점"이 없다. 대신 **코드(기여자)·피드백(사용자)·
후원(후원자)** 이 repo로 모여 메인테이너의 시간을 보전하고, 그 시간이 새 릴리스로 환류한다. 후원
(GitHub Sponsors)조차 **거래가 아니라 일방적 후원 흐름**이라 take-rate·정산이 끼어들 자리가 없다.
