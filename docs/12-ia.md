# 12 · 정보구조(IA) — TUI 레이아웃 계층 + CLI 명령 트리 (ntop)

> 이 문서는 ntop **인터페이스 정보구조(IA)의 단일 출처**다. ntop의 모든 화면(=TUI 뷰/패널/탭/
> 다이얼로그 + CLI 명령 출력)이 **어떻게 묶이고 어디서 도달되는지**를 화면 ID와 함께 트리로 정리한다.
>
> **ntop은 터미널 도구다.** 여기서 "화면(screen)"은 웹/GUI 페이지가 아니라 **TUI 뷰·패널·탭·모달
> 다이얼로그 + CLI 서브커맨드 출력**을 뜻한다. 따라서 **ASCII 레이아웃이 곧 실제 충실도**다(spec §4-0).
>
> **참조(재정의 0):** 기능 ID(F1~F21)·우선순위는 **`10-prd`**, 문제(PRB-1~8)는 **`research-s2`(→03 §2)**,
> 플로우는 **`13-user-flows`**, **키 바인딩의 정규 출처는 `21-screen-spec`**(현재는 `research-s3` §7),
> Config 기본값은 **`31-erd`** 가 단일 출처다. 본 문서는 화면 IA를 소유하고 나머지는 **참조만** 한다
> (spec §4-3). 화면 ID(V-/TAB-/DLG-/BAR-/CLI-)는 `research-s3`를 정규로 승계한다.
>
> **전제:** ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스 도구**다(codebase-facts §0). 본 IA는 신규
> 화면 설계가 아니라 **이미 구현된 TUI/CLI를 역설계**해 정리한 것이다(창작 0). 모든 화면/색/진입은
> codebase-facts(이하 "도시에") §6·§7 **또는 실제 소스코드**(`src/tui/**`, `src/cli.rs`, `src/main.rs`)에
> 역추적된다.
>
> **용어 풀이(첫 등장):** **IA**(Information Architecture, 정보구조) = 화면·기능이 어떻게 묶이고
> 이동하는지의 뼈대 / **TUI**(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트
> UI / **CLI**(Command-Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식 / **패널(panel)** =
> 화면을 나눈 영역 / **탭(tab)** = 한 패널 안에서 전환되는 하위 화면 / **모달 다이얼로그(modal
> dialog)** = 떠서 다른 입력을 막고 결정을 받는 작은 창 / **포커스(focus)** = 지금 키 입력을 받는
> 패널 / **크롬(chrome)** = 콘텐츠가 아니라 항상 떠 있는 테두리 막대(상단/하단 바) / **플래그(flag)** =
> 명령 뒤에 붙이는 `--옵션`.

## 0. 사실 정정 (코드-진실) — 반드시 준수

> 일부 선행 표현이 "좌/우 분할" 또는 "목록 NAME/FRAMEWORK 컬럼"으로 적었으나 **실제 코드와 불일치**다.
> 본 IA는 아래 **코드-진실**(`src/tui/ui.rs` L37–44, `process_list.rs` + 도시에 §6 정정값)을 정규로 쓴다.

```

 ┌─────────────────────────────────┐       목록 컬럼(코드 권위, process_list.rs):
 │ BAR-TOP   상단 상태바 (1줄)       │ [크롬]   [✓/health] PID·NAME·PORT·THR·CPU·MEM·USER·STS·UPTIME
 ├─────────────────────────────────┤         └ FRAMEWORK 없음 ─┐
 │ V-LIST    프로세스 목록 (위 55%)  │ [패널1]                    │ 프레임워크 정체는 여기서만 노출:
 ├─────────────────────────────────┤                            ├─ 상세 Info 탭(TAB-INFO)
 │ V-DETAIL  상세 패널 (아래 45%)    │ [패널2]                    └─ CLI `ntop list` 테이블 FRAMEWORK 컬럼

본 문서의 모든 레이아웃·노출 경로는 이 코드-진실을 따른다. (도시에 §6의 "좌/우" 표기는 구식 — `research-s3` §0·§10에서 메인테이너 정정 권고로 회부됨.)

> **정정(NAME 인라인 배지):** 위 ②의 "프레임워크는 Info 탭/CLI 테이블에서만"은 **별도 FRAMEWORK 컬럼**에

> 자체는 NAME에서도 보인다. (로드맵 `40` M2)

## 1. 이 문서가 답하는 것 (Scope)

ntop의 인터페이스는 **두 진입점**으로 갈린다 — 서브커맨드 **없이** `ntop`이면 **TUI 기동**(`run_tui`),
서브커맨드가 **있으면** **CLI 일회성 출력**(`scan_blocking`). 둘 다 **동일 코어**(`ProcessScanner` +
`NetworkInspector`)를 공유한다(도시에 §7). 본 문서는:

- **TUI 레이아웃 계층**(§3)과 **두 진입점을 합친 IA 트리**(§4)를 그린다.
- **CLI 명령 트리**(§5)를 플래그·출력까지 정리한다.
- **모든 화면에 ID를 배정**하고 **노출 경로(어느 키/모드/명령에서 도달)** 를 표로 확정한다(§6).
- IA의 **구조적 특성**(평평함·얕은 깊이)이 문제(PRB)를 어떻게 구조적으로 줄이는지 분석한다(§7).

> **추적 사슬에서의 위치(spec §4-2):** `문제(research-s2) → 기능 ID(10) → **화면·CLI(12·13)** →
> 데이터(31) → 계약/정책(32·33)`. 본 문서가 "화면 IA" 마디를 소유한다.

## 2. 색·역할 범례 (코드 근거) — I-5

> 색·역할 표기는 `src/tui/widgets/status_bar.rs`·`empty_state.rs`의 실제 `Color::*` 사용에 근거한다
> (research-s3 §2-1 승계). 터미널이라 "색"은 영역 구분과 위험 신호의 핵심 시각 언어다.

```
 역할 표기                                      │ 색 (코드 근거)
 ───────────────────────────────────────────────┼──────────────────────────────────────────
 [크롬]   상시 노출 막대(BAR-TOP / BAR-BOTTOM)   │ 배경 DarkGray · 글자 White   (status_bar.rs)
 [패널]   포커스 가능 영역(V-LIST / V-DETAIL)    │ 활성 테두리 강조
 [탭]     상세 패널 하위 화면(Info/Log/Net/Env)  │ 활성 탭 Yellow 강조           (detail_panel.rs)
 [모드]   키 해석을 바꾸는 상태(필터)            │ 강조 Yellow (filter_text 커서 █)
 [모달]   입력 막는 확인창(다이얼로그 5종)       │ 위험=Red(kill) · 선택=Yellow
 [빈/로딩] 프로세스 0건 안내 + 스피너            │ 스피너 Cyan                   (empty_state.rs)
 [CLI]    서브커맨드 일회성 출력                 │ 터미널 기본(파이프 친화)
 ───────────────────────────────────────────────┴──────────────────────────────────────────
 색 의미(status_bar.rs 임계값 실측):
   Cyan  = 브랜드/스피너            Green = 정상 (CPU < 50)
   Yellow= 강조/주의 (CPU > 50 · 키힌트 · Node-only · 활성 탭)
   Red   = 위험 (CPU > 80 · 파괴적 kill)      DarkGray = 크롬 배경 / 비활성
```

> 아래 트리(§3·§4)는 각 노드에 `[역할]` 태그로 색을 표기한다(터미널 md라 실색 대신 역할 태그 사용 —
> HTML(`12-ia.html`)에서 인라인 SVG로 실색 렌더).

## 3. TUI 레이아웃 계층 (★ 세로 분할) — I-1

> ntop TUI는 **세로 3단**이다: 상단바(1줄) / 메인 콘텐츠(가변) / 하단바(1줄). 메인 콘텐츠는 다시
> **세로(상/하)로 55%/45% 분할**된다(`ui.rs` L37–44 `Layout::vertical`). 프로세스가 0건이면 메인 전체가
> 빈/로딩 위젯(V-EMPTY)으로 대체된다. 필터 모드·다이얼로그는 이 위에 겹친다.

```
 ┌══════════════════════════════════════════════════════════════════════════┐
 │ BAR-TOP  [크롬 · DarkGray/White]                              (1줄 고정)   │  ← status_bar.rs::render_top_bar
 │   ntop v0.2.0 | CPU:x% | MEM:used/totalMB | Servers/Nodes:n [Node-only]    │     F1·F18 / PRB-6 맥락
 │   | Refresh:Ns | [H]elp                                                    │
 ├══════════════════════════════════════════════════════════════════════════┤
 │                                                                            │
 │   main_content (가변 높이 ≥5)  ─ 둘 중 하나로 렌더 (ui.rs L19) ─           │
 │                                                                            │
 │   ┌── (A) flat_list 비었음 → V-EMPTY [빈/로딩 · 스피너 Cyan] ──────────┐  │  ← empty_state.rs
 │   │      ⠹ Scanning for server processes...                            │  │     F1 대기 / PRB-1
 │   │      No server processes found. Waiting...                         │  │
 │   └──────────────────────────────────────────────────────────────────┘  │
 │                                                                            │
 │   ┌── (B) flat_list 있음 → 세로 분할 Layout::vertical([55%,45%]) ───────┐  │  ← ui.rs L37–44
 │   │  ┌────────────────────────────────────────────────────────────┐  │  │
 │   │  │ V-LIST  [패널·포커스1]   프로세스 목록(트리)  (위 55%)        │  │  │  ← process_list.rs
 │   │  │   컬럼: [✓]PID·NAME·PORT·THR·CPU·MEM·USER·STS·UPTIME          │  │  │     F2·F5·F10·F12 / PRB-1·2·4
 │   │  │   (정렬자 ^/v · 트리 ▾/▸ · Space 체크 □)                      │  │  │
 │   │  ├────────────────────────────────────────────────────────────┤  │  │

 │   │  │   │ [Info] · Log · Net · Env   (Tab/←→ 순환)             │  │  │  │
 │   │  │   └─────────────────────────────────────────────────────┘  │  │  │
 │   │  │   TAB-INFO  PID/Framework/Port/CPU/Mem/Uptime  (info_tab)   │  │  │     F1·F18 / PRB-1·6
 │   │  │   TAB-LOG   감지 로그파일 실시간 tail          (log_tab)    │  │  │     F16
 │   │  │   TAB-NET   LISTEN 포트 + 활성 TCP 연결        (net_tab)    │  │  │     F6 / PRB-2
 │   │  │   TAB-ENV   환경변수(민감값 마스킹)            (env_tab)    │  │  │     F17 / PRB-6
 │   │  └────────────────────────────────────────────────────────────┘  │  │
 │   └──────────────────────────────────────────────────────────────────┘  │
 │                                                                            │
 ├══════════════════════════════════════════════════════════════════════════┤
 │ BAR-BOTTOM [크롬 · DarkGray/White]  컨텍스트 의존 키힌트      (1줄 고정)    │  ← status_bar.rs::render_bottom_bar
 │   (리스트 포커스 / 상세 포커스 / 필터 활성 / 다이얼로그별로 다른 힌트)     │     전반(발견성)
 └══════════════════════════════════════════════════════════════════════════┘

   ── 위에 겹치는 오버레이(한 번에 하나) ──────────────────────────────────────
   V-FILTER [모드 · Yellow]  필터 입력 → 하단바에 " /{텍스트}█" 표시 (`/` 진입)   F11 / PRB-1·4

**계층이 말하는 것:** 깊이 2~3단으로 끝난다. 상단바·하단바는 **콘텐츠가 아닌 크롬**이라 상시 떠 있고,
실제 작업은 V-LIST(식별)→V-DETAIL 4탭(점검)에서 끝난다. **위험한 동작만 모달 한 겹**(DLG-*)으로
감싼다. 별도 라우팅·페이지 전환이 없어 PRB-8(GUI↔터미널 왕복)을 구조적으로 제거한다(F20).

**IA가 말하는 것:** 깊이 2~3단의 **얕은 트리**다. TUI는 메인 한 화면(V-MAIN)에 정보를 모으고 파괴적
동작만 모달 한 겹으로 감싼다. CLI는 TUI와 동일 코어를 재사용하는 **평면 5개 명령**이다. 별도
라우팅·페이지 전환·설정 마법사·로그인이 없다(§7).

## 5. CLI 명령 트리 (플래그·필수성·출력) — I-3

> `ntop`(서브커맨드 없음)은 TUI를 기동하고, 5개 서브커맨드는 일회성 출력을 낸다(도시에 §7, `cli.rs`
> clap derive). **CLI 출력 스키마(table 컬럼·JSON 필드·CSV 컬럼)의 정규 출처는 `32-api-spec`**이며 본
> 트리는 요지만 표기한다.

```
ntop
│
├─ (서브커맨드 없음)              ─▶  TUI 기동 (run_tui)              [A] 위 IA 트리
│
├─ list                          ─▶  CLI-LIST   분류된 서버 목록                F19 / PRB-1·7
│    ├─ --json                        (= --format json 단축)
│    └─ --format <table|csv|json>     기본 table
│         · table : PID · NAME · FRAMEWORK · PORT · CPU · MEM · UPTIME
│         · json  : 모든 필드 + depth
│         · csv   : PID · PPID · NAME · FRAMEWORK · PORTS · CPU · MEMORY · UPTIME · STATUS (9컬럼)
│
├─ kill <PID>                    ─▶  CLI-KILL    종료 (PID는 --all 없으면 필수)  F19·F7·F8 / PRB-5·7
│    ├─ --tree                        부모+자식 트리 전체(재귀 수집 → 확인 → 각각)   F8
│    ├─ --signal <SIG>                전송 시그널 지정(기본 SIGTERM, 대소문자 무시)   F9
│    ├─ --all                         모든 서버 프로세스 일괄(PID 생략 가능)
│    └─ --no-confirm                  확인 프롬프트 생략
│         · graceful flow: SIGTERM → graceful_timeout(31 정규값) 초과 시 자동 SIGKILL 에스컬레이션
│
├─ info <PID>                    ─▶  CLI-INFO    단일 프로세스 상세              F19 / PRB-1·6
│         · PID/PPID/name/runtime/framework/version/ports(LISTEN)/CPU/memory/
│           threads/uptime/user/status/health/cwd/command/open FDs/env/network
│
├─ log <PID>                     ─▶  CLI-LOG     로그 스트림                     F19·F16
│         · cwd 로그소스 자동 감지 → 200ms 폴링 → 프로세스 사망 시 종료
│
└─ config                        ─▶  CLI-CONFIG  설정 출력                       F19·F21
          · 설정 파일 경로(~/.config/ntop/config.toml) + [general][display][filter] 값
```

**CLI 설계 요지:** 평면 5명령 + 단순 플래그. `--json`/`--format csv`가 **헤드리스 자동화**(PRB-7)의
연결점이고, `kill`의 `--tree`/`--all`/`--no-confirm`이 스크립트 일괄 종료를 가능케 한다. TUI와 같은
코어라 동작 의미가 일치한다(예: graceful→force 에스컬레이션은 둘 다 적용, 트리거 방식만 다름 — §13 참조).

## 7. IA 특성 분석 — 왜 이렇게 평평한가

> IA의 구조 자체가 ntop의 가치(저마찰·안전)를 떠받친다. 깊이·분기 수를 시각화한다.

```
 깊이(depth)   화면 수    예
 ─────────────────────────────────────────────────────────────────────────
 depth 0    바이너리       ntop
 depth 1    진입 분기      [A] TUI 기동(run_tui)   ·   [B] CLI 5명령
 depth 2    주 화면        V-MAIN(=BAR-TOP/BOTTOM·V-LIST·V-DETAIL·V-EMPTY) · CLI-LIST/KILL/INFO/LOG/CONFIG
 depth 3    하위/오버레이  TAB-INFO/LOG/NET/ENV · V-FILTER · DLG-KILL/KILLTREE/SIGNAL/FORCE/HELP
 ─────────────────────────────────────────────────────────────────────────
 ⇒ 최대 깊이 3. 페이지 전환·라우팅 없음. "위험 동작(kill/signal)"만 모달 한 겹 추가.
```

**구조가 해소하는 문제(PRB):**

| IA 특성 | 해소하는 문제 | 근거 |
|---|---|---|
| 단일 메인 뷰에 식별+점검+종료 집약(V-MAIN) | **PRB-8** GUI↔터미널 왕복 | 외부 GUI로 나갈 필요 제거(F20) |
| 위험 동작만 모달 한 겹(DLG-*) | **PRB-3** 오kill | 확인 게이트가 오발사 차단(F7·F8·F9) |
| 필터/정렬/Node-only가 화면 전환 없이 인플레이스 | **PRB-1·4** 식별·다중런타임 | `rebuild_view` 실시간(F11·F12·F13) |
| 설정 화면 없음(파일+`ntop config`) | 군더더기 화면 제거 | CLI 관행, 저마찰(spec §4-0) |
| TUI/CLI 동일 코어·동일 IA 의미 | **PRB-7** 헤드리스 자동화 | 스크립트 연결성(F19) |

> **디자인 품질(4-D) 적합:** 수동 모드/역할 선택 벽 없음(런타임 자동 분류, Node-only는 토글 한 키),
> 맥락 파생 정보(포트·런타임·소유자)는 스캔으로 server-side 해결해 사용자에게 묻지 않음, 중복 단계
> 없음(필터 실시간). 상세는 `research-s3` §9가 단일 출처.
