# 03 — 핵심 페르소나 (personas)

> 이 문서는 ntop의 **페르소나 단일 출처(single source of truth)**다. 다른 문서
> (`01-service-plan`·`10-prd`·`11-user-stories`·`13-user-flows`·`41-qa-testcases`)는 여기를
> **참조**할 뿐 페르소나를 중복 정의하지 않는다(spec §4-3).
>
> ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스 도구**이고 **결제·DB·회원 계정이 없는 로컬 개발자
> 도구**이므로, 페르소나는 **돈 내는 고객이 아니라 사용자(개발자/운영자)**다(도시에 §0-1). ntop에 대한
> 모든 동작 인용은 `docs/_harness/codebase-facts.md`(이하 "도시에")와 실제 코드에 근거한다. 페르소나의
> 일과·빈도 수치는 **본 도구가 푸는 문제의 사용 맥락에서 개연성 있게 구성한 추정**이며(도시에에 없는
> 값), 그 점을 라벨로 분리해 둔다.
>
> **용어 풀이(첫 등장):**
> - **런타임(runtime)** = 코드를 실행하는 엔진(Node·Python·Java 등).
> - **프레임워크(framework)** = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등).
> - **dev 서버** = 개발 중 코드를 바로 띄워 확인하는 임시 서버.
> - **MCP(Model Context Protocol)** = AI 비서가 외부 도구·데이터에 붙는 표준 연결 방식, 보통 `npx`로
>   백그라운드 프로세스로 뜬다.
> - **LSP(Language Server)** = 에디터 자동완성·진단을 제공하는 백그라운드 서버.
> - **graceful 종료(정상 종료)** = 프로세스에 "정리하고 꺼져라"라고 **요청**하는 것(SIGTERM),
>   강제로 즉사시키는 것(SIGKILL)과 다름.
> - **고아 프로세스** = 부모가 먼저 죽어 떠도는 자식 프로세스(포트를 계속 점유할 수 있음).
> - **PID** = 프로세스 고유 번호. **cwd** = 프로세스의 현재 작업 디렉터리.
> - **오kill** = 엉뚱한 프로세스를 잘못 종료하는 것.

## 1. 페르소나 개요 — 누구를 그리는가

ntop의 페르소나는 **"한 머신에서 여러 런타임의 서버 프로세스를 동시에 띄우는 사람"**이다. 이들이
공통으로 겪는 핵심 문제는 **의미 격차(semantic gap)** — 범용 모니터(htop·top·Activity Monitor 등)는
프로세스를 **익명의 PID·CPU·메모리**로만 보여줄 뿐, "이 `node`가 어느 프레임워크의, 어느 포트의, 어느
프로젝트 서버인가"를 모른다(자세한 경쟁 비교는 `02-market-competition` §3·§4 참조).

```
 범용 모니터가 보는 것              내가 알아야 하는 것              ← 이 사이가 "의미 격차"
 ┌──────────────────────┐          ┌────────────────────────────┐
 │ PID 41822  node  2.1% │          │ Next.js dev 서버 @:3000      │
 │ PID 41835  node  0.4% │   ???    │ FastAPI @:8000 (백엔드)      │
 │ PID 41840  node  0.0% │  ─────▶  │ Claude Code MCP (건들면 끊김) │
 │ PID 41902  node  0.0% │          │ Vite 번들러 (자식)           │
 │ PID 42013  python 5%  │          │ Celery 워커 (큐 처리 중)     │
 └──────────────────────┘          └────────────────────────────┘
   "이름·번호·숫자뿐"                 "프레임워크·포트·역할·안전여부"
```

3종 페르소나는 이 격차를 서로 다른 각도에서 겪는다:

| 페르소나 | 한 줄 정체 | 대표 통증 축 | 세그먼트(02 §5-2) |
|---|---|---|---|
| **P1 박서연** | 폴리글랏 풀스택 개발자(★1순위, 완전 구체) | 식별·포트·안전 종료·GUI 왕복 | ① 폴리글랏 |
| **P2 정태호** | 플랫폼/운영 엔지니어(헤드리스·SSH, 빈도/결과 보강) | 헤드리스 자동화·트리 종료·고아 프로세스 | ⑤ 운영자 + ② 마이크로서비스 |
| **P3 이준** | AI 응용 개발자(★완전 구체, npx/MCP 헤비) | 익명 더미·오분류 회피 | ③ AI/도구 + ④ Rust·TUI 선호 |

> ★ **화면 표기 원칙(코드-진실, 반드시 준수):** ntop TUI는 **세로 2분할** 레이아웃이다 — **위 패널 =
> 프로세스 목록, 아래 패널 = 상세 패널**(좌/우 아님; 도시에 §6). **상단(위) 목록 컬럼에는 FRAMEWORK가
> 없다**(컬럼: `[health] PID·NAME·PORT·THR·CPU·MEM·USER·STS·UPTIME`). 프레임워크 정체는 **아래 상세
> 패널의 Info 탭** 또는 **CLI `list` 테이블의 FRAMEWORK 컬럼**에서만 확인한다. 본 문서의 모든 "ntop이
> 메움" 서술은 이 코드-진실을 따른다.

## 2. 문제 앵커 — PRB-1~8 참조표 (P-4)

> 이 표는 **G-a 추적의 출발점**이다(원천: `research-s2.md` §3, 문제 정의 단계의 공유 앵커). `10-prd`는
> 모든 P0/P1 기능을 여기의 PRB-ID로 역추적하고, `11-user-stories`·`41-qa-testcases`도 이 ID를 인용한다.
> "메우는 ntop 동작"은 모두 도시에 §2~§7의 실제 동작이며, 본 문서는 기능을 새로 창작하지 않는다.

| PRB-ID | 문제(한 줄) | 핵심 통증 | 주 페르소나 | 메우는 ntop 동작(도시에) |
|---|---|---|---|---|
| **PRB-1** | 익명 `node` 더미 — 어느 게 뭔지 모름 | 의미 격차(식별 불가) | P1·P3 | 런타임/프레임워크 분류·`display_name()` (§2·§3) |
| **PRB-2** | 포트 역추적 수작업(`lsof` 왕복) | 문맥 전환 | P1·P2 | `NetworkInspector` LISTEN 포트 매핑 (§4-1) |
| **PRB-3** | 엉뚱한 프로세스 kill 위험(MCP/도구 vs dev) | 오kill·세션 끊김 | P3·P1 | 프로세스-로컬 전용 탐지 + kill 확인 게이트 (§2·§3·§6) |
| **PRB-4** | 다중 런타임 문맥 전환 | 인지 부하 | P1·P2 | 8런타임 횡단 분류 + Node-only 토글 (§2·§6) |
| **PRB-5** | 안전하지 않은 종료(`kill -9` 즉사·고아) | 데이터 유실·고아 | P1·P2 | `graceful_kill`(자동 에스컬레이션)·`kill_tree`(역순)·`SignalPicker` (§5·§6) |
| **PRB-6** | 메모리 오판독 + 환경 민감값 노출 | 오판·유출 | P1·P2 | macOS `phys_footprint`·`mask_env_values` (§4-2·§3·§6) |
| **PRB-7** | 헤드리스·자동화 환경에서 점검·종료 어려움 | GUI 불가·스크립트화 | P2 | TUI+CLI 동일 코어, `list --json`/`kill --tree` (§7) |
| **PRB-8** | GUI↔터미널 문맥 전환(Activity Monitor 왕복) | 도구 이동 피로 | P1 | 터미널 내 TUI로 점검·종료 일원화 (§6·§7) |

> **세그먼트 연결(02 §5-2):** PRB-1·3 = 세그먼트 ①③(폴리글랏·AI/도구)의 1순위 통증, PRB-2·5 = 모든
> 핵심 세그먼트 공통, PRB-7 = 세그먼트 ⑤(운영자)의 핵심.

## 3. P1 — 박서연 · 폴리글랏 풀스택 개발자 (★완전 구체, 1순위)

| 항목 | 내용 |
|---|---|
| **역할** | 시드 단계 스타트업의 풀스택 개발자(4년차). 혼자 프런트·백·워커를 다 맡음. macOS(Apple Silicon) MacBook Pro. |
| **상황·맥락** | 한 맥북에서 **Next.js dev 서버(:3000) + FastAPI 백엔드(:8000, `uvicorn`) + Celery 워커 + 가끔 레거시 Spring Boot API(:8080)**를 동시 구동. 거기에 Vite 번들러(자식 프로세스), 에디터 LSP, 그리고 **Claude Code + npx MCP 서버 2~3개**까지 떠 있다. `node` 프로세스만 보통 6~8개. 세그먼트 ①(폴리글랏 풀스택, 02 §5-2). |
| **일과 한 장면** | 오전 9시, 어제 핫리로드가 꼬여 **포트 3000이 안 풀린** 상태로 출근. `pnpm dev`가 "포트 사용 중" 에러. `lsof -i :3000`으로 PID를 찾아 `kill -9`. 그런데 `ps aux \| grep node`를 보니 `node`가 7개. 어느 게 dev 서버이고 어느 게 번들러·LSP·MCP인지 한눈에 안 보여, 커맨드라인을 일일이 눈으로 대조. **지난주엔 MCP 서버를 dev 서버로 착각해 죽였다가 Claude Code 세션이 끊겼다.** |
| **페인포인트(빈도/결과)** | • **PRB-1 익명 더미** — 매 세션(하루 수 회). 결과: 식별에 매번 1~3분, 오kill의 원인.<br>• **PRB-2 포트 역추적** — 주 10회+. 결과: `lsof→ps→kill` 3단계 왕복.<br>• **PRB-3 엉뚱한 kill** — 월 1~2회 실제 사고. 결과: MCP/LSP 세션 끊김 → 재시작·재인덱싱 10~15분 손실.<br>• **PRB-5 거친 종료** — 매 재시작. 결과: `kill -9` 습관으로 임시상태 유실 위험·고아 프로세스.<br>• **PRB-6 메모리 오판** — 메모리 의심 때마다. 결과: Activity Monitor 값과 터미널 값이 달라 혼란. |
| **현재 우회법** | `lsof -i :PORT`, `ps aux \| grep node`, **Activity Monitor**(메모리는 GUI로) 세 곳을 왕복(PRB-8). 포트↔프로젝트를 노션 메모로 관리. |
| **원하는 결과** | "**한 화면에서** 어느 `node`가 어느 프레임워크·포트·프로젝트인지 즉시 보고, **그 자리에서 안전하게** 종료하고 싶다. MCP·LSP는 실수로 못 죽이게." |
| **ntop이 메우는 방식(코드-진실)** | `ntop` 실행 → **위 목록**에서 NAME·PORT로 후보를 좁히고, **아래 Info 탭**(또는 CLI `list` 테이블 FRAMEWORK 컬럼)에서 "이건 Next.js dev, 저건 MCP"를 확정(도시에 §6) → 대상 선택 후 `x`(KillConfirm) → `graceful_kill`(정상 요청 후 안 죽으면 강제 자동, 도시에 §5)로 안전 종료. |

## 4. P2 — 정태호 · 플랫폼/운영 엔지니어 (운영자 축, 빈도/결과 보강)

> 평가자 비고(S2-비고2)를 반영해 P2의 "일과 한 장면"과 페인포인트 **빈도·결과를 P1 수준으로 보강**한다.
> 빈도 수치는 운영 맥락 기반 개연 추정이다(도시에에 없는 값), ntop 동작 인용은 전부 도시에 근거.

| 항목 | 내용 |
|---|---|
| **역할** | 중견 IT 회사의 플랫폼 엔지니어(7년차). 여러 팀이 공유하는 **Linux 빌드/스테이징 서버 3대**를 관리. **GUI 없음, SSH로만 접속.** 인시던트 1차 대응(on-call) 로테이션. |
| **상황·맥락** | 공용 서버에 여러 팀의 서비스 프로세스가 섞여 있다 — Node 빌드 에이전트, Python 배치(`gunicorn`/`celery`), Java 데몬. 한 서버에 분류 대상 서버 프로세스가 10~20개. 배포 직후 **좀비·고아 프로세스가 포트를 점유**하는 인시던트가 잦다. 세그먼트 ⑤(다수 서비스 점검 운영자) + ②(마이크로서비스 로컬), 02 §5-2. |
| **일과 한 장면** | **새벽 2시 배포 후 알림**: 모니터링이 ":8080 응답 없음"을 띄운다. SSH 접속 → `netstat -ano`/`lsof -i :8080`로 포트 점유 PID를 찾는 데 **5~10분**(어느 팀 어느 서비스인지 사내 위키와 대조). 급한 마음에 `kill -9`로 **부모만** 죽였더니, **자식 워커가 고아로 남아** 30분 뒤 같은 포트를 다시 잡아 **동일 인시던트가 재호출**된다. 결국 자식 PID를 일일이 찾아 또 죽이고서야 정리. |
| **페인포인트(빈도/결과)** | • **PRB-7 헤드리스·자동화** — **점검할 때마다(주 5~10회 SSH 세션)**. 결과: GUI(Activity Monitor) 사용 불가, `ps/lsof/netstat`를 묶은 자작 스크립트를 매번 손으로 — 점검당 수 분 소모.<br>• **PRB-5 안전 종료(트리)** — **매 정리(주 수회)**. 결과: `kill -9` 부모-only로 고아 발생 → 포트 재점유 → **월 2~3회 2차 인시던트 재호출**, 야간 대응 시간 30분+ 추가.<br>• **PRB-2 포트 역추적** — **매 인시던트(주 5회+)**. 결과: `netstat→위키 대조→kill` 3단계 왕복, 새벽 시간 압박 속 PID 오독 위험.<br>• **PRB-4 다중 런타임** — **배포 직후마다(주 수회)**. 결과: Node·Python·Java 런타임별 점검·종료 절차가 달라 인지 부하 + 잘못된 절차 적용 위험. |
| **현재 우회법** | `ps`/`lsof`/`netstat`를 묶은 자작 쉘 스크립트, 팀별 포트 위키, 인시던트 런북. 고아 정리는 `pkill -P <ppid>`를 기억나는 대로. |
| **원하는 결과** | "SSH 터미널에서 **식별·필터·안전 종료(트리)**를 한 번에. 모니터링 스크립트에 붙이게 **JSON으로** 뽑고, 죽일 땐 **자식까지** 정상 정리해 고아 재점유를 끊고 싶다." |
| **ntop이 메우는 방식(코드-진실)** | TUI에서 필터(`/`)·Node-only(`n`)로 대상 좁히고 `K`(KillTreeConfirm) → `kill_tree`가 **자식부터 역순** 종료해 고아 방지(도시에 §5·§6). 자동화는 `ntop list --json`으로 점유 현황을 스크립트에 넣고 `ntop kill <PID> --tree --no-confirm`으로 정리(도시에 §7). 헤드리스라 GUI 왕복 자체가 사라짐. |

## 6. 원하는 결과 → ntop 능력 매핑 (P-3)

> 각 페르소나의 "원하는 결과"를 ntop의 3대 능력 — **식별(identify) / 포트(port) / 안전 종료(safe-kill)**
> — 로 연결한다. 이 매핑이 `10-prd`의 **기능↔문제 추적표(G-a)**로 그대로 이어진다. ★ 화면 위치는
> 코드-진실(목록 위/상세 아래, FRAMEWORK는 Info 탭·CLI 테이블)로 표기.

```
 페르소나 원하는 결과            ntop 능력군        구체 동작(도시에)                            PRB
 ─────────────────────────────────────────────────────────────────────────────────────────────────────
 P1 "어느 node가 뭔지 즉시" ─▶  [식별]   ──▶  런타임/FW 분류·display_name·Info탭(§2·§3·§6)     PRB-1
 P1/P3 "MCP·dev 정확 구분" ─▶  [식별]   ──▶  프로세스-로컬 전용 탐지(fs 안 읽음)(§2)           PRB-3
 P1/P2 "포트 그 자리에서"   ─▶  [포트]   ──▶  NetworkInspector LISTEN·목록 PORT컬럼·Net탭(§4-1)  PRB-2
 P1 "그 자리서 안전 종료"   ─▶  [안전]   ──▶  KillConfirm + graceful_kill 에스컬레이션(§3·§5)   PRB-3,5
 P2 "자식까지 정상 정리"    ─▶  [안전]   ──▶  kill_tree 역순 종료(§5)                           PRB-5
 P2 "JSON으로 자동화"       ─▶  [식별]   ──▶  CLI list --json / kill --tree(§7)                 PRB-7
 P1/P2 "런타임 횡단/Node만" ─▶  [식별]   ──▶  8런타임 분류 + n키 Node-only 토글(§2·§6)          PRB-4
 P1 "정확한 메모리/안전 Env"─▶  [식별]   ──▶  macOS phys_footprint·mask_env_values(§4-2·§3)     PRB-6
 P1 "GUI↔터미널 왕복 제거"  ─▶  [식별+안전] 터미널 내 TUI 일원화(§6·§7)                         PRB-8
```

**3대 능력 요약(도시에 근거):**
- **식별(identify):** 데이터 주도 규칙 테이블(`FRAMEWORK_RULES`/`RUNTIME_RULES`)로 8런타임·다수
  프레임워크를 분류, **파일시스템을 읽지 않는** 프로세스-로컬 전용 탐지(도시에 §2). 화면상 정체는
  **위 목록의 NAME·PORT + 아래 Info 탭 / CLI `list` 테이블의 FRAMEWORK 컬럼**으로 노출(목록 컬럼에
  FRAMEWORK 없음 — 도시에 §6). → PRB-1·3·4 해소.
- **포트(port):** `NetworkInspector`가 LISTEN 포트를 PID별 매핑(도시에 §4-1) → 위 목록 PORT 컬럼·Net 탭.
  → PRB-2 해소.
- **안전 종료(safe-kill):** kill 확인 게이트 → `graceful_kill`(정상 종료 후 timeout 시 강제 자동
  에스컬레이션) → `kill_tree`(자식부터 역순) → 필요 시 `SignalPicker`(도시에 §3·§5·§6). → PRB-3·5 해소.

## 7. 시각화 (G-g)

### 7-1. 페르소나 여정 맵 (P-5) — P1(박서연) 기준

> 단계별 **행동 / 접점(도구) / 감정 / 통증(PRB) / ntop이 메우는 지점**. 가운데 점선(=ntop 도입)을 기준으로
> "도구 없는 현재"와 "ntop 사용"이 갈린다. 감정: 😀좋음 😐보통 😟혼란 😣불안 😡분노 😩피로 😌안도 😄확신.
> ★ ⑥ 단계는 코드-진실(위 목록 + 아래 상세 패널, FRAMEWORK는 Info 탭/CLI 테이블)로 표기.

```
 단계 ▶  ①서버 여러 개 띄움  →  ②문제 발생       →  ③어느 게 뭔지 모름  →  ④포트 역추적/엉뚱한 kill
 ─────────────────────────────────────────────────────────────────────────────────────────────────────
 행동    pnpm dev / uvicorn      포트 충돌·리로드      ps aux | grep node      lsof -i :3000 → kill -9
         / celery / npx MCP      꼬임으로 재시작 시도   (node 7개 발견)         (PID 추정해서 발사)

 감정    😀 집중·생산적          😐 약간 짜증          😟 혼란                 😣 불안 → (오kill 시) 😡
 통증    —                      PRB-2 시작            PRB-1                   PRB-2·PRB-3·PRB-5
 ─────────────────────────────────────────────────────────────────────────────────────────────────────
 단계 ▶  ⑤(대안)Activity Monitor 왕복
 행동    GUI 열어 메모리·이름 재확인
 접점    Activity Monitor(GUI) ↔ 터미널
 감정    😩 문맥 전환 피로
 통증    PRB-6·PRB-8
 ═════════════════════════════════ ntop 도입 (전환점) ═══════════════════════════════════════════════
 단계 ▶  ⑥ntop으로 식별                              →  ⑦안전 종료
 ─────────────────────────────────────────────────────────────────────────────────────────────────────
 행동    `ntop` 실행 → 위 목록에서 NAME·PORT 확인,        대상 선택 → x(KillConfirm) → Enter
         아래 Info/Net 탭(또는 CLI list FRAMEWORK         → graceful(정상요청)→안 죽으면 강제 자동
         컬럼)으로 정체 확정 (목록엔 FRAMEWORK 없음)
 접점    ntop TUI (단일 화면·세로 상/하 분할)            ntop TUI (KillConfirm/SignalPicker)
 감정    😌 안도 ("아, 이게 Next dev, 저건 MCP구나")      😄 확신 ("dev만 안전하게 정리")
 통증    PRB-1·2·4·6·8 해소                              PRB-3·5 해소
```

**여정이 말하는 것:** ①~⑤는 **도구를 오가며 사람이 의미를 복원**하는 구간(통증 누적, 감정 악화). ⑥에서
ntop이 **의미를 화면에 직접 표시**하는 순간 문맥 전환이 사라지고(😩→😌), ⑦에서 **안전 장치가 내장된 종료**
