02 — 시장·경쟁·생태계 분석 (market-competition)

이 문서는 ntop의 경쟁/생태계 단일 출처(single source of truth)다. 다른 문서(00-business-model·01-service-plan)는 여기를 참조할 뿐 경쟁자 비교·생태계 맵을 중복 정의하지 않는다(spec §4-3).

ntop은 이미 v0.2.0까지 구현된 MIT 오픈소스(OSS) 도구다. 따라서 이 문서는 신규 아이디어 검증이 아니라 이미 만들어진 도구를 시장·경쟁·생태계 좌표에 위치시키는 역설계 정리다. ntop에 대한 모든 주장은 docs/_harness/codebase-facts.md(이하 "도시에")와 실제 코드로 역추적 가능하다. 경쟁자 사실은 널리 알려진 실재 도구의 공개 동작이며, 버전별로 달라질 수 있는 세부 기능은 보수적으로(없는/제한적인 쪽으로) 표기했다(§3 버전 주석 참조).

용어 풀이(첫 등장):

  • TUI(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI.
  • CLI(Command Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식.
  • 런타임(runtime) = 코드를 실행하는 엔진(Node·Python·Java 등).
  • 프레임워크(framework) = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등).
  • 생태계(ecosystem) / 가치 네트워크 = 한 제품을 둘러싼 모든 참여자와, 그들이 서로 주고받는 가치(코드·피드백·후원·배포 등)의 그물.
  • 해자(moat) = 경쟁자가 쉽게 따라 하지 못하게 막아 주는 구조적 차별점.

Sprint Contract (self-proposed checks)

이 문서가 충족시키는 관찰 가능한 체크(spec §5·§6·§8, playbook S4-g1 02 바에서 도출):

0. 한눈에 보는 결론 (Executive snapshot)

ntop은 범용 시스템 모니터(htop·btop·top·glances·gotop·Activity Monitor)와 다른 평면에 있다. 범용 모니터는 모든 프로세스를 익명의 PID·CPU·메모리로 보여주지만, "이 node가 어느 프레임워크의, 어느 포트의, 어느 프로젝트 서버인가"는 모른다. ntop은 그 의미 격차(semantic gap)를 런타임/프레임워크 인지 분류로 메우고, 그 자리에서 포트 확인 + 안전한 종료(정상 종료 요청 → 안 죽으면 강제 종료, 트리 단위 종료)까지 끝낸다.

1. 생태계(가치 네트워크) 분석 M-1

한 제품은 혼자 존재하지 않는다. ntop을 둘러싼 모든 참여자를 7범주로 식별하고, 각자의 역할 / 인센티브(왜 참여하나) / 가치 교환(무엇을 주고받나) / 의존(무엇에 기대나)을 정리한다. ntop은 무료 OSS 도구이므로 "결제/정산 파트너"는 없으며 후원/배포 인프라로 해석한다(spec §6).

1-1. 공급측 (Supply) — 가치를 만드는 쪽

참여자역할인센티브(왜 참여하나)가치 교환의존
메인테이너 (greeun)코드·릴리스·이슈 분류(트리아지)·방향 결정자기 문제 해결(직접 쓰는 도구 = dogfooding), 평판, 학습코드/릴리스 제공 ↔ 피드백·기여·후원 수령crates.io·GitHub·CI·의존 크레이트
기여자 (contributors)PR(특히 새 규칙/런타임 추가), 버그 리포트, 문서필요 기능 직접 추가, 포트폴리오·학습, OSS 기여 이력코드/패치 제공 ↔ 머지·크레딧 수령규칙 테이블 확장 구조(도시에 §2: Rule 1개 + enum variant 1개)

1-2. 수요측 (Demand) — 가치를 쓰는 쪽

참여자역할인센티브가치 교환의존
개발자(채택자)일상 사용: 식별·포트·안전 종료의미 격차 해소, 문맥 전환 제거, 오kill(엉뚱한 프로세스 종료) 방지사용 ↔ 이슈/Stars/입소문 제공단일 바이너리·터미널·대상 런타임
운영자다수 서비스 프로세스 점검·정리빠른 점검, 안전한 종료, CLI 자동화사용 + 자동화 스크립트 ↔ 피드백CLI list --json/kill 계약(도시에 §7)

2. 생태계 맵 M-2 — 이해관계자 / 가치 네트워크 다이어그램

노드-엣지 다이어그램. 화살표 = 가치 흐름. 범례: 코드=코드/패치, 피드백=이슈/Stars/입소문, 후원=GitHub Sponsors 흐름, 배포=설치 아티팩트, 의존=빌드/런타임 의존, 호출=외부 명령 호출. (외부 렌더러 0 — Mermaid 미사용, 순수 인라인 SVG.)
플랫폼 허브(GitHub repo) 참여자 노드 가치 흐름 후원/제도(간접)

맵이 보여주는 가치 순환(요약):

  • 코드 흐름: 메인테이너·기여자 → GitHub repo → CI → 배포 채널 → 개발자(설치).
  • 피드백/후원 흐름: 개발자 → (피드백·Stars·입소문) → repo → 메인테이너; 개발자/후원자 → (GitHub Sponsors 후원) → 메인테이너. 둘 다 무료 OSS 순환이며 결제·정산은 없다.
  • 의존 흐름: ntop → 의존 크레이트(빌드 의존) + 시스템 명령(런타임 호출). 이 외부 명령 의존이 §6의 약점이기도 하다.
  • 보완재↔대상: 대상 런타임/프레임워크가 ntop의 모니터링 대상이자 가치 증폭기. 대체재는 이 순환 밖에서 동일 사용자를 두고 경쟁한다(의미 인지 축 부재).

3. 실명 경쟁자 비교 매트릭스 M-3

3-1. 비교 축 정의 (무엇을 비교하는가)

각 축은 "그 도구가 하는가 / 안 하는가"를 가른다(spec §6).

축 코드축 이름의미(쉬운 설명)
RT런타임 인지프로세스 이름 문자열을 넘어 "이건 Node/Python/Java 런타임이다"를 의미로 분류하는가

탐지 범위scope무엇을 보는가 — 시스템 전체 / 컨테이너 / 자기 실행분 / 서버 선별
탐지 범위(scope) 보충: 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor)는 시스템 전체 프로세스, ctop은 컨테이너만, pm2/systemd/launchd는 자신이 실행/구성한 것만 본다. ntop은 여러 런타임의 "서버 프로세스"만 선별해 본다(규칙에 안 맞는 프로세스는 표시하지 않음 — 도시에 §2). 이 범위 차이가 포지셔닝의 핵심이다(§4).

3-2. 매트릭스 (✓ 한다 / △ 부분·조건부 / ✗ 안 한다)

✓ 한다 △ 부분·조건부 ✗ 안 한다

RT·FW·PORT 세 열이 동시에 ✓인 도구는 ntop뿐 — 히트맵에서 ntop 행만 전 열이 초록이다.

셀 판정 근거(주요 △/✗ 설명):

  • htop/btop/top/gotop/glances/Activity Monitor RT·FW = ✗: 프로세스 이름 문자열만 보여줄 뿐 "Node 런타임의 Next.js 서버"라는 의미 분류가 없다. 이것이 §4 포지셔닝의 가로축 격차다.
  • KILL: htop/btop/top/gotop/glances/launchd = △(시그널은 보내지만 정상→강제 자동 에스컬레이션·트리 종료가 없거나 제한적). Activity Monitor = △(Quit/Force Quit 2종만). ctop = △(컨테이너 stop/kill). pm2/systemd = ✓(정상→강제 에스컬레이션·라이프사이클 제어가 강함, 단 범위가 좁음). ntop = ✓(임의 시그널 6종[Unix]/3종[Win] + graceful_kill 자동 에스컬레이션 + kill_tree 역순 종료 — 도시에 §5).
  • PORT: ctop = △(컨테이너 published 포트만). 나머지 범용 모니터 = ✗. ntop = ✓(NetworkInspector로 LISTEN 포트를 PID별 매핑 — 도시에 §4-1).
  • XPLAT: glances·pm2 = ✓(Win 포함), htop·btop·top·gotop·ctop = △(Win 네이티브 미흡/WSL 필요), Activity Monitor·launchd = ✗(macOS 전용), systemd = ✗(Linux 전용). ntop = ✓(mac·Linux·Win, platform.rs + windows-sys — 도시에 §4-2).
  • T+C: top = △(배치 모드 top -b), systemd = △(CLI 강력 + systemd-cgtop 별도 도구). glances·pm2 = ✓. ntop = ✓(TUI 기본 + list/kill/info/log/config 서브커맨드, --json/--format csv — 도시에 §7).
  • BIN: glances·pm2 = ✗(파이썬/Node 런타임 필요). systemd·launchd = △(OS 구성요소라 "설치" 개념이 아님). ntop = ✓(Rust 단일 바이너리: cargo install/Homebrew/릴리스 바이너리 — 도시에 §11).
  • pm2 RT = △: Node 전제로 동작하나 임의 프로세스를 발견·분류하진 않는다.
매트릭스가 말하는 것: RT·FW·PORT 세 열이 동시에 ✓인 도구는 ntop뿐이다. 범용 모니터는 RT·FW·PORT가 모두 ✗, 매니저(pm2/systemd)는 KILL은 강하지만 RT·FW·PORT가 비어 있고 범위가 좁다.

4. 포지셔닝 쿼드런트 M-5

두 축으로 도구들을 배치한다.

  • 가로축(X) = 런타임/프레임워크 "의미 인지": 프로세스를 PID·이름으로만 보는가(왼쪽) → "어느 런타임의 어느 프레임워크 서버"로 이해하는가(오른쪽).
  • 세로축(Y) = 프로세스 "제어 깊이": 보기만 하는가(아래) → 임의 시그널·정상/강제·트리·라이프사이클 제어까지 하는가(위).

읽는 법:

  • 오른쪽 위(고-의미인지 × 고-제어) = ntop이 거의 독점. 의미를 알면서(오른쪽) 동시에 안전하게 제어(위)하는 도구가 ntop이다.
  • 왼쪽 위 = pm2·systemd·launchd: 제어는 깊지만 의미 인지는 낮고(런타임 분류·프레임워크 식별 없음) 범위가 좁다(자신이 관리/구성한 것만).
  • 왼쪽 가운데~아래 = 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor): 의미 인지가 거의 없고 제어도 시그널 전송 수준에 머문다.
  • ctop은 사실상 다른 평면(컨테이너)에 있어 좌표상 한 점으로 두지만 직접 경쟁이 아니다.
핵심 통찰: 경쟁자들은 모두 가로축 왼쪽(의미 인지 낮음)에 몰려 있다. ntop의 차별화는 이 가로축(의미 인지)을 오른쪽으로 끌고 가면서 세로축(제어)도 높게 유지한 데 있다. "더 편리"가 아니라 축이 다르다.

5. 시장 / 사용자 맥락 M-6 (정성)

ntop은 무료 OSS 도구이므로 매출 기반 시장 규모(TAM/SAM/SOM 금액)를 산정하지 않는다. 대신 spec/playbook 지시에 따라 "채택 가능 사용자 모집단/세그먼트"를 정성적으로 추정한다.

5-1. 시장 흐름 — 왜 지금 이 도구가 의미 있나

  1. 폴리글랏(polyglot, 한 프로젝트에 여러 언어) 로컬 개발의 일상화. 프런트엔드는 Node(Next.js/Nuxt), 백엔드는 Python(FastAPI/Django)·Java(Spring Boot)·.NET, 거기에 워커·큐·캐시까지 — 한 개발자가 한 머신에서 여러 런타임의 서버를 동시에 띄우는 것이 표준이 됐다.
  2. 한 머신에 동종 프로세스 더미. 특히 node 프로세스는 dev 서버·번들러·LSP(에디터 자동완성 서버)·MCP 서버·CLI 도구가 모두 같은 이름으로 떠 있어, 범용 모니터에서는 구분이 불가능한 익명 더미가 된다.
  3. AI 도구·MCP 서버의 폭증. npx로 띄우는 MCP(AI 비서가 외부 도구에 붙는 표준 연결)/도구 프로세스가 늘면서, "건드리면 안 되는 도구 프로세스 vs 죽여도 되는 dev 서버"를 가려야 할 필요가 커졌다. (ntop의 프로세스-로컬 전용 탐지 원칙은 바로 이 오분류를 막기 위한 설계 — 도시에 §2.)
  4. 터미널 우선·키보드 중심 워크플로의 부활. TUI 모니터(htop·btop)의 인기가 보여주듯, 개발자는 GUI보다 터미널에서 빠르게 점검·조치하길 원한다. ntop은 이 흐름 위에 의미 인지를 얹었다.

5-2. 채택 가능 사용자 모집단/세그먼트 (정성, 깔때기형)

  • 세그먼트 우선순위 근거: ①·③이 "의미 격차" 통증이 가장 크고 빈도가 높다(매일·세션마다 발생). 이들이 03-personas의 P1(폴리글랏)·P3(AI 도구) 페르소나에 대응한다.
  • 모집단 추정의 성격: 금액이 아니라 "폴리글랏 로컬 개발 × 터미널 사용 × 다중 런타임"의 교집합으로 좁힌 정성 추정이다. 수치 목표(Stars/다운로드 등)는 00-business-model/10-prd에서 "가설"로 다룬다(여기서 산정하지 않음).

7. 차별화 모방난이도(해자/moat) 논거 M-7 후반부

spec/playbook은 "더 편리/더 빠름" 같은 일반론을 실패로 규정한다. 따라서 차별화는 실명 경쟁자 대비 명시 축 + 모방난이도 논거로 제시한다. 이 §7은 01-service-plan 차별화 섹션의 단일 출처다.

명시 비교 한 줄 정리: htop·btop·top·glances·gotop·Activity Monitor는 RT·FW·PORT 축이 모두 ✗이고, pm2·systemd·launchd는 KILL은 ✓지만 RT/FW/PORT가 비고 범위가 좁다. 세 축(RT·FW·PORT)이 동시에 ✓인 도구는 ntop뿐이다(§3 매트릭스).

이 차별점이 쉽게 복제되지 않는 이유 — 모방난이도 3가지:

  1. 데이터 주도 규칙 테이블 = 축적된 경험 사전. FRAMEWORK_RULES/RUNTIME_RULES는 "각 프레임워크/런타임이 프로세스 이름·커맨드라인에 드러나는 방식"을 수집한 사전이다 — node_modules/.bin/next, -m flask, bin/rails, org.springframework, macOS의 comm 16자 절단을 푸는 normalize_name(도시에 §2)… 이건 기능 하나가 아니라 현장에서 하나씩 검증해 쌓은 경험치다. 범용 모니터가 이를 "한 번에 추가"할 수 없는 이유다.
  2. 프로세스-로컬 전용 탐지 원칙 = 비자명한 설계 결정. ntop은 분류에 파일시스템(package.json)을 절대 읽지 않는다(도시에 §2). 순진하게 복제하려는 쪽은 당연히 cwd의 package.json을 읽으려 하고, 그 순간 전역 실행 프로세스(npx MCP 서버, CLI 도구)를 상속된 cwd로 오분류하는 함정에 빠진다. ntop의 정확성은 이 함정을 의도적으로 피한 결과이며, 같은 통찰 없이는 재현하기 어렵다. (그래서 spec은 fs 기반 탐지 제안 자체를 금지한다.)
  3. 확장 구조가 해자를 시간에 따라 키운다. 새 런타임/프레임워크는 Rule 1개 + enum variant 1개만 추가하면 되고 다른 코드 변경이 불필요하다(도시에 §2). 즉 기여가 쌓일수록 사전이 복리로 커진다 — 후발 복제자는 매번 격차를 따라잡아야 하는 움직이는 목표를 상대하게 된다.
결론: ntop의 해자는 "단일 기능"이 아니라 ① 경험이 응축된 데이터 사전 × ② 비자명한 탐지 설계 원칙 × ③ 복리로 커지는 확장 구조의 결합이다. 이는 범용 모니터가 분류 계층 없이 흉내 낼 수 없고, 프로세스 매니저가 발견·분류 범위를 넓히지 않고는 도달할 수 없는 위치다.
