# 02 — 시장·경쟁·생태계 분석 (market-competition)

> 이 문서는 ntop의 **경쟁/생태계 단일 출처(single source of truth)**다. 다른 문서
> (`00-business-model`·`01-service-plan`)는 여기를 **참조**할 뿐 경쟁자 비교·생태계 맵을
> 중복 정의하지 않는다(spec §4-3).
>
> ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스(OSS) 도구**다. 따라서 이 문서는 신규 아이디어 검증이
> 아니라 **이미 만들어진 도구를 시장·경쟁·생태계 좌표에 위치시키는 역설계 정리**다. ntop에 대한 모든
> 주장은 `docs/_harness/codebase-facts.md`(이하 "도시에")와 실제 코드로 역추적 가능하다. 경쟁자 사실은
> 널리 알려진 실재 도구의 공개 동작이며, 버전별로 달라질 수 있는 세부 기능은 **보수적으로(없는/제한적인
> 쪽으로) 표기**했다(§3 버전 주석 참조).
>
> **용어 풀이(첫 등장):**
> - **TUI(Text User Interface)** = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI.
> - **CLI(Command Line Interface)** = 명령 한 줄로 실행해 결과를 출력하는 방식.
> - **런타임(runtime)** = 코드를 실행하는 엔진(Node·Python·Java 등).
> - **프레임워크(framework)** = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등).
> - **생태계(ecosystem) / 가치 네트워크** = 한 제품을 둘러싼 모든 참여자와, 그들이 서로 주고받는
>   가치(코드·피드백·후원·배포 등)의 그물.
> - **해자(moat)** = 경쟁자가 쉽게 따라 하지 못하게 막아 주는 구조적 차별점.

## 0. 한눈에 보는 결론 (Executive snapshot)

ntop은 범용 시스템 모니터(htop·btop·top·glances·gotop·Activity Monitor)와 **다른 평면**에 있다.
범용 모니터는 모든 프로세스를 **익명의 PID·CPU·메모리**로 보여주지만, "이 `node`가 **어느
프레임워크의, 어느 포트의, 어느 프로젝트** 서버인가"는 모른다. ntop은 그 **의미 격차(semantic gap)**를
**런타임/프레임워크 인지 분류**로 메우고, 그 자리에서 **포트 확인 + 안전한 종료(정상 종료 요청 → 안
죽으면 강제 종료, 트리 단위 종료)**까지 끝낸다.

```
  ┌────────────────────────────────────────────────────────────────────┐
  │  ntop이 이기는 축                ntop이 약한 축                       │
  │  ─────────────────              ─────────────────                    │
  │  · 런타임 인지(8종)             · 시스템 전역 그래프·실시간 이력      │
  │  · 프레임워크 식별(독점)          (btop·glances·gotop 우위)          │
  │  · 포트→프로세스 매핑 내장       · 외부 명령(lsof/netstat) 의존       │
  │  · 안전한 종료(graceful→force·   · 프로세스 감독·자동 재시작 없음     │
  │    tree)                          (pm2·systemd 우위 — ntop은         │
  │  · 크로스플랫폼 + 단일 바이너리    "죽이는 도구"이지 "살려두는        │
  │    + TUI/CLI 동시                  도구"가 아님)                      │
  └────────────────────────────────────────────────────────────────────┘
   해자(moat) = ① 데이터 주도 규칙 테이블 × ② 프로세스-로컬 전용 탐지 × ③ 복리 확장구조 (→ §7)
```

## 1. 생태계(가치 네트워크) 분석 (M-1)

> 한 제품은 혼자 존재하지 않는다. ntop을 둘러싼 **모든 참여자**를 7범주로 식별하고, 각자의
> **역할 / 인센티브(왜 참여하나) / 가치 교환(무엇을 주고받나) / 의존(무엇에 기대나)**을 정리한다.
> ntop은 무료 OSS 도구이므로 "결제/정산 파트너"는 없으며 **후원/배포 인프라로 해석**한다(spec §6).

### 1-1. 공급측 (Supply) — 가치를 만드는 쪽

| 참여자 | 역할 | 인센티브(왜 참여하나) | 가치 교환 | 의존 |
|---|---|---|---|---|
| **메인테이너 (greeun)** | 코드·릴리스·이슈 분류(트리아지)·방향 결정 | 자기 문제 해결(직접 쓰는 도구 = dogfooding), 평판, 학습 | 코드/릴리스 제공 ↔ 피드백·기여·후원 수령 | crates.io·GitHub·CI·의존 크레이트 |
| **기여자 (contributors)** | PR(특히 새 규칙/런타임 추가), 버그 리포트, 문서 | 필요 기능 직접 추가, 포트폴리오·학습, OSS 기여 이력 | 코드/패치 제공 ↔ 머지·크레딧 수령 | 규칙 테이블 확장 구조(도시에 §2: `Rule` 1개 + enum variant 1개) |

### 1-2. 수요측 (Demand) — 가치를 쓰는 쪽

| 참여자 | 역할 | 인센티브 | 가치 교환 | 의존 |
|---|---|---|---|---|
| **개발자(채택자)** | 일상 사용: 식별·포트·안전 종료 | 의미 격차 해소, 문맥 전환 제거, 오kill(엉뚱한 프로세스 종료) 방지 | 사용 ↔ 이슈/Stars/입소문 제공 | 단일 바이너리·터미널·대상 런타임 |
| **운영자** | 다수 서비스 프로세스 점검·정리 | 빠른 점검, 안전한 종료, CLI 자동화 | 사용 + 자동화 스크립트 ↔ 피드백 | CLI `list --json`/`kill` 계약(도시에 §7) |
| **팀 리드/도입 검토자** | 팀 도구 채택 판단 | 검증된 안전성·크로스플랫폼 | 채택 결정 ↔ 조직 단위 사용 | 안전·프라이버시 정책(→ `33-policy`) |

### 1-3. 플랫폼 / 중개 (Platform / Intermediary)

| 참여자 | 역할 | 가치 교환 | 의존 관계 |
|---|---|---|---|
| **GitHub** | 코드 호스팅·이슈·Stars·Sponsors·Releases | 메인테이너↔기여자↔사용자 협업/배포 허브 | ntop 저장소가 여기에 존재(repo: github.com/greeun/ntop) |
| **crates.io** | Rust 패키지 레지스트리(`cargo install`) | 빌드 가능한 소스 배포 | 의존 크레이트도 여기서 해소 |
| **Homebrew** | macOS/Linux 패키지 배포 채널 | `brew install`로 손쉬운 설치 | 릴리스 아티팩트/포뮬러 |
| **GitHub Releases(릴리스 바이너리)** | 사전 빌드 단일 바이너리 배포 | 빌드 없이 설치하는 경로 | CI(GitHub Actions)가 산출 |

### 1-4. 보완재 (Complements) — ntop의 가치를 키우는 것

| 보완재 | 역할/관계 | 의존 방향 |
|---|---|---|
| **대상 런타임 8종** (Node·Python·Java·Deno·Bun·Ruby·PHP·.NET) | 이들이 많이 쓰일수록 ntop의 식별 가치↑ | ntop이 이들을 **인지**(RUNTIME_RULES, 도시에 §2) |
| **대상 프레임워크** (Next.js·Nuxt·NestJs·Django·Flask·FastAPI·Spring Boot·Rails·Laravel·ASP.NET) | 프레임워크 식별 = ntop의 핵심 가치 | FRAMEWORK_RULES에 규칙 등록(도시에 §2) |
| **터미널 에뮬레이터** (iTerm·Terminal·Windows Terminal 등) | ntop의 TUI 렌더 무대 | crossterm/ratatui 통해 의존 |
| **의존 크레이트** | ntop을 구성하는 빌딩블록 | ntop → 이들에 **빌드/런타임 의존** |
| └ sysinfo 0.33 | 프로세스 열거·CPU delta·메모리 | 코어 스캔(도시에 §4) |
| └ ratatui 0.29 / crossterm 0.28 | TUI 위젯 / 터미널 제어 | 렌더·이벤트 |
| └ clap 4 | CLI 파싱(서브커맨드/플래그) | CLI 계약 |
| └ tokio 1 | async 이벤트 루프(tick/key/resize) | TUI 루프 |
| └ nix 0.29 (Unix) / windows-sys 0.59 (Win) | 시그널·OS FFI(저수준 OS 호출) | 종료·플랫폼 FFI |
| **의존 시스템 명령** (lsof·netstat·taskkill) | 포트 매핑·Windows 종료 | ntop이 **외부 호출**(없으면 빈 결과 — 도시에 §12) |

### 1-5. 대체재 (Substitutes) — 미채택 시의 대안

§3의 경쟁자 전부 = htop·btop·top·glances·gotop·Activity Monitor·ctop·pm2·systemd·launchd, 그리고
수동 baseline(`lsof`/`ps | grep`/`kill`). 이들은 ntop을 안 쓸 때의 대안이며, **의미 인지 축이 비어
있다는 점**이 공통 약점(§3·§4).

### 1-6. 제도 / 규범 (Institutions)

| 제도 | 역할/영향 |
|---|---|
| **MIT 라이선스(OSI 승인)** | 자유로운 사용·수정·포크·상업적 채택 허용 → 채택 장벽 0, 신뢰 형성 |
| **Rust 생태계 관행** | `cargo fmt`/`clippy`-clean · `develop→main` 브랜치 · SemVer 버전(도시에 §10) → 기여 일관성 |
| **의존 크레이트 메인테이너** | 상류(upstream) 업데이트·보안 패치 → ntop 지속성에 간접 영향 |

### 1-7. 후원 / 배포 인프라 (Sponsorship / Distribution — "결제 파트너" 치환)

| 인프라 | 역할 | 비고 |
|---|---|---|
| **GitHub Sponsors** | 메인테이너 후원 경로(**가설적** 수익화) | 정산/take-rate 아님 — **후원 흐름** |
| **GitHub Actions (CI)** | 빌드·테스트(테스트 10모듈)·릴리스 자동화 | 품질 게이트·바이너리 산출 |
| **crates.io 레지스트리** | 배포·의존 해소 인프라 | §1-3과 중첩(중개+인프라) |

> 수익화(GitHub Sponsors·기업 후원·향후 Pro/Team 기능)는 전부 **"가설"**이며 그 단일 출처는
> `00-business-model`이다. 본 문서는 생태계 좌표만 정리한다 — 정산/take-rate는 ntop에 **없다**.

**맵이 보여주는 가치 순환(요약):**
- **코드 흐름:** 메인테이너·기여자 → GitHub repo → CI → 배포 채널 → 개발자(설치).
- **피드백/후원 흐름:** 개발자 → (피드백·Stars·입소문) → repo → 메인테이너; 개발자/후원자 →
  (GitHub Sponsors 후원) → 메인테이너. **둘 다 무료 OSS 순환이며 결제·정산은 없다.**
- **의존 흐름:** ntop → 의존 크레이트(빌드 의존) + 시스템 명령(런타임 호출). 이 외부 명령 의존이
  §6의 약점이기도 하다.
- **보완재↔대상:** 대상 런타임/프레임워크가 ntop의 모니터링 대상이자 가치 증폭기. 대체재는 이
  순환 밖에서 동일 사용자를 두고 경쟁한다(의미 인지 축 부재).

## 3. 실명 경쟁자 비교 매트릭스 (M-3)

### 3-1. 비교 축 정의 (무엇을 비교하는가)

각 축은 "그 도구가 **하는가 / 안 하는가**"를 가른다(spec §6).

| 축 코드 | 축 이름 | 의미(쉬운 설명) |
|---|---|---|
| **RT** | 런타임 인지 | 프로세스 이름 문자열을 넘어 "이건 Node/Python/Java 런타임이다"를 **의미로** 분류하는가 |
| **FW** | 프레임워크 식별 | "Next.js/FastAPI/Spring Boot 서버다"까지 식별하는가 |
| **PORT** | 포트→프로세스 매핑 | "이 프로세스가 포트 3000에서 듣고 있다"를 그 자리에서 보여주는가 |
| **KILL** | 시그널/graceful 종료 관리 | 임의 시그널 전송 + 정상 종료→강제 종료 자동 단계 + 트리(부모-자식) 종료 |
| **TREE** | 트리 뷰 | 프로세스 부모-자식 계층을 트리로 보여주는가 |
| **XPLAT** | 크로스플랫폼 | macOS·Linux·**Windows** 세 곳 모두 동작하는가 |
| **T+C** | TUI+CLI 둘 다 | 대화형 TUI **와** 스크립트용 일회성 CLI를 **둘 다** 제공하는가 |
| **BIN** | 단일 바이너리 설치 | 인터프리터/런타임 VM 없이 단일 실행파일로 간편 설치되는가 |
| **탐지 범위** | scope | 무엇을 보는가 — 시스템 전체 / 컨테이너 / 자기 실행분 / 서버 선별 |

> **탐지 범위(scope) 보충:** 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor)는 **시스템 전체
> 프로세스**, `ctop`은 **컨테이너만**, `pm2`/`systemd`/`launchd`는 **자신이 실행/구성한 것만** 본다.
> ntop은 **여러 런타임의 "서버 프로세스"만 선별**해 본다(규칙에 안 맞는 프로세스는 표시하지 않음 —
> 도시에 §2). 이 범위 차이가 포지셔닝의 핵심이다(§4).

### 3-2. 매트릭스 (✓ 한다 / △ 부분·조건부 / ✗ 안 한다)

```
도구                | RT  | FW  | PORT| KILL| TREE|XPLAT| T+C | BIN |  탐지 범위
--------------------+-----+-----+-----+-----+-----+-----+-----+-----+-----------------
▶ ntop              |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  | 서버 프로세스(선별)
--------------------+-----+-----+-----+-----+-----+-----+-----+-----+-----------------
htop                |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  △  |  ✗  |  ✓  | 시스템 전체
btop                |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  △  |  ✗  |  ✓  | 시스템 전체
top                 |  ✗  |  ✗  |  ✗  |  △  |  △  |  △  |  △  |  ✓  | 시스템 전체
glances             |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  ✓  |  ✓  |  ✗  | 시스템 전체
gotop               |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  △  |  ✗  |  ✓  | 시스템 전체
Activity Monitor    |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  ✗  |  ✗  |  ✓  | 시스템 전체(GUI)
ctop                |  ✗  |  ✗  |  △  |  △  |  ✗  |  △  |  ✗  |  ✓  | 컨테이너만
pm2                 |  △  |  ✗  |  ✗  |  ✓  |  ✗  |  ✓  |  ✓  |  ✗  | 자신이 실행한 앱
systemd             |  ✗  |  ✗  |  ✗  |  ✓  |  △  |  ✗  |  △  |  △  | 구성된 유닛(Linux)
launchd             |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  ✗  |  ✗  |  △  | 구성된 잡(macOS)
```

**셀 판정 근거(주요 △/✗ 설명):**
- `htop/btop/top/gotop/glances/Activity Monitor` **RT·FW = ✗**: 프로세스 이름 문자열만 보여줄 뿐
  "Node 런타임의 Next.js 서버"라는 **의미 분류가 없다**. 이것이 §4 포지셔닝의 가로축 격차다.
- `KILL`: htop/btop/top/gotop/glances/launchd = **△**(시그널은 보내지만 **정상→강제 자동
  에스컬레이션·트리 종료가 없거나 제한적**). Activity Monitor = △(Quit/Force Quit 2종만). ctop =
  △(컨테이너 stop/kill). **pm2/systemd = ✓**(정상→강제 에스컬레이션·라이프사이클 제어가 강함, 단
  범위가 좁음). **ntop = ✓**(임의 시그널 6종[Unix]/3종[Win] + `graceful_kill` 자동 에스컬레이션 +
  `kill_tree` 역순 종료 — 도시에 §5).
- `PORT`: ctop = △(컨테이너 published 포트만). 나머지 범용 모니터 = ✗. **ntop = ✓**
  (`NetworkInspector`로 LISTEN 포트를 PID별 매핑 — 도시에 §4-1).
- `XPLAT`: glances·pm2 = ✓(Win 포함), htop·btop·top·gotop·ctop = △(Win 네이티브 미흡/WSL 필요),
  Activity Monitor·launchd = ✗(macOS 전용), systemd = ✗(Linux 전용). **ntop = ✓**(mac·Linux·Win,
  `platform.rs` + `windows-sys` — 도시에 §4-2).
- `T+C`: top = △(배치 모드 `top -b`), systemd = △(CLI 강력 + `systemd-cgtop` 별도 도구). glances·pm2
  = ✓. **ntop = ✓**(TUI 기본 + `list`/`kill`/`info`/`log`/`config` 서브커맨드, `--json`/`--format csv`
  — 도시에 §7).
- `BIN`: glances·pm2 = ✗(파이썬/Node 런타임 필요). systemd·launchd = △(OS 구성요소라 "설치" 개념이
  아님). **ntop = ✓**(Rust 단일 바이너리: `cargo install`/Homebrew/릴리스 바이너리 — 도시에 §11).
- `pm2 RT = △`: Node 전제로 동작하나 임의 프로세스를 발견·분류하진 않는다.

**매트릭스가 말하는 것:** **RT·FW·PORT 세 열이 동시에 ✓인 도구는 ntop뿐이다.** 범용 모니터는
RT·FW·PORT가 모두 ✗, 매니저(pm2/systemd)는 KILL은 강하지만 RT·FW·PORT가 비어 있고 범위가 좁다.

### 3-3. 경쟁자 버전 주석 (M-4 — 검증성 보강)

> 평가자 비고(S1-비고1)를 반영해 **판정 기준이 된 대략적 도구 계열**을 부기한다. 외부 도구 사실은
> 도시에로 역추적할 수 없으므로, 세부 기능은 **보수적으로(없는/제한적인 쪽으로)** 판정했다. 버전은
> "대략/계열" 표기이며, 실제 동작은 버전·빌드·OS에 따라 달라질 수 있다.

| 도구 | 판정 기준 버전(대략) | 비고 |
|---|---|---|
| **htop** | 3.x 계열 | F5 트리·F9 시그널 메뉴 기준. 자동 에스컬레이션은 없음(△). |
| **btop (btop++)** | 1.x 계열(C++ 재작성판) | 실시간 그래프 강점. Windows 지원은 비교적 최근/제한적이라 XPLAT=△. |
| **top** | Linux `procps-ng` 4.x / macOS BSD `top` | 트리·kill 동작이 OS마다 들쭉날쭉(macOS top은 트리·kill 없음) → TREE=△. |
| **glances** | 3.x~4.x 계열(Python) | 크로스플랫폼·웹/REST·익스포트는 강하나 단일 바이너리 아님(BIN=✗). |
| **gotop** | 4.x 계열(현재 유지보수 중심) | 단일 바이너리(Go)이나 포트 매핑·트리 없음. |
| **Activity Monitor** | macOS 번들(버전=OS 버전) | GUI 전용·macOS 전용. ntop의 `phys_footprint` 메모리가 이 값과 일치(도시에 §4-2). |
| **ctop** | 0.7.x 계열 | 컨테이너 범위. published 포트만 표시(PORT=△). |
| **pm2** | 5.x 계열(Node) | graceful + kill_timeout 에스컬레이션·자동 재시작 강점, 단 자기 실행 앱만. |
| **systemd** | 250+ 계열(주요 배포판) | cgroup 트리 종료·자동 에스컬레이션 강점, 단 구성 유닛만·Linux 전용. |
| **launchd** | macOS 번들(버전=OS 버전) | 구성 잡만·macOS 전용·통합 TUI 없음(CLI만). |
| **▶ ntop** | **0.2.0** | 본 패키지의 역설계 대상. 모든 셀은 도시에/코드로 역추적 가능. |

## 4. 포지셔닝 쿼드런트 (M-5)

두 축으로 도구들을 배치한다.
- **가로축(X) = 런타임/프레임워크 "의미 인지"**: 프로세스를 PID·이름으로만 보는가(왼쪽) →
  "어느 런타임의 어느 프레임워크 서버"로 이해하는가(오른쪽).
- **세로축(Y) = 프로세스 "제어 깊이"**: 보기만 하는가(아래) → 임의 시그널·정상/강제·트리·
  라이프사이클 제어까지 하는가(위).

```
제어 깊이
(높음)
  ▲
  │  systemd            pm2                          ▶ ntop
  │  (구성 유닛/         (Node 앱/                   (시그널6·graceful
  │   cgroup 트리,       graceful reload,             →force·tree kill,
  │   graceful→force)    감독·재시작)                 8런타임·FW식별)
  │
  │                          launchd
  │  ──────────────────────────(구성 잡)─────────────────────────────
  │       htop          btop
  │     (F9 시그널,    (시그널,
  │      트리 F5)       이력 그래프)        ctop
  │                                        (컨테이너 stop/kill,
  │   Activity Monitor                      published 포트)
  │   (Quit/Force,
  │    계층 보기)
  │      gotop      glances
  │     (kill)     (k kill, 웹/API)
  │       top
  │     (k/ V)
  └────────────────────────────────────────────────────────────────────▶
   (낮음)        런타임/프레임워크 의미 인지              (높음)
   "익명 PID·이름만"                              "Next.js/FastAPI 수준 식별"
```

**읽는 법:**
- **오른쪽 위(고-의미인지 × 고-제어) = ntop이 거의 독점.** 의미를 알면서(오른쪽) 동시에 안전하게
  제어(위)하는 도구가 ntop이다.
- **왼쪽 위 = pm2·systemd·launchd**: 제어는 깊지만 의미 인지는 낮고(런타임 분류·프레임워크 식별
  없음) **범위가 좁다**(자신이 관리/구성한 것만).
- **왼쪽 가운데~아래 = 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor)**: 의미 인지가 거의
  없고 제어도 시그널 전송 수준에 머문다.
- **ctop**은 사실상 **다른 평면(컨테이너)**에 있어 좌표상 한 점으로 두지만 직접 경쟁이 아니다.

**핵심 통찰:** 경쟁자들은 모두 **가로축 왼쪽**(의미 인지 낮음)에 몰려 있다. ntop의 차별화는 이
**가로축(의미 인지)을 오른쪽으로 끌고 가면서 세로축(제어)도 높게 유지**한 데 있다. "더 편리"가
아니라 **축이 다르다.**

## 5. 시장 / 사용자 맥락 (M-6, 정성)

> ntop은 **무료 OSS 도구**이므로 매출 기반 시장 규모(TAM/SAM/SOM 금액)를 산정하지 않는다. 대신
> spec/playbook 지시에 따라 **"채택 가능 사용자 모집단/세그먼트"를 정성적으로** 추정한다.

### 5-1. 시장 흐름 — 왜 지금 이 도구가 의미 있나

1. **폴리글랏(polyglot, 한 프로젝트에 여러 언어) 로컬 개발의 일상화.** 프런트엔드는 Node
   (Next.js/Nuxt), 백엔드는 Python(FastAPI/Django)·Java(Spring Boot)·.NET, 거기에 워커·큐·캐시까지 —
   한 개발자가 **한 머신에서 여러 런타임의 서버를 동시에** 띄우는 것이 표준이 됐다.
2. **한 머신에 동종 프로세스 더미.** 특히 `node` 프로세스는 dev 서버·번들러·LSP(에디터 자동완성
   서버)·MCP 서버·CLI 도구가 모두 같은 이름으로 떠 있어, 범용 모니터에서는 **구분이 불가능한 익명
   더미**가 된다.
3. **AI 도구·MCP 서버의 폭증.** npx로 띄우는 MCP(AI 비서가 외부 도구에 붙는 표준 연결)/도구 프로세스가
   늘면서, "건드리면 안 되는 도구 프로세스 vs 죽여도 되는 dev 서버"를 가려야 할 필요가 커졌다.
   (ntop의 프로세스-로컬 전용 탐지 원칙은 바로 이 오분류를 막기 위한 설계 — 도시에 §2.)
4. **터미널 우선·키보드 중심 워크플로의 부활.** TUI 모니터(htop·btop)의 인기가 보여주듯, 개발자는
   GUI보다 터미널에서 빠르게 점검·조치하길 원한다. ntop은 이 흐름 위에 **의미 인지**를 얹었다.

### 5-2. 채택 가능 사용자 모집단/세그먼트 (정성, 깔때기형)

```
[ 모집단 — 넓음 ]
 터미널을 쓰는 모든 개발자 (htop/top을 한 번이라도 써 본 사람)
        │  ↓ 좁힘: "한 머신에서 여러 런타임 서버를 띄운다"
[ 핵심 세그먼트 ]
 ① 폴리글랏 풀스택 개발자  — Node 프런트 + Python/Java 백 동시 구동
 ② 마이크로서비스 로컬 개발자 — 한 머신에서 여러 서비스 프로세스
 ③ AI/도구 개발자          — npx/MCP·CLI 프로세스를 다수 구동(오분류 회피가 중요)
        │  ↓ 좁힘: "익명 프로세스 더미 / 엉뚱한 kill에 데인 경험"
[ 1차 채택자 (early adopter) ]
 ④ Rust·TUI 선호 개발자    — 단일 바이너리·키보드 워크플로 애호
 ⑤ 다수 서비스 점검 운영자  — 빠른 식별·안전 종료가 잦은 작업
```

- **세그먼트 우선순위 근거:** ①·③이 "의미 격차" 통증이 가장 크고 빈도가 높다(매일·세션마다 발생).
  이들이 `03-personas`의 P1(폴리글랏)·P3(AI 도구) 페르소나에 대응한다.
- **모집단 추정의 성격:** 금액이 아니라 **"폴리글랏 로컬 개발 × 터미널 사용 × 다중 런타임"의
  교집합**으로 좁힌 정성 추정이다. 수치 목표(Stars/다운로드 등)는 `00-business-model`/`10-prd`에서
  **"가설"로** 다룬다(여기서 산정하지 않음).

## 6. 갭 / 기회 (M-7 전반부)

### 6-1. ntop이 이기는 지점 (기회)

| 강점 | 근거(도시에) | 어느 경쟁자 대비 |
|---|---|---|
| **런타임 인지 분류**(8종) | §2 RUNTIME_RULES, Runtime enum | 범용 모니터 전부(RT=✗) |
| **프레임워크 식별** | §2 FRAMEWORK_RULES(Next/FastAPI/Spring Boot…) | **모든 경쟁자(FW=✗)** — ntop 독점 |
| **포트→프로세스 매핑 내장** | §4-1 NetworkInspector(LISTEN) | 범용 모니터(PORT=✗) |
| **안전한 종료 관리** | §5 KillSignal·graceful_kill·kill_tree | htop/btop/top/glances(△, 에스컬레이션·트리 약함) |
| **크로스플랫폼 + 단일 바이너리 + TUI/CLI 동시** | §7·§11, Cargo.toml | 어느 한 도구도 셋을 동시 충족 못 함 |
| **정확한 메모리(macOS phys_footprint)** | §4-2 (Activity Monitor 일치) | sysinfo RSS 과소보고 회피 |

### 6-2. ntop이 약한/비어 있는 지점 (한계)

| 약점 | 근거(도시에) | 우위 경쟁자 |
|---|---|---|
| **시스템 전역 그래프·실시간 이력 없음** | 프로세스 선별 중심, 이력 차트 모듈 없음 | btop·glances·gotop |
| **프로세스 감독/재시작/데몬화 없음** | ntop은 죽이는 도구(killer.rs), 살려두는 supervisor 아님 | **pm2·systemd**(자동 재시작) |
| **외부 명령 의존** | §12 lsof/netstat 없으면 포트 빈 결과 | (자체 구현 도구 대비 취약) |
| **Windows 시그널 제한** | §12 HUP/USR1/USR2 없음(Win 3종) | (Unix 환경에선 비문제) |
| **로그 캡처 한계** | §12 cwd glob + Linux `/proc/<pid>/fd/1` fallback만 | pm2(통합 로그 관리) |
| **프레임워크 규칙 공백** | §12 Express/Fastify/Koa/Hapi enum만 있고 규칙 미등록 | (로드맵 근거 — `40-backlog`) |
| **컨테이너 비대상** | 호스트 프로세스 범위 | ctop(컨테이너) |
| **GUI 아님** | 터미널 전용 | Activity Monitor(GUI 선호자) |

> 이 한계들은 **로드맵 근거**이자(`40-backlog`, 도시에 §12), 동시에 **포지셔닝의 선택**이다 —
> ntop은 "전역 모니터"가 아니라 "**서버 의미 인지 + 안전 종료**"에 집중한 도구다.

## 7. 차별화 모방난이도(해자/moat) 논거 (M-7 후반부)

> spec/playbook은 "더 편리/더 빠름" 같은 일반론을 **실패**로 규정한다. 따라서 차별화는 **실명 경쟁자
> 대비 명시 축 + 모방난이도 논거**로 제시한다. 이 §7은 `01-service-plan` 차별화 섹션의 **단일 출처**다.

**명시 비교 한 줄 정리:** *htop·btop·top·glances·gotop·Activity Monitor*는 **RT·FW·PORT 축이 모두 ✗**
이고, *pm2·systemd·launchd*는 **KILL은 ✓지만 RT/FW/PORT가 비고 범위가 좁다.* **세 축(RT·FW·PORT)이
동시에 ✓인 도구는 ntop뿐이다**(§3 매트릭스).

이 차별점이 **쉽게 복제되지 않는** 이유 — 모방난이도 3가지:

```
   ┌──────────────────────────────────────────────────────────────────────┐
   │  ntop의 해자 = ① × ② × ③ (단일 기능 아님 — 설계 결정의 곱)            │
   ├──────────────────────────────────────────────────────────────────────┤
   │  ① 데이터 주도 규칙 테이블      ② 프로세스-로컬 전용 탐지              │
   │     (축적된 경험 사전)             (비자명한 설계 결정)                │
   │     node_modules/.bin/next,       이름+커맨드라인만 본다.             │
   │     -m flask, bin/rails,          package.json을 절대 안 읽음 →       │
   │     org.springframework,          npx 전역 도구의 cwd 오분류 함정     │
   │     normalize_name(comm 16자)…    을 의도적으로 회피                  │
   │            │                              │                           │
   │            └──────────────┬───────────────┘                           │
   │                           ▼                                           │
   │            ③ 복리로 커지는 확장 구조                                  │
   │               새 런타임/FW = `Rule` 1개 + enum variant 1개            │
   │               → 기여가 쌓일수록 사전이 복리로 성장                    │
   │               → 후발 복제자는 "움직이는 목표"를 상대                  │
   └──────────────────────────────────────────────────────────────────────┘
```

1. **데이터 주도 규칙 테이블 = 축적된 경험 사전.**
   `FRAMEWORK_RULES`/`RUNTIME_RULES`는 "각 프레임워크/런타임이 **프로세스 이름·커맨드라인에
   드러나는 방식**"을 수집한 사전이다 — `node_modules/.bin/next`, `-m flask`, `bin/rails`,
   `org.springframework`, macOS의 `comm` 16자 절단을 푸는 `normalize_name`(도시에 §2)… 이건 기능
   하나가 아니라 **현장에서 하나씩 검증해 쌓은 경험치**다. 범용 모니터가 이를 "한 번에 추가"할 수
   없는 이유다.

2. **프로세스-로컬 전용 탐지 원칙 = 비자명한 설계 결정.**
   ntop은 분류에 **파일시스템(`package.json`)을 절대 읽지 않는다**(도시에 §2). 순진하게 복제하려는
   쪽은 당연히 cwd의 `package.json`을 읽으려 하고, 그 순간 **전역 실행 프로세스(npx MCP 서버, CLI
   도구)를 상속된 cwd로 오분류**하는 함정에 빠진다. ntop의 정확성은 이 함정을 **의도적으로 피한**
   결과이며, 같은 통찰 없이는 재현하기 어렵다. (그래서 spec은 fs 기반 탐지 제안 자체를 금지한다.)

3. **확장 구조가 해자를 시간에 따라 키운다.**
   새 런타임/프레임워크는 **`Rule` 1개 + enum variant 1개**만 추가하면 되고 다른 코드 변경이
   불필요하다(도시에 §2). 즉 기여가 쌓일수록 사전이 **복리로** 커진다 — 후발 복제자는 매번 격차를
   따라잡아야 하는 움직이는 목표를 상대하게 된다.

**결론:** ntop의 해자는 "단일 기능"이 아니라 **① 경험이 응축된 데이터 사전 × ② 비자명한 탐지 설계
원칙 × ③ 복리로 커지는 확장 구조**의 결합이다. 이는 범용 모니터가 분류 계층 없이 흉내 낼 수 없고,
프로세스 매니저가 발견·분류 범위를 넓히지 않고는 도달할 수 없는 위치다.
