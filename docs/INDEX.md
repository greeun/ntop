# INDEX — ntop 기획 패키지 안내 (무엇을 / 누가 / 왜 먼저 읽나)

> `ntop`("Node Top") = 여러 런타임(Node·Python·Java·Deno·Bun·Ruby·PHP·.NET)의 서버 프로세스를
> 한 화면에서 모니터링·종료하는 **Rust TUI+CLI 단일 바이너리**(MIT OSS, v0.2.0).
> 이 패키지는 **이미 구현된 코드를 역설계**해 16종 기획 문서로 정리한 것이다 — "앞으로 만들 것"이 아니라
> "이미 동작하는 것"의 가치·범위·명세·정책·로드맵이다.
>
> 본 INDEX는 **신규 내용을 만들지 않는다.** 16문서를 ① 중요도 3티어 ② 작업(읽기) 순서 ③ 역할별 번들
> ④ 최소 시작 세트로 **분류·정렬**만 한다. (HTML 허브 `index.html`이 이 분류를 그대로 시각화한다.)
> 각 문서는 읽기용 `NN-name.html`로도 제공된다(원본 `.md` 보존). 종합 시각 대시보드는 `overview.html`.

---

## ① 중요도 3티어 (도메인 판정)

판정 기준: **개발팀이 ntop을 이해·유지·확장하는 데 직접 필요한가.** ntop은 안전(프로세스 종료)·프라이버시
(env 마스킹)가 도구 정체성이라 `33-policy`를 **T1로 승격**했다. 결제형이 아니므로 `00`(지속가능성)은
T1 강제가 아니라 후원·지속가능성 설득용(T2).

### 🔴 T1 — 필수 (코어 정체성·구현 정본·안전)
| 문서 | 한줄 목적 | 왜 T1 |
|---|---|---|
| `01-service-plan` | 핵심 가치·MVP 범위·성공지표·차별화 **단일 출처** | 제품 정체성의 기준점. 모든 하위 문서가 참조 |
| `10-prd` | 기능 목록(F1~F21)·우선순위·기능↔문제 추적표·비기능 요구 | 기능 ID 단일 출처. 구현/QA의 출발 |
| `13-user-flows` | 유저 플로우·저니(종료 2-lane 상태기계 포함) | 핵심 동작 흐름. kill 안전 흐름의 정본 시각화 |

| `30-functional-spec` | 동작·알고리즘 **단일 출처**(스캔·분류·종료·CPU 측정) | 구현 정본. "코드가 실제로 무엇을 하나" |

| `33-policy` | **안전·프라이버시 단일 출처**(종료 정책·env 마스킹·권한) | ⬆승격: 파괴적 동작·민감값 보호 = 도구 정체성 |
| `40-backlog` | 완료 백로그(v0.2.0) + 향후 로드맵 + **정직 발견**(버그·죽은 코드) | 현 상태와 다음 작업. 유지보수 진입점 |

## ② 작업(읽기) 순서 — 의존 기반

```
 [정체성]        [문제·맥락]        [구조]          [구현 정본]         [실행]
 01 ──────┬───▶ 03 ─▶ 02 ─▶ 11 ─▶ 12 ─▶ 13 ─▶ 20 ─▶ 21 ─▶ 30 ─┬─▶ 31 ─▶ 32 ─▶ 33 ─▶ 40 ─▶ 41
 (가치)   │     (페르소나·경쟁·스토리)  (IA·플로우·화면)        (동작) │   (모델·계약·정책)   (로드맵·QA)
 00 ──────┘ (지속가능성: 01과 함께 또는 후순위)                       └ 30이 31/32/33의 동작 근거
```

1. **`01`**(+필요시 `00`) — 무엇을 왜 만드는가(가치·MVP·지표·차별화).
2. **`03` → `02` → `11`** — 누구의 어떤 문제를, 경쟁 대비 어떻게(문제·맥락).
3. **`12` → `13` → `20` → `21`** — 구조·흐름·화면(IA→플로우→시안→명세).
4. **`30` → `31` → `32` → `33`** — 동작 정본 → 데이터 모델 → 인터페이스 계약 → 안전/정책.
5. **`40` → `41`** — 무엇이 남았나(로드맵·정직 발견) → 어떻게 검증하나(QA).

> 핵심 의존: `01`(가치)·`10`(기능 ID)·`30`(동작)·`31`(Config)·`32`(계약)·`33`(정책)·`21`(키)는 **단일 출처**다.
> 나머지 문서는 이들을 **참조**하며 재정의하지 않는다(중복 정의 0).

## ③ 역할별 번들 (그 역할이 먼저 볼 문서)

| 역할 | 필수 | 보조 |
|---|---|---|
| 👩‍💻 **개발(코어/기여자)** | `30` 동작 · `31` 모델/Config · `32` 계약 · `10` 기능 · `40` 로드맵·정직발견 | `01` · `13` · `33` |
| 🧪 **QA** | `41` 테스트케이스 · `13` 플로우 · `30` 동작 · `33` 정책(예외·권한) | `10` 추적표 · `21` 키 |
| 🎨 **디자인(TUI/UX)** | `21` 화면명세·키 · `20` 와이어프레임 · `12` IA · `13` 플로우 | `03` 페르소나 |
| 🧭 **PO/기획** | `01` 가치·MVP·지표 · `10` 우선순위 · `02` 경쟁 · `03` 페르소나 | `00` · `40` |
| 🛠 **메인테이너/후원검토** | `40` 로드맵·정직발견 · `33` 안전/프라이버시 · `00` 지속가능성 | `01` · `30` |

## ④ 최소 시작 세트 (처음 6개만)

> ntop을 빠르게 파악하려면 이 6개로 충분하다. 나머지는 필요할 때 펼친다.

```
1. 01-service-plan   ← 무엇을 왜 (가치·MVP·지표·차별화)
2. 10-prd            ← 기능 목록·우선순위·문제 추적
3. 30-functional-spec← 실제 동작·알고리즘 정본
4. 31-erd            ← 데이터 모델·Config 기본값
5. 21-screen-spec    ← 화면·키 바인딩(상호작용 계약)
6. 40-backlog        ← 현 상태·다음 작업·정직 발견
```

핵심 안전 동작 1줄: **TUI 종료는 SIGTERM 1회 직접 전송**(폴링·자동 강제승격 없음), **자동 graceful→force
에스컬레이션은 CLI `ntop kill` 단건 전용**(정본 `30`·정책 `33`). 이 구분을 먼저 이해하면 종료 관련 문서가
모두 일관되게 읽힌다.
