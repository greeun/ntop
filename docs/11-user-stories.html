11 · 유저 스토리 / 요구사항 (ntop)

이 문서는 ntop의 유저 스토리(user story) 단일 출처다. 각 스토리는 "~로서, ~하고 싶다, 왜냐하면 ~" 형식 + 수용 기준(Given-When-Then) + 연결 기능 ID + 연결 문제(PRB) 로 구성된다.

참조(재정의 0): 기능 ID(F1~F21)·우선순위는 10-prd, 페르소나(P1 박서연·P2 정태호·P3 이준)는 03-personas, 문제(PRB-1~8)는 research-s2(→03 §2), 화면/키/플로우는 research-s3 (→12·13·21), Config 기본값(graceful_timeout=10 등)은 31-erd 가 단일 출처다. 본 문서는 이들을 인용만 하고 값을 재정의하지 않는다(spec §4-3).

전제: ntop은 이미 v0.2.0까지 구현된 MIT 오픈소스 도구다(codebase-facts §0). 따라서 모든 스토리의 수용 기준은 "앞으로 만들 동작"이 아니라 이미 구현된 실제 동작(도시에)을 기준으로 한다(창작 0).

용어 풀이(첫 등장): 유저 스토리 = 사용자 관점에서 "누가/무엇을/왜"를 한 문장으로 적은 요구 단위 / 수용 기준(Given-When-Then) = 그 스토리가 "완료됐다"고 판정하는 조건(전제 Given → 행동 When → 결과 Then) / Epic(에픽) = 여러 스토리를 묶는 큰 주제 / graceful 종료 = "정리하고 꺼져라" 요청(SIGTERM), 강제 종료(SIGKILL)와 다름 / 트리 종료 = 부모-자식을 묶어 함께 종료 / 헤드리스 (headless) = 화면(GUI) 없이 명령만으로 쓰는 환경(예: SSH 서버) / MCP = AI 비서가 외부 도구에 붙는 표준 연결, 보통 npx로 백그라운드 실행.

1. 스토리 작성 규칙§

  • 형식: US-<에픽>.<번호> · "[페르소나]로서, [원하는 것]하고 싶다, 왜냐하면 [이유]" · 수용 기준(Given-When-Then) · 연결(기능 ID + PRB).
  • 페르소나: 03의 P1(박서연·폴리글랏 풀스택) · P2(정태호·플랫폼/운영) · P3(이준·AI 응용). 03이 단일 출처.
  • 수용 기준은 구현된 사실만. 예: "x로 KillConfirm 후 send_signal(SIGTERM)을 1회 직접 전송(TUI는 폴링·자동 force 없음); 자동 graceful→force(graceful_timeout 기본 10초·31값 초과 시 SIGKILL)는 CLI ntop kill 단건 전용." — 코드(ui.rs L95-108·main.rs)·30 §2-6·§8-3·33 §2-1과 일치.
  • 추적: 모든 스토리는 ≥1 기능 ID와 ≥1 PRB로 역추적된다(G-a). 커버리지는 §9.

4. EP-1 — 프로세스 식별§

주 문제 PRB-1(익명 node 더미)·PRB-3(오분류)·PRB-4(다중런타임). ★ 목록 컬럼에 FRAMEWORK 없음 — 프레임워크 정체는 상세 Info 탭 또는 CLI list 테이블의 FRAMEWORK 컬럼에서만 확인(코드-진실, 10-prd §0).

US-1.1 — 의미로 식별하기§

  • P1 박서연(폴리글랏 풀스택)로서, 떠 있는 여러 node 중 어느 것이 어느 프레임워크·포트의 서버인지 한 화면에서 즉시 보고 싶다, 왜냐하면 매 세션 식별에 1~3분을 쓰고 그게 오kill의 원인이기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given Next.js dev(:3000)·FastAPI(:8000)·MCP 등 서버 프로세스가 떠 있고,<br><span style="color:#64748b">↳ </span>When ntop을 실행하면,<br><span style="color:#64748b">↳ </span>Then 위 목록(V-LIST)에 각 프로세스의 NAME(display_name)·PORT 등이 표시되고, 행을 선택해 Tab으로 아래 상세 Info 탭(TAB-INFO)을 보면 Framework(예: "Next.js (Node)") 가 노출된다.<br><span style="color:#64748b">↳ </span>And 규칙에 안 맞는 비서버 프로세스는 목록에 뜨지 않는다(runtime None=미표시).
  • 연결: 기능 F1·F2·F5·F15 / 문제 PRB-1·PRB-2

US-1.2 — npx/MCP 오분류 회피§

  • P3 이준(AI 응용)로서, npx로 전역 실행된 MCP/도구를 dev 서버로 오분류하지 않길 원한다, 왜냐하면 작업 디렉터리(cwd)가 제각각이라 cwd나 package.json으로 추정하면 반드시 틀리기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given npx로 띄운 MCP 서버 여러 개와 실제 dev 서버가 섞여 있고(node 10+개),<br><span style="color:#64748b">↳ </span>When ntop이 이들을 분류하면,<br><span style="color:#64748b">↳ </span>Then 분류는 이름+커맨드라인만 사용하고 파일시스템(package.json)을 읽지 않으며, 상속된 cwd로 인한 오분류가 발생하지 않는다.<br><span style="color:#64748b">↳ </span>And 규칙에 매칭되지 않는 프로세스는 "죽일 후보 풀"에 아예 안 떠, MCP를 dev로 착각할 여지가 준다.
  • 연결: 기능 F1·F3 / 문제 PRB-1·PRB-3

US-1.3 — 이름·포트로 빠르게 좁히기§

  • P1 박서연로서, 떠 있는 프로세스가 많을 때 이름/포트로 빠르게 좁혀 보고 싶다, 왜냐하면 목록이 길면 대상 찾기가 또 다른 수작업이 되기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 목록에 다수 프로세스가 있고,<br><span style="color:#64748b">↳ </span>When /를 눌러 필터 모드로 들어가 "next"를 입력하면,<br><span style="color:#64748b">↳ </span>Then name·command·pid·framework·runtime·ports 중 하나라도 부분일치(대소문자 무시) 하는 행만 남고, 입력 한 글자마다 즉시 갱신된다(다음 스캔 tick을 기다리지 않음).<br><span style="color:#64748b">↳ </span>And Enter는 필터를 유지하고, Esc는 필터를 비우고 모드를 빠져나온다.
  • 연결: 기능 F11 / 문제 PRB-1·PRB-4

US-1.4 — 정렬로 우선순위 보기§

  • P1 박서연로서, CPU/메모리/포트 등 기준으로 정렬해 보고 싶다, 왜냐하면 "지금 뭐가 무겁나/ 어느 포트가 떴나"를 빨리 가려야 하기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 목록이 표시된 상태에서,<br><span style="color:#64748b">↳ </span>When s를 반복해 정렬 컬럼을 순환(PID→NAME→PORT→THR→CPU→MEM→USER→STATUS→UPTIME)하고 r로 방향을 반전하면,<br><span style="color:#64748b">↳ </span>Then 목록이 즉시 해당 기준으로 재정렬되고 헤더에 정렬 방향 표시자(↑/↓)가 보인다.
  • 연결: 기능 F12 / 문제 PRB-1

US-1.5 — 부모-자식 구조 보기§

  • P1 박서연로서, "이 node가 dev 서버인지, 그 서버가 띄운 자식 번들러인지"를 트리로 보고 싶다, 왜냐하면 부모-자식 관계를 알아야 무엇을 죽일지 판단되기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 서버와 그 부모(launchd/claude/shell)·자식이 함께 스캔되고(2-pass),<br><span style="color:#64748b">↳ </span>When 목록에서 Enter(토글)·→/l(확장)·←/h(축소) 또는 e(전체 토글)를 누르면,<br><span style="color:#64748b">↳ </span>Then 트리 노드가 들여쓰기와 함께 펼쳐지거나 접히고, 첫 로드 시에는 전체가 펼쳐진 상태로 시작한다.
  • 연결: 기능 F10 / 문제 PRB-1·PRB-4

5. EP-2 — 안전한 종료§

주 문제 PRB-3(오kill)·PRB-5(거친 종료·고아). 모든 파괴적 동작은 확인 게이트(기본 on)와 graceful 우선으로 감싸진다(도시에 §3·§5·§7).

US-2.1 — 확인 후 graceful 종료§

  • P1 박서연로서, 포트로 식별한 dev 서버를 확인 한 번 거쳐 정상 종료하고 싶다, 왜냐하면 kill -9 습관은 임시상태 유실 위험이 있고 가끔 엉뚱한 프로세스를 죽이기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 종료할 프로세스를 선택한 상태에서,<br><span style="color:#64748b">↳ </span>When x를 눌러 KillConfirm(DLG-KILL)에서 Enter로 확인하면,<br><span style="color:#64748b">↳ </span>Then send_signal(pid, SIGTERM)이 1회 직접 전송되고(TUI는 폴링 없음 — ui.rs L95-108), needs_rescan=true로 다음 스캔에서 종료가 목록에 반영된다.<br><span style="color:#64748b">↳ </span>And 권한 부족이면 PermissionDenied 결과 메시지를 표시한다.<br><span style="color:#64748b">↳ </span>And Esc로 취소하면 어떤 시그널도 전송되지 않는다.<br><span style="color:#64748b">↳ </span>참고(코드-진실 — 정규 출처 30 §2-6·§8-3 / 정책 33 §2-1): 강제 종료 자동 승격(graceful→force)은 CLI ntop kill 단건 전용이다(main.rs graceful_kill: SIGTERM→200ms 폴링→graceful_timeout(기본 10초·31값) 초과 시 자동 SIGKILL). TUI KillConfirm은 이 자동 승격을 쓰지 않는다. TUI 강제 모달 ForceKillPrompt(DLG-FORCE) 핸들러는 코드에 존재하나 현재 트리거 경로가 없는 미도달 코드다(40 M1).
  • 연결: 기능 F7 / 문제 PRB-3·PRB-5

US-2.2 — 트리 종료로 고아 방지§

  • P2 정태호(플랫폼/운영)로서, 부모만 죽여 자식이 고아로 남는 사고를 막고 싶다, 왜냐하면 고아 워커가 포트를 계속 잡아 같은 인시던트가 재호출되기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 부모+자식으로 이뤄진 프로세스 트리를 선택하고,<br><span style="color:#64748b">↳ </span>When K로 KillTreeConfirm(DLG-KILLTREE)에서 Enter로 확인하면,<br><span style="color:#64748b">↳ </span>Then kill_tree가 트리 PID를 자식부터 역순으로 SIGTERM 전송해, 자식이 먼저 정리된 뒤 부모가 정리되어 고아 프로세스가 남지 않는다.
  • 연결: 기능 F8 / 문제 PRB-5

US-2.3 — 임의 시그널 정밀 제어§

  • P1 박서연로서, 죽이지 않고 설정 리로드(SIGHUP)나 디버거 활성(SIGUSR1) 같은 정밀 시그널을 보내고 싶다, 왜냐하면 모든 제어가 "종료"인 것은 아니기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 대상 프로세스를 선택한 상태에서,<br><span style="color:#64748b">↳ </span>When S로 SignalPicker(DLG-SIGNAL)를 열어 시그널을 고르고 Enter로 보내면,<br><span style="color:#64748b">↳ </span>Then 선택한 시그널이 전송된다. 목록은 Unix 6종(TERM/KILL/HUP/INT/USR1/USR2), Windows 3종 (TERM/KILL/INT)이며 각 항목에 설명이 붙는다(예: HUP="reload", USR1="Node.js 디버거 활성").
  • 연결: 기능 F9 / 문제 PRB-5

US-2.4 — 여러 대상 한 번에 선택§

  • P1 박서연로서, 정리할 프로세스 여러 개를 미리 체크해 두고 싶다, 왜냐하면 재시작 때 dev 서버·번들러를 한 묶음으로 다루는 일이 잦기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 목록에 여러 프로세스가 있고,<br><span style="color:#64748b">↳ </span>When 각 행에서 Space를 누르면,<br><span style="color:#64748b">↳ </span>Then 해당 PID가 다중 선택 집합(selected_pids)에 토글되어 체크 표시가 보인다.
  • 연결: 기능 F14 / 문제 PRB-4

US-2.5 — 안전 기본값 신뢰§

  • P1 박서연로서, 별도 설정 없이도 기본값이 안전하길 원한다, 왜냐하면 매번 옵션을 신경 쓰면 실수가 늘기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 설정 파일이 없거나 일부만 있을 때,<br><span style="color:#64748b">↳ </span>When ntop이 설정을 읽으면,<br><span style="color:#64748b">↳ </span>Then confirm_before_kill=true(확인 게이트 on), default_signal="SIGTERM"(graceful 우선), graceful_timeout=10초, mask_env_values=true 등 안전 기본값으로 동작한다(값 정규 출처 31).
  • 연결: 기능 F21·F7 / 문제 PRB-3·PRB-5

6. EP-3 — 리소스·상태 점검§

주 문제 PRB-2(포트 역추적)·PRB-6(메모리 오판·민감값 노출). 점검은 상세 4탭(F15) 안에서 완결된다.

US-3.1 — 포트를 그 자리에서§

  • P1 박서연로서, "포트 3000을 누가 잡고 있나"를 lsof 왕복 없이 그 자리에서 보고 싶다, 왜냐하면 lsof→ps→kill 3단계 왕복이 주 10회+ 반복되기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 서버가 포트를 LISTEN 중이고,<br><span style="color:#64748b">↳ </span>When 목록을 보면,<br><span style="color:#64748b">↳ </span>Then 각 행의 PORT 컬럼에 LISTEN 포트가 표시되고(분류 후 NetworkInspector가 PID별로 채움), 상세 Net 탭(TAB-NET)에는 LISTEN 포트 + 활성 TCP 연결이 함께 보인다.<br><span style="color:#64748b">↳ </span>And lsof(Unix)/netstat(Win)가 없는 환경에서는 크래시 없이 빈 결과로 수렴한다.
  • 연결: 기능 F5·F6·F15 / 문제 PRB-2

US-3.2 — 정확한 메모리§

  • P1 박서연로서, macOS에서 Activity Monitor와 일치하는 메모리를 보고 싶다, 왜냐하면 일반 도구의 RSS 값은 유휴 Node를 최대 100배 과소보고해 오판을 부르기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given macOS에서 서버 프로세스가 떠 있고,<br><span style="color:#64748b">↳ </span>When 상세 Info 탭의 Memory를 보면,<br><span style="color:#64748b">↳ </span>Then phys_footprint(Activity Monitor와 일치) 기반 값이 표시되어 동일 PID의 Activity Monitor 값과 육안 비교로 일치한다.
  • 연결: 기능 F18·F15 / 문제 PRB-6

US-3.3 — 환경변수 안전 점검§

  • P1 박서연로서, 프로세스의 환경변수를 보되 비밀번호·토큰은 가려진 채로 보고 싶다, 왜냐하면 화면 공유·어깨너머로 민감값이 노출될 수 있기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 서버 프로세스에 PASSWORD/SECRET/TOKEN 등이 포함된 환경변수가 있고,<br><span style="color:#64748b">↳ </span>When 상세 Env 탭(TAB-ENV)을 보면,<br><span style="color:#64748b">↳ </span>Then mask_env_values(기본 true)에 따라 민감 키의 값이 마스킹(예: ****)되어 표시된다.
  • 연결: 기능 F17·F15 / 문제 PRB-6

US-3.4 — 로그를 도구 안에서§

  • P1 박서연로서, 서버 로그를 별도 터미널 없이 도구 안에서 보고 싶다, 왜냐하면 "이게 정상 동작 중인지" 점검에 로그가 빠르기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 프로세스 cwd에 로그파일(예: *.log, .next/.../*.log)이 있고,<br><span style="color:#64748b">↳ </span>When 상세 Log 탭(TAB-LOG)을 보거나 ntop log <PID>를 실행하면,<br><span style="color:#64748b">↳ </span>Then LogStreamer가 패턴으로 가장 최근 수정 로그를 자동 감지해 끝에서부터 새 줄을 실시간 tail 하고(버퍼 최대 1000줄, 31 정규값), 프로세스가 죽으면 스트림이 종료된다.
  • 연결: 기능 F16·F15 / 문제 PRB-1(점검)

7. EP-4 — 멀티런타임 관리§

주 문제 PRB-4(다중런타임 문맥 전환)·PRB-8(GUI↔터미널 왕복).

US-4.1 — 8런타임 한 화면 횡단§

  • P1 박서연로서, Node·Python·Java 등 여러 런타임 서버를 한 화면·같은 키 조작으로 다루고 싶다, 왜냐하면 런타임마다 점검/종료 절차가 달라 머릿속 모드 전환이 인지 부하이기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given Node·Python(uvicorn/celery)·Java 등 서로 다른 런타임 서버가 떠 있고,<br><span style="color:#64748b">↳ </span>When ntop을 실행하면,<br><span style="color:#64748b">↳ </span>Then 8런타임(Node·Python·Java·Deno·Bun·Ruby·PHP·.NET)이 하나의 목록에 분류돼 보이고, 선택·점검·종료가 동일한 키 조작으로 처리된다.
  • 연결: 기능 F4 / 문제 PRB-4

US-4.2 — Node만 집중§

  • P3 이준(AI 응용)로서, 필요할 때 Node 서버만 골라 보고 싶다, 왜냐하면 node 프로세스가 10+개라 다른 런타임이 섞이면 시야가 흐려지기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 여러 런타임 서버가 표시된 상태에서,<br><span style="color:#64748b">↳ </span>When n을 눌러 Node-only 토글을 켜면,<br><span style="color:#64748b">↳ </span>Then Node 서버만(+트리 컨텍스트 부모) 남고 Deno/Bun 등 다른 런타임은 숨겨지며, 상단바 라벨이 Servers↔Nodes로 바뀌고 [Node-only] 표식이 보인다. 다시 n을 누르면 전체로 복귀한다.
  • 연결: 기능 F13 / 문제 PRB-4

US-4.3 — GUI↔터미널 왕복 제거§

  • P1 박서연로서, 식별·점검·종료를 한 터미널 화면에서 끝내고 싶다, 왜냐하면 Activity Monitor와 터미널을 오가는 문맥 전환이 피로하기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given ntop TUI가 떠 있고,<br><span style="color:#64748b">↳ </span>When 위 목록(식별)→아래 상세 4탭(점검)→확인 모달(종료)을 한 화면에서 수행하면,<br><span style="color:#64748b">↳ </span>Then 별도 로그인·모드 선택·설정 마법사 없이 단일 메인 뷰에서 가치 루프가 완결되어 외부 GUI로 나갈 필요가 없다.
  • 연결: 기능 F20·F10·F11 / 문제 PRB-8

8. EP-5 — CLI 자동화§

주 문제 PRB-7(헤드리스·자동화). TUI와 동일 코어(ProcessScanner+NetworkInspector)를 공유한다(도시에 §7).

US-5.1 — JSON으로 현황 추출§

  • P2 정태호(플랫폼/운영)로서, SSH 헤드리스 환경에서 서버 현황을 JSON으로 뽑고 싶다, 왜냐하면 GUI를 쓸 수 없고 모니터링 스크립트에 그대로 붙여야 하기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given GUI 없는 Linux 서버에서,<br><span style="color:#64748b">↳ </span>When ntop list --json(또는 --format csv/table)을 실행하면,<br><span style="color:#64748b">↳ </span>Then 분류된 서버 목록이 출력된다 — JSON은 모든 필드 + depth, CSV는 9컬럼, table은 PID·NAME·FRAMEWORK·PORT·CPU·MEM·UPTIME(출력 스키마 단일 출처 32-api-spec). 1회성 측정(scan_blocking).
  • 연결: 기능 F19 / 문제 PRB-7

US-5.2 — 스크립트에서 트리 종료§

  • P2 정태호로서, 점유 PID를 찾은 뒤 자식까지 한 번에 종료하고 싶다, 왜냐하면 야간 인시던트 대응에서 고아 재점유를 끊어야 하기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given list --json으로 대상 PID를 추출했고,<br><span style="color:#64748b">↳ </span>When ntop kill <PID> --tree [--no-confirm]을 실행하면,<br><span style="color:#64748b">↳ </span>Then 재귀로 트리 PID를 전부 수집해(확인 프롬프트, --no-confirm이면 생략) 각각 graceful (Term→timeout→자동 force)로 종료하고 결과를 출력한 뒤 종료코드를 반환한다.<br><span style="color:#64748b">↳ </span>And --signal <SIG>로 시그널을, --all로 전체 서버 일괄 종료를 지정할 수 있다(PID는 --all 없으면 필수).
  • 연결: 기능 F19·F8 / 문제 PRB-7·PRB-5

US-5.3 — 설정·기본값 확인§

  • P2 정태호로서, 현재 적용 중인 설정과 기본값을 확인하고 싶다, 왜냐하면 자동화 전에 종료 타임아웃·마스킹 같은 동작을 알아야 하기 때문이다.
  • 수용 기준(GWT):<br><span style="color:#64748b">↳ </span>Given 설정 파일이 있거나 없는 상태에서,<br><span style="color:#64748b">↳ </span>When ntop config를 실행하면,<br><span style="color:#64748b">↳ </span>Then 설정 파일 경로(~/.config/ntop/config.toml)와 [general]·[display]·[filter] 값이 출력된다(파일 부재/파싱 실패 시 기본값 — 값 정규 출처 31).
  • 연결: 기능 F21·F19 / 문제 PRB-7(설정 확인)
