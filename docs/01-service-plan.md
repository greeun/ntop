# 01 · 서비스 기획 — 핵심 가치 / MVP / 성공지표 / 차별화 (ntop)

> **이 문서는 단일 출처(single source)다.** ntop의 **핵심 가치·MVP 범위·성공지표·차별화**는 여기서만
> 확정하고, 하위 문서(`10-prd` 기능·우선순위, `40-backlog` 로드맵)는 이 문서를 **참조**한다. 반대로
> 이 문서는 **OSS 지속가능성/수익화/배포 채널은 `00-business-model`**, **경쟁/생태계 분석은
> `02-market-competition`**, **페르소나 상세는 `03-personas`** 를 단일 출처로 참조하며 **재정의하지
> 않는다.**
>
> **전제:** ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스 도구**다(codebase-facts §0). 따라서 이 기획서는
> "앞으로 만들 것"이 아니라 **이미 만들어진 동작을 역설계(reverse-engineering)** 해 가치·범위·지표·
> 차별점으로 정리한 것이다.
>
> **용어 풀이(첫 등장):** **TUI**(Text User Interface) = 터미널 안에서 키보드로 조작하는 전체화면 텍스트
> UI / **CLI**(Command-Line Interface) = 명령 한 줄로 실행해 결과를 출력하는 방식 / **런타임(runtime)** =
> 코드를 실행하는 엔진(Node·Python·Java 등) / **프레임워크(framework)** = 런타임 위에서 서버를 만드는
> 틀(Next.js·FastAPI·Spring Boot 등) / **dev 서버** = 개발 중 코드를 바로 띄워 확인하는 임시 서버 /
> **MCP**(Model Context Protocol) = AI 비서가 외부 도구·데이터에 붙는 표준 연결, 보통 `npx`로 백그라운드
> 실행 / **graceful 종료(정상 종료)** = "정리하고 꺼져라"라고 **요청**(SIGTERM), 강제 종료(SIGKILL)와
> 다름 / **에스컬레이션** = 정상 요청이 안 통하면 강제로 단계 올림 / **트리 종료** = 부모-자식 프로세스를
> 묶어 함께 종료 / **PRB-n** = research-s2가 정의한 문제(problem) 번호 / **MVP**(Minimum Viable Product) =
> 가치를 전달하는 최소 제품 범위.

## 1. 문제 정의 (who-what-why) — 의미 격차

> 출처: research-s2 §1~§3(PRB-1~8). 본 절은 요약이며, 문제 목록의 단일 출처는 research-s2(→ `03-personas`)다.

- **누가:** 한 머신에서 **여러 런타임의 서버 프로세스를 동시에** 띄우는 개발자/운영자.
- **무엇이 문제:** 범용 프로세스 모니터(htop·top·Activity Monitor 등)는 프로세스를 **익명의 PID·CPU·
  메모리**로만 보여줄 뿐, "이 `node`가 **어느 프레임워크의, 어느 포트의, 어느 프로젝트** 서버인가"를
  모른다 — 이것이 **의미 격차(semantic gap)** 다.
- **왜 문제:** 개발자는 그 격차를 `lsof -i :3000`(포트→PID), `ps aux | grep node`(이름 검색),
  Activity Monitor(메모리) **사이를 손으로 오가며** 메우고, 종종 **엉뚱한 프로세스를 죽이는 사고**로
  이어진다(research-s2 §1).

```
 범용 모니터가 보는 것              내가 알아야 하는 것              ← 이 사이가 "의미 격차"
 ┌──────────────────────┐          ┌────────────────────────────┐
 │ PID 41822  node  2.1% │          │ Next.js dev 서버 @:3000      │
 │ PID 41835  node  0.4% │   ???    │ FastAPI @:8000 (백엔드)      │
 │ PID 41840  node  0.0% │  ─────▶  │ Claude Code MCP (건들면 끊김) │
 │ PID 42013  python 5%  │          │ Celery 워커 (큐 처리 중)     │
 └──────────────────────┘          └────────────────────────────┘
   "이름·번호·숫자뿐"                 "프레임워크·포트·역할·안전여부"
```

**하위 문제(PRB) 요약 — G-a 추적의 출발점(research-s2 §3):**

| PRB | 문제(한 줄) | 핵심 통증 |
|---|---|---|
| **PRB-1** | 익명 `node` 더미 — 어느 게 뭔지 모름 | 의미 격차(식별 불가) |
| **PRB-2** | 포트 역추적 수작업(`lsof` 왕복) | 문맥 전환 |
| **PRB-3** | 엉뚱한 프로세스 kill 위험(MCP/도구 vs dev) | 오kill·세션 끊김 |
| **PRB-4** | 다중 런타임 문맥 전환 | 인지 부하 |
| **PRB-5** | 안전하지 않은 종료(`kill -9` 즉사·고아) | 데이터 유실·고아 |
| **PRB-6** | 메모리 오판독 + 환경 민감값 노출 | 오판·유출 |
| **PRB-7** | 헤드리스·자동화 환경에서 점검·종료 어려움 | GUI 불가·스크립트화 |
| **PRB-8** | GUI↔터미널 문맥 전환(Activity Monitor 왕복) | 도구 이동 피로 |

```
      │ (Scan)   │      │(Identify)│      │(Inspect) │      │ (Safe-kill)  │
      └──────────┘      └──────────┘      └──────────┘      └──────────────┘
       2-pass 스캔        런타임/FW 분류    Info/Log/Net/    KillConfirm →
       + 포트 채움        + display_name    Env 4탭          graceful→force
                                                            + 트리 역순
       ─────────────     ─────────────     ─────────────    ──────────────
       PRB-1·2 진입       PRB-1·3·4 해소     PRB-2·6 해소      PRB-3·5 해소
       (event Tick)       (목록·Info 탭)     (상세 탭)        (모달 확인 게이트)
```

**루프가 말하는 것:** ntop의 모든 동작은 **식별 → 점검 → 안전 종료**라는 한 루프 위에 있다(research-s2
§5의 3대 능력군과 1:1). 위험한 분기(종료·시그널)에는 반드시 **모달 확인 또는 graceful 우선**이 끼어
있어, 단계가 군더더기가 아니라 **안전 게이트**다(research-s3 §3·§9). 별도 로그인·모드 선택·설정 마법사가
없어 PRB-8(도구 왕복)을 구조적으로 제거한다.

## 5. MVP 범위 / Out-of-scope (G-e) — v0.2.0 = MVP

> **MVP = 이미 출시된 v0.2.0 범위.** ntop은 역설계 기획이므로 "구현 완료된 v0.2.0이 곧 MVP"다. 향후
> 항목은 `40-backlog`(로드맵, codebase-facts §12 근거)가 단일 출처이며, 여기서는 **포함/제외 경계와
> 우선순위 근거**만 확정한다.

### 5-1. MVP에 포함 (v0.2.0 출시 범위)

| 우선순위 | 기능(핸들) | 우선순위 근거 |
|---|---|---|
| **P0 (코어)** | F-식별분류·F-정확분류·F-포트매핑·F-안전종료·F-트리종료·F-멀티런타임 | 의미 격차(PRB-1~5)를 직접 메우는 가치 루프의 척추 — 없으면 ntop이 아님 |
| **P1 (핵심 보조)** | F-시그널선택·F-필터정렬Node전용·F-네트워크탭·F-Env마스킹·F-정확메모리·F-CLI | 코어를 실사용 가능하게 만드는 점검·자동화·정확성(PRB-4·6·7) |
| **P2 (편의)** | F-로그·F-TUI일원화의 세부(스크롤·도움말·테마) | 점검 편의·발견성, 코어 가치엔 종속 |

### 5-2. Out-of-scope (명시적 제외)

| 제외 항목 | 왜 제외(근거) | 출처 |
|---|---|---|
| **웹 GUI / 데스크톱 앱** | ntop은 터미널 도구(TUI+CLI) — GUI는 설계 정체성 밖, PRB-8(터미널 일원화)과 상충 | spec §7, research-s1 §8-2 |
| **파일시스템(`package.json`) 기반 탐지** | ★**영구 제외(설계 원칙).** fs를 읽으면 npx/전역 프로세스를 상속 cwd로 **오분류** → PRB-3 악화. 이를 안 읽는 것이 해자(§7) | codebase-facts §2, spec §7 |
| **프로세스 감독/자동 재시작/데몬화** | ntop은 "죽이는 도구"이지 "살려두는 supervisor"가 아님 — pm2·systemd 영역 | research-s1 §8-2 |
| **시스템 전역 그래프·실시간 이력 차트** | 전역 모니터(btop·glances)의 영역 — ntop은 "서버 선별 + 안전 종료"에 집중 | research-s1 §8-2 |

## 7. 차별화 (Differentiation, G-c) — 실명 경쟁자 명시 비교

> **단일 출처는 `02-market-competition`**(전체 비교 매트릭스·포지셔닝 쿼드런트·생태계 맵). 본 절은 그
> 결론을 **인용·요약**한다. 근거: research-s1 §3(매트릭스)·§9(해자).

**한 줄 명시 비교(research-s1 §3·§9):** *htop·btop·top·glances·gotop·Activity Monitor* 는 **런타임 인지
(RT)·프레임워크 식별(FW)·포트 매핑(PORT) 세 축이 모두 ✗**, *pm2·systemd·launchd* 는 **종료(KILL)는
✓지만 RT/FW/PORT가 비고 범위가 좁다.* **세 축(RT·FW·PORT)이 동시에 ✓인 도구는 ntop뿐이다.**

```
비교 매트릭스 요약 (전체는 02-market-competition / research-s1 §3)
 ✓=한다 · △=부분/조건부 · ✗=안 함        RT  FW  PORT KILL TREE XPLAT T+C BIN
 ──────────────────────────────────────────────────────────────────────────
 ▶ ntop                                    ✓   ✓   ✓    ✓    ✓    ✓    ✓   ✓   ← RT·FW·PORT 동시 ✓ 유일
 ──────────────────────────────────────────────────────────────────────────
 htop / btop                               ✗   ✗   ✗    △    ✓    △    ✗   ✓
 top / gotop / glances                     ✗   ✗   ✗    △    △/✗  △/✓  △/✓ ✓/✗
 Activity Monitor (GUI)                    ✗   ✗   ✗    △    ✓    ✗    ✗   ✓
 pm2 (Node 매니저)                          △   ✗   ✗    ✓    ✗    ✓    ✓   ✗
 systemd / launchd (서비스 관리)            ✗   ✗   ✗    ✓    △    ✗    △   △
 RT=런타임인지 FW=프레임워크식별 PORT=포트매핑 KILL=시그널/graceful TREE=트리뷰
 XPLAT=크로스플랫폼 T+C=TUI+CLI BIN=단일바이너리
```

```
포지셔닝 한 줄 (research-s1 §4)
  제어 깊이 높음 │ pm2·systemd·launchd        ▶ ntop (의미 인지 × 안전 제어 동시)
                │ (제어 깊지만 의미 인지 낮고 범위 좁음)
  ──────────────┼────────────────────────────────────────────────────────
  제어 깊이 낮음 │ htop·btop·top·glances·gotop·Activity Monitor
                │ (의미 인지 거의 없음, 시그널 전송 수준)
                └──────────────────────────────────────────────────────▶
                 의미 인지 낮음("익명 PID")        의미 인지 높음("Next.js@:3000")
```

**모방난이도(해자/moat) 3논거 — research-s1 §9 / `02-market-competition` 인용:**
1. **데이터 주도 규칙 테이블 = 축적된 경험 사전.** `node_modules/.bin/next`·`-m flask`·`bin/rails`·
   `org.springframework`·macOS `comm` 16자 절단을 푸는 `normalize_name` 등은 기능 하나가 아니라
   **현장에서 검증해 쌓은 경험치**라 "한 번에 추가" 불가(codebase-facts §2).
2. **프로세스-로컬 전용 탐지 = 비자명한 설계 결정.** 순진한 복제는 cwd의 `package.json`을 읽어
   npx/전역 프로세스를 **오분류**하는 함정에 빠진다 — ntop은 이를 **의도적으로 회피**(PRB-3 해소,
   §5-2 영구 제외).
3. **확장 구조가 해자를 복리로 키운다.** 새 런타임/FW = `Rule` 1개 + enum 값 1개 → 기여가 쌓일수록
   사전이 복리로 커진다(00-business-model §6 지속가능성 엔진과 동일 메커니즘).

**결론:** ntop의 차별점은 "더 편리/더 빠름"이 아니라 **축이 다르다** — 의미 인지(RT·FW·PORT)와 안전
제어(graceful·트리)를 **동시에** 갖춘 유일한 좌표다(research-s1 §4·§9).
