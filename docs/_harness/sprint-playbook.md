# Sprint Playbook — ntop 역설계 풀 기획 패키지 (Mode 5, Full tier)

> 이 파일은 PLANNER가 발행한 **4개 고정 리서치 스프린트 계획**의 인스턴스다. 각 스프린트마다 Generator가
> `sprint_contract.md`를 제안하고, Evaluator가 본 플레이북의 관찰 가능한 기준(observable bar)과 게이트에 비추어
> 승인/보강한 뒤 산출물을 평가한다. **승인 전 작성 금지.**
>
> **고정 파라미터:** Mode 5(16종 패키지) · HTML=**Both**(문서별 + 허브 index.html + 종합 overview.html) · 출력 언어=**한국어**.
> **사실 원천:** `docs/_harness/codebase-facts.md`(도시에) + 실제 소스코드. 추측·창작 금지. ntop은 **이미 v0.2.0 구현된 OSS** → 역설계 기획.

## 왜 4 스프린트인가 (Full-tier 근거)

기획 문서는 **리서치 단계 분해 + 단계별 품질 게이팅**이 필요하다. 약한 시장 리서치/문제 정의 위에 기획서를 쌓으면 차별화(C2)와 실현가능성(C3)이 구조적으로 무너진다. 따라서 Evaluator가 S1~S3 리서치를 게이트로 통과시킨 **다음에야** S4가 그 위에 16종 문서를 합성한다. 이 게이팅은 컨텍스트 절약용이 아니라 **직교적 품질 메커니즘**이다(강한 모델이어도 무효화되지 않음).

ntop 특수성: 이미 구현된 도구의 역설계이므로 각 스프린트의 "리서치"는 **도시에/코드에서 사실을 추출·정리·시각화**하는 작업 + (S1만) 외부 경쟁/생태계 맥락 정리다. 사실은 도시에를 벗어날 수 없다.

## 공통 게이트/기준 참조

- **기준(criteria):** C1 문제정의 명료성 / C2 차별화 / C3 실현가능성·통합 일관성.
- **게이트:** G-a 기능↔문제 추적 / G-b 타깃 구체성 / G-c 실명 경쟁자 명시 비교 / G-d 지표 수치+방법 / G-e MVP·Out-of-scope 분리 / G-f 플레이스홀더 0 / G-g 인포그래픽 우선(외부 렌더러 0). 추가 P-1 = 와이어프레임 STEP 7 휴먼 체크포인트.

## S1 — 리서치 / 시장·경쟁·생태계 → `research-s1.md`

- **목적:** 시장 맥락 + **실명 경쟁자** 식별·벤치마크 + **생태계·참여자 분석**. 차별화(C2)의 근거 제공. → 후속 `02-market-competition.md`의 토대.
- **공급(feeds):** C2, 게이트 G-c·G-g.
- **관찰 가능한 체크(계약에 포함될 기준):**
  - **실명 경쟁자 ≥3** 식별. 후보(반드시 실재 도구로): `htop`, `btop`, `top`, `glances`, `gotop`, macOS **Activity Monitor**, `ctop`(컨테이너), `pm2`(Node 프로세스 매니저). `systemd`/`launchd`는 간접 비교 가능.
  - 각 경쟁자별 **비교 축**(무엇을 하고/안 하는지) 기록 — 축 예: 런타임 인지·프레임워크 식별·포트 매핑·시그널/graceful 종료·트리 뷰·크로스플랫폼·TUI+CLI·단일 바이너리.
  - **시장/사용자 맥락** 서술(폴리글랏 로컬 개발 증가, 한 머신 다중 서버 런타임 상황).
  - **생태계 전 참여자** 식별 + 역할·인센티브·가치교환·의존: 공급(메인테이너/기여자), 수요(개발자/운영자), 플랫폼/중개(GitHub·crates.io·Homebrew·릴리스 바이너리), 보완재(대상 런타임/프레임워크·터미널·의존 크레이트 sysinfo/ratatui/crossterm/clap/tokio·의존 시스템명령 lsof/netstat/taskkill), 대체재(위 경쟁자), 제도(MIT/OSI·Rust 관행), 후원/배포 인프라(GitHub Sponsors·GitHub Actions). **결제 파트너 아님 — 후원/배포로 해석.**
  - **생태계 맵(이해관계자/가치 네트워크 다이어그램)** 시각화(G-g, Mermaid 등 외부 렌더러 0).
- **Mode 5 적응 범위:** 확장 — 비교 매트릭스(✓/△/✗) + 포지셔닝 쿼드런트 + 생태계 분석·맵을 `02` 문서 수준으로 정리. (시장 규모 TAM/SAM/SOM은 OSS·무료 도구 특성상 "채택 가능 사용자 모집단/세그먼트" 정성 추정으로 치환, 매출 규모 산정 금지.)
- **Evaluator 스프린트 통과:** C2 채점 + G-c/G-g 게이트 + 차별화 명시 비교 probe + 생태계/시각화 적합성 probe.

## S2 — 문제·타깃·페르소나 → `research-s2.md`

- **목적:** who-what-why 문제 정의 + 구체 페르소나(상황·페인포인트). 문제정의 명료성(C1)의 근거. → 후속 `03-personas.md`·`01`·`10`의 앵커.
- **공급:** C1, 게이트 G-a(추적의 시작점인 문제 목록)·G-b(타깃 구체성).
- **관찰 가능한 체크:**
  - 문제가 **누가/무엇이/왜** 형태로 날카롭게 정의(핵심 문제 = 범용 모니터와 "내 개발 서버" 사이 **의미 격차**; 익명 `node` 더미·포트 역추적·엉뚱한 kill 위험·다중 런타임 문맥전환).
  - **개발자/운영자 페르소나 ≥2** 중 핵심 1개는 **상황·맥락·페인포인트(빈도/결과)** 까지 구체화 — 추상 인구통계("개발자")만이면 실패. 권장 페르소나 축: ① 폴리글랏 풀스택 로컬 개발자(Node+Python 다중 dev 서버), ② 다수 서비스 프로세스를 점검하는 운영자, ③ npx/MCP·CLI 도구를 다수 띄우는 사용자(오분류 회피가 중요).
  - 각 페르소나의 "원하는 결과"가 ntop 기능(식별/포트/안전 종료)으로 연결 가능.
- **Mode 5 적응 범위:** 다수 페르소나 허용, 단 핵심 1개는 구체적. 문제 목록은 PRD·유저스토리의 추적 앵커가 됨.
- **Evaluator 스프린트 통과:** C1 채점 + G-a(문제 목록 존재)·G-b 게이트 + 타깃 구체성 probe.

## S3 — UX/UI 플로우·화면 → `research-s3.md`

- **목적:** 유저 플로우 + 화면(=TUI 뷰/탭/다이얼로그 + CLI 출력)별 기능 + 와이어프레임/화면 구조. → 후속 `12`/`13`/`20`/`21`의 토대.
- **공급:** 와이어프레임 게이트(STEP 7/P-1), G-a(기능↔문제 추적용 기능 후보).
- **관찰 가능한 체크:**
  - **핵심 유저 플로우** 정의 + 시각화: 유저 플로우 플로우차트 + 핵심 세션 시퀀스/상태 다이어그램(G-g). 필수 플로우: 기동→스캔(2-pass)→선택→상세 4탭 점검→종료; 필터/정렬/Node-only 토글; **kill graceful→force 에스컬레이션**; SignalPicker; CLI `list`→`kill --tree`.
  - **인터페이스 정보구조(IA)** 정리: TUI 레이아웃 계층(상단바/좌 프로세스 목록/우 상세[Info·Log·Net·Env]/다이얼로그 5종/필터 모드) + **CLI 명령 트리**(`ntop`+`list`/`kill`/`info`/`log`/`config`). → IA 트리 다이어그램.
  - **키 바인딩 정리**(전역/리스트포커스/상세포커스/필터/다이얼로그) — 도시에 §6 권위값 그대로, `21`의 정규 출처가 됨.
  - **화면/상태 식별**: 핵심 화면의 4상태(empty=프로세스 없음, loading=스캔 중 스피너 tick, normal, error=권한거부/명령부재).
  - **지속적 UI/화면/디자인 시각화(필수):** S3은 UI/UX·화면명세의 핵심 스프린트 — 화면·플로우·IA를 산문만으로 두지 말고 반드시 시각(와이어프레임·레이아웃·상태전이·플로우·컴포넌트 다이어그램)을 동반. ntop은 TUI라 ASCII 레이아웃이 실제 충실도.
- **Mode 5 적응 범위:** 무겁게 — IA·플로우·화면·**와이어프레임**까지. 와이어프레임 존재 → **STEP 7 휴먼 체크포인트 트리거(`HUMAN_CHECKPOINT_REQUIRED`)**.
- **Evaluator 스프린트 통과:** C1 일부(플로우↔문제 일관) + G-g(플로우·화면 다이어그램 적합) + 와이어프레임 존재 시 P-1 표기(시각 품질은 채점 제외).

## S4 — 통합 기획 문서 작성 (그룹별 다수 문서) → `docs/` 16종 + INDEX + HTML

- **목적:** S1~S3 리서치를 Mode 5 구조로 통합. **신규 리서치 없음 — 통합 + 범위/우선순위/지표 확정만.** Mode 5는 단일 문서가 아니라 **그룹별 다수 문서**로 확장된다(각 그룹 = 1 sprint_contract + 1 Evaluator 통과).
- **공급:** C3, 게이트 G-a/G-d/G-e/G-g + 최종 G-f.
- **순서 강제:** g1의 **00-business-model(OSS 지속가능성·수익화 가설 = 단일 출처)** 을 가장 먼저 확정해야 g4(31·32·33 등 기술/정책 문서)가 그 위에 일관되게 쌓인다. 단일 출처 레지스트리(spec §4-3) 준수 — 같은 수치는 정규 출처에만, 나머지는 참조. 항상 마지막 순서는 g6(HTML) → g7(INDEX).

### S4 그룹 분할 (g0~g7)

- **g0 이미지 에셋(가능 시, 인포그래픽 티어 ①):** 이미지 생성 능력이 있으면 페르소나 웹툰·생태계 일러스트·개념도를 `docs/assets/`에 생성해 관련 md/HTML에서 참조. 능력 없으면 인라인 SVG/ASCII로 폴백(스킵 가능).
- **g1 발견·전략:** `00-business-model`(단일 출처 먼저 확정) → `01-service-plan` → `02-market-competition`(S1 정리) → `03-personas`(S2 정리).
  - 관찰 바: 00=배포 채널 ≥3 비교·채택 퍼널·수익화 "가설" 표기·정산/take-rate 0 / 01=7섹션+MVP(=v0.2.0)·Out-of-scope 분리·차별화 실명비교·마인드맵·가치루프 / 02=실명 경쟁자 ≥3·비교 매트릭스·생태계 맵 / 03=핵심 페르소나 구체·여정 맵.
  - 게이트: G-c(02), G-b(03), G-e(01), G-g 전체.
- **g2 정의:** `10-prd` → `11-user-stories` → `12-ia` → `13-user-flows`.
  - 관찰 바: 10=기능 ID·P0/P1/P2·**기능↔문제 추적표(G-a)**·비기능 정량 / 11=Epic→Story·GWT 수용기준·기능 ID 링크 / 12=인터페이스 IA 트리+CLI 명령 트리 / 13=핵심 플로우·분기·화면·기능 ID 링크·플로우차트+시퀀스.
  - 게이트: G-a(10), G-g(11·12·13).
- **g3 디자인:** `20-wireframes` → `21-screen-spec`. (**STEP 7 휴먼 체크포인트 대상**)
  - 관찰 바: 20=화면당 1 저충실도 시안(ASCII/인라인-SVG 터미널 프레임) / 21=화면당 시안+컴포넌트 배치 + 핵심 화면 4상태(empty/loading/normal/error) + 화면 명세(목적/컴포넌트/상호작용/상태/예외/연결 기능·인터페이스) + **키 바인딩 정규 출처**. 산문만 화면 설명 = G-g 실패.
  - 게이트: G-g, P-1(와이어프레임 존재 → HUMAN_CHECKPOINT_REQUIRED).
- **g4 기술:** `30-functional-spec` → `31-erd` → `32-api-spec` → `33-policy`. (00 및 31 단일 출처 참조, 상호 일관)
  - 관찰 바: 30=기능별 입력/처리/출력/예외 + **탐지 2단계 분류 플로우차트(name_exact→command_binary→command_contains, None=미표시)**·스캔 2-pass·graceful 폴링 / 31=구조체·enum 필드/관계(ProcessInfo.children 자기참조 1:N 등) + **Config 기본값**·**DB 부재 명시** + ERD 다이어그램 / 32=**CLI 계약(서브커맨드/플래그/종료코드/출력 스키마: table·JSON 전필드+depth·CSV 9컬럼) + 공개 lib API + 시스템명령(lsof/netstat/taskkill) 계약** + 시퀀스 다이어그램 / 33=권한 매트릭스 히트맵(Unix 6시그널/Win 3시그널·EPERM) + 마스킹 + **fs 미독 프라이버시 원칙** + graceful→force·트리 역순 + 31 참조(중복정의 0).

- **g6 HTML 출력(Both, md 항상 보존, g1~g5 PASS + STEP 7 클리어 후):** 각 `.md`→읽기용 `NN-name.html`(ASCII 보존 + 인라인-SVG 인포그래픽 + 가능 시 생성 이미지) + 허브 `index.html`(INDEX.md 반영, 중요도/순서/역할 링크) + 종합 `overview.html`(마인드맵·차트·생태계 맵·플로우 종합 대시보드). **신규 콘텐츠 생성 0(기존 산출물 변환·시각화만), 외부 렌더러 의존 0.** 텍스트/ASCII-only 변환이면 실패 — 페이지당 인라인 `<svg>` 여러 개.
- **g7 가이드:** `INDEX.md` — **모든 문서(g1~g6) 완료 후 최후 생성.** 3티어 중요도(T1 필수/T2 조건부/T3 보조) + 의존 기반 작업순서 + 역할별 번들 + 최소 시작 세트. 도메인에 맞게 실제 티어 판정. 신규 콘텐츠 0(분류·정렬만).
  - **ntop 도메인 티어 가이드(예시, Generator가 최종 판정):** T1 = `01`·`10`·`13`·`21`·`30`·`31`·`32`·`40` / T2(조건부, ntop에서 승격) = `33`(안전·프라이버시가 도구 정체성 → T1 승격 권장)·`41`·`00`(지속가능성) / T3(정렬/온보딩/설득) = `02`·`03`·`11`·`12`·`20`. (`00`은 결제형이 아니므로 T1 강제 아님 — 후원/지속가능성 설득용.)
  - **역할 번들:** 기여자/개발자=`21`·`31`·`32`·`30`·`40` / QA=`11`·`41`·`13` / (TUI)디자인=`12`·`13`·`20`·`21`·`03` / 메인테이너·PO=`01`·`10`·`40` / 채택자·후원검토=`00`·`02`·`03`.
  - **최소 시작 세트:** `10`·`13`·`21`·`31`·`32`·`40`(+ 안전 정체성 때문에 `33` 추가 권장). "전체 세트는 킥오프/전달/온보딩용, 이 세트면 일상 작업 충분."

### S4 관찰 가능한 체크 / 완료 정의(공통)

- 선택 모드(Mode 5)의 모든 문서 섹션이 실제 내용으로 채워짐(플레이스홀더 0 — G-f).
- 모든 핵심 기능이 문제 정의의 특정 문제로 매핑(기능↔문제 추적표 — G-a).
- MVP(=v0.2.0)/Out-of-scope 명시 분리(G-e) + 우선순위 근거.
- 모든 성공지표에 측정 수치 + 방법(도구/주기/판정), 채택 지표는 "가설" 표기(G-d).
- 차별화 섹션이 실명 경쟁자 ≥1과 명시 비교(G-c, S1 근거 활용).
- 구조가 있는 모든 문서/섹션에 개념 적합 시각화 임베드(G-g — 텍스트 벽 0·빈 다이어그램 0·외부 렌더러 0·가능 시 생성 이미지).
- 교차 일관성 상호 점검(단일 출처 레지스트리 위반·모순 0; 결제/DB/REST 창작 0; fs 미독 원칙 유지).
- **Evaluator 통과:** 본 스프린트 통과 + **STEP 5 최종 통합 패스에서 4축 + 8 probe + §8 게이트 7종 전체.**

## 스프린트 루프 (각 스프린트 공통)

```
1) Generator: sprint_contract.md 제안 (산출물 + 관찰 가능한 체크 + 출력 파일/포맷, 모드 적응)
2) Evaluator: 계약 승인 또는 보강 (spec/플레이북 기준보다 약하면 강화) — 승인 전 작성 금지
3) Generator: 산출물 작성 → 자체검증(모든 계약 체크) → generator_report.md → READY_FOR_QA
4) Evaluator: 계약 체크 + 스프린트 관련 probe/게이트로 평가 → critique.md → PASS/FAIL
5) PASS → 다음 스프린트 / FAIL & round<cap → 같은 스프린트 재시도 / FAIL & round=cap → 미해결 이슈 투명 보고 + 사용자 판단
```

## 반복 캡 (스프린트당)

- **스프린트당 5~15 라운드, 기본 시작 ~8**(단일 값 아닌 범위). 낮은 끝(5)은 후반 라운드의 도약을 놓칠 수 있음("iteration 10 leap" 경고). 캡은 스프린트당 적용, 전체 = 4 스프린트 합. S4는 그룹(g1~g7)별로 계약·평가가 반복되므로 그룹마다 캡이 적용된다.
