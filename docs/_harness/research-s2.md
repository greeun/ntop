# research-s2.md — 문제·타깃·페르소나 리서치 (ntop, S2)

> 본 문서는 PLANNER가 발행한 `spec.md`·`sprint-playbook.md`, **사실의 원천**인
> `docs/_harness/codebase-facts.md`(이하 "도시에"), 그리고 선행 산출물 `research-s1.md`(시장·경쟁·생태계)를
> 근거로 작성한 **S2 리서치 산출물**이다. ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스 도구**이므로,
> 이 리서치는 신규 아이디어 검증이 아니라 **이미 만들어진 도구가 푸는 문제와 그 도구의 사용자(개발자/
> 운영자)를 구체화**하는 작업이다. ntop은 결제·DB·회원 계정이 없는 로컬 개발자 도구이므로 페르소나는
> **돈 내는 고객이 아니라 개발자/운영자**다(도시에 §0-1).
>
> ntop에 대한 모든 주장은 도시에/실제 코드로 역추적 가능하다(§-번호 부기). 페르소나의 일과·빈도는
> 본 도구가 푸는 문제에서 **개연성 있게 구성한 사용 시나리오**이며, 인용한 ntop의 동작은 모두 도시에에
> 근거한다.
>
> **용어 풀이(첫 등장)**: **런타임(runtime)** = 코드를 실행하는 엔진(Node·Python·Java 등) / **프레임워크
> (framework)** = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등) / **dev 서버** =
> 개발 중 코드를 바로 띄워 확인하는 임시 서버 / **MCP(Model Context Protocol)** = AI 비서가 외부 도구·
> 데이터에 붙는 표준 연결 방식, 보통 `npx`로 백그라운드 프로세스로 뜬다 / **LSP(Language Server)** =
> 에디터 자동완성·진단을 제공하는 백그라운드 서버 / **graceful 종료(정상 종료)** = 프로세스에 "정리하고
> 꺼져라"라고 **요청**하는 것(SIGTERM), 강제로 죽이는 것(SIGKILL)과 다름 / **좀비·고아 프로세스** =
> 죽었는데 정리 안 된(좀비) 또는 부모가 먼저 죽어 떠도는(고아) 프로세스 / **PID** = 프로세스 고유 번호.

## 1. 핵심 문제 한 줄 (Executive snapshot) — 의미 격차

> **누가:** 한 머신에서 여러 런타임의 서버를 동시에 띄우는 개발자/운영자.
> **무엇이 문제:** 범용 프로세스 모니터(htop·top·Activity Monitor 등)는 프로세스를 **익명의 PID·CPU·
> 메모리**로만 보여줄 뿐, "이 `node`가 **어느 프레임워크의, 어느 포트의, 어느 프로젝트** 서버인가"를
> 모른다 — 이것이 **의미 격차(semantic gap)**다.
> **왜 문제:** 개발자는 그 격차를 `lsof -i :3000`(포트→PID), `ps aux | grep node`(이름 검색), 활성
> 모니터(Activity Monitor) **사이를 손으로 오가며** 메운다. 이 수작업·문맥 전환이 시간을 갉아먹고,
> 종종 **엉뚱한 프로세스를 죽이는 사고**로 이어진다(research-s1.md §1·§2-(D)).

```
 범용 모니터가 보는 것              내가 알아야 하는 것              ← 이 사이가 "의미 격차"
 ┌──────────────────────┐          ┌────────────────────────────┐
 │ PID 41822  node  2.1% │          │ Next.js dev 서버 @:3000      │
 │ PID 41835  node  0.4% │   ???    │ FastAPI @:8000 (백엔드)      │
 │ PID 41840  node  0.0% │  ─────▶  │ Claude Code MCP (건들면 끊김) │
 │ PID 41902  node  0.0% │          │ Vite 번들러 (자식)           │
 │ PID 42013  python 5%  │          │ Celery 워커 (큐 처리 중)     │
 └──────────────────────┘          └────────────────────────────┘
   "이름·번호·숫자뿐"                 "프레임워크·포트·역할·안전여부"
```

ntop은 이 격차를 **런타임/프레임워크 인지 분류**(도시에 §2)로 메우고, 그 자리에서 **포트 확인**(도시에
§4-1)과 **안전한 종료**(정상 종료 요청 → 안 죽으면 강제, 트리 단위 — 도시에 §5)까지 끝낸다. 본 S2는 그
"메워야 할 격차"를 **하위 문제 단위로 분해**하고(§2·§3), **그 통증을 실제로 겪는 사람**으로 구체화한다(§4).

## 2. who-what-why 문제 정의 (D-1) — 하위 문제 분해

> 핵심 문제(의미 격차)를 사용자가 실제로 부딪히는 **6개 하위 문제**로 쪼갠다. 각 문제는 **누가 / 무엇이 /
> 왜 / 빈도 / 결과(consequence)** 형식. 빈도는 본 도구가 겨냥하는 사용 맥락(폴리글랏 로컬 개발·다수
> 프로세스 점검)에서의 개연적 추정이며, "결과"는 그 문제가 방치될 때의 손해다. 각 문제는 §3에서 PRB-ID로
> 묶이고, ntop의 어떤 동작이 메우는지(도시에 근거)를 함께 표시한다.

### 2-1. 익명 `node` 더미 — "어느 게 뭔지 모름"

- **누가:** Node 기반 도구를 여러 개 띄우는 개발자(dev 서버 + 번들러 + LSP + MCP + CLI).
- **무엇이:** 이들이 OS에는 **모두 똑같은 `node`** 라는 이름으로 보인다. macOS는 프로세스 이름(`comm`)을
  **16자로 잘라** 더 구분이 어렵다(도시에 §2 `normalize_name`).
- **왜:** 범용 모니터에는 "이 `node`가 Next.js dev 서버다 / 저건 MCP다"라는 **의미 분류가 없다**
  (research-s1.md §3, RT·FW 열 모두 ✗). 사람이 매번 추론해야 한다.
- **빈도:** 개발 세션마다 — 하루 수 회 ~ 수십 회.
- **결과:** 매 점검마다 포트·커맨드라인을 눈으로 대조하느라 평균 1~3분 허비, 그리고 §2-3(엉뚱한 kill)의
  직접 원인.
- **ntop이 메움:** 런타임/프레임워크 분류(도시에 §2) + `display_name()`(커맨드에서 사람이 읽을 이름 추출,
  도시에 §3) → 좌측 목록 `NAME`/`FRAMEWORK` 컬럼·Info 탭.

### 2-2. 포트 역추적 수작업 — `lsof -i :3000` 왕복

- **누가:** 포트 충돌·리로드 꼬임을 겪는 개발자, 잔여 프로세스를 점검하는 운영자.
- **무엇이:** "포트 3000을 누가 잡고 있나?"를 알려면 `lsof -i :3000` → PID 확인 → 다시 `ps`로 정체 확인 →
  `kill`의 **3단계를 별도 명령으로** 오가야 한다(research-s1.md §2-(D)).
- **왜:** 범용 모니터는 **포트→프로세스 매핑이 없다**(research-s1.md §3, PORT 열 ✗; ctop은 컨테이너
  published 포트만 △). 모니터 화면과 포트 정보가 분리돼 있다.
- **빈도:** 주 10회 이상(포트 충돌·"이미 사용 중" 에러·리로드 꼬임마다).
- **결과:** 명령 왕복 + 결과 대조의 문맥 전환. 급할 때 PID를 잘못 읽어 §2-3로 이어진다.
- **ntop이 메움:** `NetworkInspector`가 LISTEN 포트를 PID별로 매핑(도시에 §4-1) → 목록 `PORT` 컬럼·Net
  탭. 흥미롭게도 ntop은 `lsof`/`netstat`를 **내부적으로 호출**해 수작업 baseline을 도구 안으로 흡수했다
  (research-s1.md §2-(D)).

### 2-3. 엉뚱한 프로세스 kill 위험 — MCP/도구 vs dev 서버

- **누가:** AI 도구(Claude Code·MCP 서버)·LSP·CLI 도구를 dev 서버와 **함께** 띄우는 개발자.
- **무엇이:** 익명 더미(§2-1)에서 포트로 추론해 `kill -9`를 날렸는데, 그게 dev 서버가 아니라 **건드리면
  안 되는 MCP/도구 프로세스**였던 사고. 특히 `npx`로 전역 실행된 도구는 작업 디렉터리(cwd)를 **상속**해
  제각각이라, cwd나 `package.json`으로 정체를 추정하면 **오분류**된다(도시에 §2).
- **왜:** 의미 분류가 없으니 "안전하게 죽일 수 있는 것 vs 아닌 것"을 구분할 근거가 사람의 기억뿐이다.
- **빈도:** 월 1~2회 실제 오kill + 매 종료마다 "이거 죽여도 되나" 망설임.
- **결과:** MCP/LSP 세션 강제 종료 → 재시작·재인덱싱으로 작업 흐름이 10~15분 끊김, 미저장 상태 손실 위험.
- **ntop이 메움:** ① **프로세스-로컬 전용 탐지**(이름+커맨드라인만, **파일시스템 안 읽음**)로 npx/전역
  실행 도구의 오분류를 의도적으로 회피(도시에 §2, research-s1.md §9-2) → 정체를 정확히 표시.
  ② kill 확인 게이트(`KillConfirm` 다이얼로그, `confirm_before_kill` 기본 true — 도시에 §3·§6)로 오발사
  방지. ③ **규칙에 안 맞는 프로세스는 애초에 표시 안 됨**(도시에 §2) → 죽일 후보 풀 자체가 "서버"로 한정.

### 2-4. 다중 런타임 문맥 전환 — 런타임마다 도구가 다름

- **누가:** Node 프런트 + Python 백엔드 + Java 레거시 + 워커를 한 머신에서 돌리는 폴리글랏 개발자
  (research-s1.md §5-1·§5-2 세그먼트 ①).
- **무엇이:** Node는 이렇게, Python(`uvicorn`/`gunicorn`/`celery`)은 저렇게, Java는 또 다르게 — 런타임별로
  점검·종료 방식·도구가 제각각이라 머릿속에서 **모드 전환**을 해야 한다.
- **왜:** 기존 도구는 한 평면만 본다 — 범용 모니터는 의미 없이 전체를, pm2는 Node 자기 앱만, ctop은
  컨테이너만(research-s1.md §3 탐지 범위 열). **여러 런타임의 서버를 하나로 횡단해 보는 도구가 없다.**
- **빈도:** 폴리글랏 세션 내내(상시).
- **결과:** 같은 작업(식별·종료)을 런타임마다 다른 절차로 반복 → 인지 부하·실수 증가.
- **ntop이 메움:** **8개 런타임 횡단** 분류(Node·Python·Java·Deno·Bun·Ruby·PHP·.NET — 도시에 §2) +
  동일한 키 조작으로 모두 처리(도시에 §6). Node만 보고 싶으면 `n` 키로 Node-only 토글(도시에 §6).

### 2-5. 안전하지 않은 종료 — `kill -9` 즉사 + 트리 미정리

- **누가:** 프로세스를 자주 재시작·정리하는 개발자/운영자.
- **무엇이:** 손에 익은 `kill -9`(SIGKILL)는 **정리할 틈 없이 즉사**시킨다 — 데이터 플러시·임시파일
  정리·자식 정리를 못 한다. 또 부모만 죽이면 **자식이 고아로 남아** 포트를 계속 점유한다.
- **왜:** 범용 모니터의 종료는 "시그널 보내기" 수준에 머문다 — **정상 종료→강제 종료 자동 단계
  (에스컬레이션)나 트리 단위 종료가 없거나 약하다**(research-s1.md §3 KILL 열: htop/btop/top/glances =
  △). 사람이 직접 시그널을 고르고, 안 죽으면 또 강제로 날려야 한다.
- **빈도:** 매 종료(하루 수 회) — 특히 dev 서버 재시작.
- **결과:** 데이터 손상·임시상태 유실 위험, 고아 프로세스가 포트를 점유해 §2-2(포트 역추적)를 재발시키는
  악순환.
- **ntop이 메움:** `graceful_kill` = SIGTERM 후 `graceful_timeout`(기본 10초)까지 200ms 간격 폴링,
  안 죽으면 SIGKILL **자동 에스컬레이션**(도시에 §5·§3); `kill_tree` = 자식부터 **역순** 종료로 고아
  방지(도시에 §5); `SignalPicker`로 필요 시 임의 시그널 선택(Unix 6종/Windows 3종 — 도시에 §5·§6).

### 2-6. 메모리·환경의 오판독 — "유휴인 줄 알았는데" + 민감값 노출

- **누가:** macOS 개발자(메모리 판단), 환경변수를 확인하는 모두(민감값).
- **무엇이:** ① macOS에서 일반 도구가 보고하는 메모리(sysinfo RSS)는 유휴 Node를 **최대 100배 과소
  보고**해(도시에 §4-2) "거의 안 쓰는 줄" 오판하게 만든다. ② 환경변수에는 비밀번호·토큰 같은 민감값이
  섞여 있어, 그냥 보여주면 어깨너머·로그로 **노출**된다.
- **왜:** 범용 도구는 정확한 메모리 지표 선택이나 민감값 마스킹을 신경 쓰지 않는다.
- **빈도:** 메모리 점검·환경 확인 시마다.
- **결과:** 메모리 오판으로 엉뚱한 프로세스를 의심/방치; 환경변수 민감값 유출.
- **ntop이 메움:** macOS는 `phys_footprint`(Activity Monitor와 일치하는 값)를 RSS 대신 사용(도시에 §4-2);
  Env 탭에서 `mask_env_values`(기본 true)로 PASSWORD/SECRET/TOKEN 등 자동 마스킹(도시에 §3·§6).

> **참고(비목표):** ntop은 **프로세스를 살려두는 도구가 아니다** — 감독·자동 재시작·데몬화는 pm2/systemd의
> 영역이고 ntop의 문제 범위가 아니다(research-s1.md §8-2). 본 S2의 문제 정의도 "식별 + 점검 + 안전 종료"
> 안에 한정한다(spec §7 비목표).

## 5. 원하는 결과 → ntop 기능 매핑 (D-3)

> 각 페르소나의 "원하는 결과"를 ntop의 3대 능력 — **식별(identify) / 포트(port) / 안전 종료(safe-kill)** —
> 로 연결한다. 이 매핑이 S4 `10-prd`의 **기능↔문제 추적표(G-a)** 로 그대로 이어진다.

```
 페르소나 원하는 결과            ntop 능력군        구체 동작(도시에)                          PRB / 잠정 기능
 ───────────────────────────────────────────────────────────────────────────────────────────────────
 P1 "어느 node가 뭔지 즉시" ─▶  [식별]   ──▶  런타임/FW 분류·display_name·Info탭(§2·§3·§6)   PRB-1 / F-분류
 P1/P3 "MCP·dev 정확 구분" ─▶  [식별]   ──▶  프로세스-로컬 전용 탐지(fs 안 읽음)(§2)         PRB-3 / F-정확분류
 P1/P2 "포트 그 자리에서"   ─▶  [포트]   ──▶  NetworkInspector LISTEN·PORT컬럼·Net탭(§4-1)    PRB-2 / F-포트매핑
 P1 "그 자리서 안전 종료"   ─▶  [안전]   ──▶  KillConfirm + graceful_kill 에스컬레이션(§3·§5) PRB-3,5 / F-안전종료
 P2 "자식까지 정상 정리"    ─▶  [안전]   ──▶  kill_tree 역순 종료(§5)                         PRB-5 / F-트리종료
 P2 "JSON으로 자동화"       ─▶  [식별]   ──▶  CLI list --json / kill --tree(§7)               PRB-7 / F-CLI·F-JSON
 P1/P2 "런타임 횡단/Node만" ─▶  [식별]   ──▶  8런타임 분류 + n키 Node-only 토글(§2·§6)        PRB-4 / F-멀티런타임
 P1 "정확한 메모리/안전 Env"─▶  [식별]   ──▶  macOS phys_footprint·mask_env_values(§4-2·§3)   PRB-6 / F-정확메모리
 P1 "GUI↔터미널 왕복 제거"  ─▶  [식별+안전] 터미널 내 TUI 일원화(§6·§7)                       PRB-8 / F-TUI일원화
```

**3대 능력 요약(도시에 근거):**
- **식별(identify):** 데이터 주도 규칙 테이블(`FRAMEWORK_RULES`/`RUNTIME_RULES`)로 8런타임·다수
  프레임워크를 분류, **파일시스템을 읽지 않는** 프로세스-로컬 전용 탐지(도시에 §2). → PRB-1·3·4 해소.
- **포트(port):** `NetworkInspector`가 LISTEN 포트를 PID별 매핑(도시에 §4-1). → PRB-2 해소.
- **안전 종료(safe-kill):** kill 확인 게이트 → `graceful_kill`(정상 종료 후 timeout 시 강제 자동
  에스컬레이션) → `kill_tree`(자식부터 역순) → 필요 시 `SignalPicker`(도시에 §3·§5·§6). → PRB-3·5 해소.

## 6. 시각화 (G-g)

### 6-1. 페르소나 여정 맵 (D-5) — P1(박서연) 기준

> 단계별 **행동 / 접점(도구) / 감정 / 통증(PRB) / ntop이 메우는 지점**. 가운데 점선(=ntop 도입)을 기준으로
> "도구 없는 현재"와 "ntop 사용"이 갈린다. 감정: 😀좋음 😐보통 😟혼란 😣불안 😡분노 😩피로 😌안도 😄확신.

```
 단계 ▶  ①서버 여러 개 띄움  →  ②문제 발생       →  ③어느 게 뭔지 모름  →  ④포트 역추적/엉뚱한 kill
```
