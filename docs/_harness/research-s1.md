# research-s1.md — 시장·경쟁·생태계 리서치 (ntop, S1)

> 본 문서는 PLANNER가 발행한 `spec.md`·`sprint-playbook.md`와 **사실의 원천**인
> `docs/_harness/codebase-facts.md`(이하 "도시에")만을 근거로 작성한 **S1 리서치 산출물**이다.
> ntop은 **이미 v0.2.0까지 구현된 MIT 오픈소스(OSS) 도구**이므로, 이 리서치는 신규 아이디어 검증이
> 아니라 **이미 만들어진 도구를 역설계해 시장·경쟁·생태계 좌표에 위치시키는 작업**이다.
> ntop에 대한 모든 주장은 도시에/실제 코드로 역추적 가능하다. 경쟁자 사실은 널리 알려진 실재 도구의
> 공개 동작이며, 확실치 않은 세부 기능은 **보수적으로(없는 쪽으로) 표기**했다.
>
> 용어 풀이: **TUI** = 터미널 안에서 키보드로 조작하는 전체화면 텍스트 UI / **CLI** = 명령 한 줄로
> 실행해 결과를 출력하는 방식 / **런타임(runtime)** = 코드를 실행하는 엔진(Node·Python·Java 등) /
> **프레임워크(framework)** = 그 런타임 위에서 서버를 만드는 틀(Next.js·FastAPI·Spring Boot 등).

## 1. 한눈에 보는 결론 (Executive snapshot)

ntop은 범용 시스템 모니터(htop·btop·top·glances·gotop·Activity Monitor)와 **다른 평면**에 있다.
범용 모니터는 모든 프로세스를 **익명의 PID·CPU·메모리**로 보여주지만, "이 `node`가 **어느
프레임워크의, 어느 포트의, 어느 프로젝트** 서버인가"는 모른다. ntop은 그 의미 격차(semantic
gap)를 **런타임/프레임워크 인지 분류**로 메우고, 그 자리에서 **포트 확인 + 안전한 종료(정상 종료
요청 → 안 죽으면 강제 종료, 트리 단위 종료)** 까지 끝낸다.

- ntop이 **이기는 축**: 런타임 인지, 프레임워크 식별, 포트→프로세스 매핑 내장, 안전한 종료 관리,
  크로스플랫폼(mac·Linux·Win), TUI+CLI 동일 코어, 8개 런타임 횡단.
- ntop이 **약한 축**: 시스템 전역 그래프/이력(btop·glances 우위), 외부 명령(`lsof`/`netstat`) 의존,
  **프로세스 감독/재시작 없음**(pm2·systemd 우위 — ntop은 죽이는 도구이지 살려두는 도구가 아님).
- **모방난이도(해자/moat)**: 차별점은 단일 기능이 아니라 ① **데이터 주도 규칙 테이블**(각 프레임워크가
  프로세스 이름·커맨드라인에 드러나는 방식을 축적한 사전)과 ② **프로세스-로컬 전용 탐지 원칙**
  (파일시스템을 절대 읽지 않음)이라는 **설계 결정의 조합**이다. 둘 다 "기능 하나 추가"로 흉내 낼 수
  없다(§9 상세).

## 2. 실명 경쟁자 식별과 비교 축 (C-1, C-2)

### 2-0. 비교 축 정의 (무엇을 비교하는가)

spec §6의 축을 그대로 사용한다. 각 축은 "그 도구가 **하는가/안 하는가**"를 가른다.

| 축 코드 | 축 이름 | 의미(쉬운 설명) |
|---|---|---|
| **RT** | 런타임 인지 | 프로세스 이름 문자열을 넘어 "이건 Node/Python/Java 런타임이다"를 **의미로** 분류하는가 |
| **FW** | 프레임워크 식별 | "Next.js/FastAPI/Spring Boot 서버다"까지 식별하는가 |
| **PORT** | 포트→프로세스 매핑 | "이 프로세스가 포트 3000에서 듣고 있다"를 그 자리에서 보여주는가 |
| **KILL** | 시그널/graceful 종료 관리 | 임의 시그널 전송 + 정상 종료→강제 종료 자동 단계 + 트리(부모-자식) 종료 |
| **TREE** | 트리 뷰 | 프로세스 부모-자식 계층을 트리로 보여주는가 |
| **XPLAT** | 크로스플랫폼 | macOS·Linux·**Windows** 세 곳 모두 동작하는가 |
| **T+C** | TUI+CLI 둘 다 | 대화형 TUI **와** 스크립트용 일회성 CLI를 **둘 다** 제공하는가 |
| **BIN** | 단일 바이너리 설치 | 인터프리터/런타임 VM 없이 단일 실행파일로 간편 설치되는가 |

> 추가 맥락 축 — **탐지 범위(scope)**: 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor)는
> **시스템 전체 프로세스**를 보고, `ctop`은 **컨테이너**만, `pm2`/`systemd`/`launchd`는 **자신이
> 실행/구성한 것만** 본다. ntop은 **여러 런타임의 "서버 프로세스"만 선별**해 본다(규칙에 안 맞는
> 프로세스는 표시하지 않음 — 도시에 §2). 이 범위 차이가 포지셔닝의 핵심이다.

### 2-1. 경쟁자별 프로파일 (한 도구씩, 하는 것 / 안 하는 것)

#### (A) 범용 시스템/프로세스 모니터 — ntop의 직접 대체재

**htop** — 유닉스용 대화형 프로세스 뷰어(TUI).
- 한다: CPU/메모리 미터, **트리 뷰(F5)**, **임의 시그널 선택 전송(F9 시그널 메뉴)**, 태그(여러 개
  선택) 후 일괄 종료, 정렬/필터.
- 안 한다: **런타임/프레임워크 의미 분류 없음**(문자열 이름만), **포트→프로세스 매핑 없음**,
  **정상→강제 자동 에스컬레이션 없음**(시그널을 사람이 직접 고름), **Windows 네이티브 없음**(WSL/Cygwin
  필요), **스크립트용 CLI 모드 없음**(TUI 전용).

**btop / btop++** — 현대적 리소스 모니터(TUI).
- 한다: CPU/메모리/디스크/**네트워크 그래프 + 실시간 이력 차트**(이 점은 ntop보다 우위), 트리 뷰,
  프로세스 선택 후 시그널 전송 메뉴.
- 안 한다: 런타임/프레임워크 분류 없음, 포트→프로세스 매핑 없음(전체 net I/O만), 정상→강제 자동
  에스컬레이션 없음, 스크립트용 CLI 없음(TUI 전용), Windows 지원은 비교적 최근/제한적.

**top** — 고전 유닉스 프로세스 모니터.
- 한다: 실시간 갱신 목록, 정렬, (Linux) `k`로 시그널 전송·`V` 포레스트(트리) 뷰, **배치 모드
  `top -b`로 스크립트 출력**, 거의 모든 유닉스에 기본 탑재.
- 안 한다: 런타임/프레임워크 분류 없음, 포트 매핑 없음, 트리/종료 기능이 OS마다 들쭉날쭉
  (macOS `top`은 트리·kill 없음), Windows 없음(거기선 `tasklist`/작업 관리자).

**glances** — 파이썬 기반 크로스플랫폼 모니터.
- 한다: **macOS·Linux·Windows 모두 동작**, TUI + CLI + **웹/REST API + 익스포트(InfluxDB 등)**, 프로세스
  목록·정렬, `k`로 프로세스 종료.
- 안 한다: 런타임/프레임워크 분류 없음, **포트→프로세스 매핑 없음**(포트 플러그인은 "특정 포트가
  열려있나" 점검일 뿐 소유 프로세스 매핑이 아님), 트리 뷰 없음, **단일 바이너리 아님**(파이썬
  인터프리터 + pip 의존 필요).

**gotop** — Go로 만든 그래픽형 터미널 모니터.
- 한다: CPU/메모리/네트워크/디스크 그래프, 프로세스 목록·정렬, 프로세스 선택 후 종료, **단일
  바이너리(Go)**.
- 안 한다: 런타임/프레임워크 분류 없음, 포트 매핑 없음, 트리 뷰 없음(이름별 그룹화 정도),
  정상→강제 에스컬레이션 없음, 스크립트용 CLI 없음(TUI 전용), Windows 지원 제한적.

**macOS Activity Monitor(활성 상태 보기)** — macOS 기본 GUI 앱.
- 한다: 프로세스 목록, CPU/메모리/에너지/디스크/네트워크 탭, **계층(트리) 보기**(보기 > 모든
  프로세스, 계층 구조), **종료(graceful Quit) / 강제 종료(Force Quit, SIGKILL)**, OS 기본 포함(무설치).
  (참고: ntop이 macOS 메모리에 쓰는 `phys_footprint` 지표는 바로 이 Activity Monitor 값과 일치 — 도시에 §4-2.)
- 안 한다: 런타임/프레임워크 분류 없음(이름만 `node`), 포트→프로세스 매핑 없음, **CLI 없음·터미널
  도구 아님(GUI 전용)**, **macOS 전용**, 임의 시그널/트리 종료/자동 에스컬레이션 없음.

#### (B) 컨테이너/프로세스 매니저 — 인접·간접 대체재

**ctop** — 컨테이너(주로 Docker) 메트릭용 top형 인터페이스.
- 한다: 실행 중 컨테이너별 CPU/메모리/네트워크, 컨테이너 start/stop/exec, **컨테이너 published
  포트 표시**, 단일 바이너리(Go).
- 안 한다: **로컬 호스트의 dev 서버 프로세스는 안 봄**(컨테이너 범위), 언어 런타임/프레임워크 분류
  없음, 호스트 프로세스 트리/시그널 없음, 스크립트용 CLI 약함.

**pm2** — Node.js 프로세스 매니저(데몬 + CLI).
- 한다: `pm2 start/stop/restart/reload/delete`, **정상 종료(graceful) + kill_timeout 후 강제 종료
  에스컬레이션**, **무중단 reload**(cluster), 로그 관리, `pm2 monit` 대시보드, `pm2 list`/`pm2 jlist`(JSON),
  **프로세스 감독·자동 재시작**(ntop에 없는 강점), 크로스플랫폼(Node 위에서).
- 안 한다: **자신이 실행한 앱만 관리**(머신의 임의 실행 프로세스를 발견·분류하지 않음), 프레임워크
  식별 없음, 포트→프로세스 매핑 없음, **Node 전제**(npm 전역 설치 + Node 필요 → 단일 바이너리 아님).

#### (C) 서비스 관리자 — 간접 비교(spec §6에서 간접 비교로 명시)

**systemd** — 리눅스 init/서비스 관리자.
- 한다: `systemctl`로 서비스 시작/중지, **정상 종료(SIGTERM)→TimeoutStopSec 후 SIGKILL 자동
  에스컬레이션**, **cgroup 단위 트리 종료**(KillMode=control-group), `systemd-cgls`(트리)·`systemd-cgtop`(TUI류).
- 안 한다: **유닛 파일로 구성된 것만 관리**(임의 dev 서버 발견 안 함), 런타임/프레임워크 분류 없음,
  포트 매핑 없음, **리눅스 전용**.

**launchd** — macOS init/서비스 관리자.
- 한다: `launchctl`로 데몬/에이전트 관리, ExitTimeOut 기반 종료, plist로 구성.
- 안 한다: 구성된 잡만 관리, 런타임/프레임워크 분류 없음, 포트 매핑 없음, **macOS 전용**, 통합 TUI
  없음(CLI만).

#### (D) 개발자가 지금 수동으로 쓰는 "도구 없는 대체재" (baseline)

경쟁 도구 외에, 현재 개발자는 의미 격차를 **수작업**으로 메운다: `lsof -i :3000`(포트→PID),
`ps aux | grep node`(이름 검색), `kill -9 <pid>`(종료). 이 세 가지를 **오가는 문맥 전환**이 곧 ntop이
없애려는 문제다. 흥미롭게도 ntop은 이 중 `lsof`/`netstat`를 **내부적으로 호출**해 포트 정보를 채운다
(도시에 §4-1) — 즉 수작업 baseline을 도구 안으로 흡수했다.

## 3. 비교 매트릭스 (C-3) — 경쟁자 × 축 (✓/△/✗)

> 범례: **✓** = 한다 / **△** = 부분적·조건부(괄호 주) / **✗** = 안 한다.
> `▶` 표시 행이 ntop. 셀 판정은 §2 프로파일과 도시에에 근거.

```

--------------------+-----+-----+-----+-----+-----+-----+-----+-----+-----------------
▶ ntop              |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  |  ✓  | 서버 프로세스(선별)
--------------------+-----+-----+-----+-----+-----+-----+-----+-----+-----------------
htop                |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  △  |  ✗  |  ✓  | 시스템 전체
btop                |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  △  |  ✗  |  ✓  | 시스템 전체
top                 |  ✗  |  ✗  |  ✗  |  △  |  △  |  △  |  △  |  ✓  | 시스템 전체
glances             |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  ✓  |  ✓  |  ✗  | 시스템 전체
gotop               |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  △  |  ✗  |  ✓  | 시스템 전체
Activity Monitor    |  ✗  |  ✗  |  ✗  |  △  |  ✓  |  ✗  |  ✗  |  ✓  | 시스템 전체(GUI)
ctop                |  ✗  |  ✗  |  △  |  △  |  ✗  |  △  |  ✗  |  ✓  | 컨테이너만
pm2                 |  △  |  ✗  |  ✗  |  ✓  |  ✗  |  ✓  |  ✓  |  ✗  | 자신이 실행한 앱
systemd             |  ✗  |  ✗  |  ✗  |  ✓  |  △  |  ✗  |  △  |  △  | 구성된 유닛(Linux)
launchd             |  ✗  |  ✗  |  ✗  |  △  |  ✗  |  ✗  |  ✗  |  △  | 구성된 잡(macOS)
```

**셀 판정 근거(주요 △ 설명):**
- `htop/btop/top/gotop/glances/Activity Monitor` **RT·FW = ✗**: 프로세스 이름 문자열만 보여줄 뿐
  "Node 런타임의 Next.js 서버"라는 **의미 분류가 없다**. 이것이 §4 포지셔닝의 가로축 격차다.
- `KILL`: htop/btop/top/gotop/glances/launchd = **△**(시그널은 보내지만 **정상→강제 자동
  에스컬레이션·트리 종료가 없거나 제한적**). Activity Monitor = △(Quit/Force Quit 2종만). ctop =
  △(컨테이너 stop/kill). **pm2/systemd = ✓**(정상→강제 에스컬레이션·라이프사이클 제어가 강함, 단
  범위가 좁음). **ntop = ✓**(임의 시그널 6종[Unix]/3종[Win] + `graceful_kill` 자동 에스컬레이션 +
  `kill_tree` 역순 종료 — 도시에 §5).
- `PORT`: ctop = △(컨테이너 published 포트만). 나머지 범용 모니터 = ✗. **ntop = ✓**(`NetworkInspector`로
  LISTEN 포트를 PID별 매핑 — 도시에 §4-1).
- `XPLAT`: glances·pm2 = ✓(Win 포함), htop·btop·top·gotop·ctop = △(Win 네이티브 미흡/WSL),
  Activity Monitor·launchd = ✗(macOS), systemd = ✗(Linux). **ntop = ✓**(mac·Linux·Win, `platform.rs` +
  `windows-sys` — 도시에 §4-2, Cargo.toml).
- `T+C`: top = △(배치 모드 `-b`), systemd = △(CLI 강력 + `cgtop` 별도). glances·pm2 = ✓. **ntop =
  ✓**(TUI 기본 + `list`/`kill`/`info`/`log`/`config` 서브커맨드, `--json`/`--format csv` — 도시에 §7).
- `BIN`: glances·pm2 = ✗(파이썬/Node 런타임 필요). systemd·launchd = △(OS 구성요소라 "설치" 개념
  아님). **ntop = ✓**(Rust 단일 바이너리: `cargo install`/Homebrew/릴리스 바이너리 — 도시에 §11).
- `pm2 RT = △`: Node 전제로 동작하나 임의 프로세스를 발견·분류하진 않음.

**매트릭스가 말하는 것:** **RT·FW·PORT 세 열이 동시에 ✓인 도구는 ntop뿐이다.** 범용 모니터는
RT·FW·PORT가 모두 ✗, 매니저(pm2/systemd)는 KILL은 강하지만 RT·FW·PORT가 비어 있고 범위가 좁다.

## 4. 포지셔닝 쿼드런트 (C-4)

두 축으로 도구들을 배치한다.
- **가로축(X) = 런타임/프레임워크 "의미 인지"**: 프로세스를 PID·이름으로만 보는가(왼쪽) →
  "어느 런타임의 어느 프레임워크 서버"로 이해하는가(오른쪽).
- **세로축(Y) = 프로세스 "제어 깊이"**: 보기만 하는가(아래) → 임의 시그널·정상/강제·트리·라이프사이클
  제어까지 하는가(위).

```
제어 깊이
(높음)
  ▲
  │  systemd            pm2                          ▶ ntop
  │  (구성 유닛/         (Node 앱/                   (시그널6·graceful
  │   cgroup 트리,       graceful reload,             →force·tree kill,
  │   graceful→force)    감독·재시작)                 8런타임·FW식별)
  │
  │                          launchd
  │  ──────────────────────────(구성 잡)─────────────────────────────
  │       htop          btop
  │     (F9 시그널,    (시그널,
  │      트리 F5)       이력 그래프)        ctop
  │                                        (컨테이너 stop/kill,
  │   Activity Monitor                      published 포트)
  │   (Quit/Force,
  │    계층 보기)
  │      gotop      glances
  │     (kill)     (k kill, 웹/API)
  │       top
  │     (k/ V)
  └────────────────────────────────────────────────────────────────────▶
   (낮음)        런타임/프레임워크 의미 인지              (높음)
   "익명 PID·이름만"                              "Next.js/FastAPI 수준 식별"
```

**읽는 법:**
- **오른쪽 위(고-의미인지 × 고-제어) = ntop이 거의 독점.** 의미를 알면서(오른쪽) 동시에 안전하게
  제어(위)하는 도구가 ntop이다.
- **왼쪽 위 = pm2·systemd·launchd**: 제어는 깊지만 의미 인지는 낮고(런타임 분류·프레임워크 식별
  없음) **범위가 좁다**(자신이 관리/구성한 것만).
- **왼쪽 가운데~아래 = 범용 모니터(htop·btop·top·glances·gotop·Activity Monitor)**: 의미 인지가 거의
  없고 제어도 시그널 전송 수준에 머문다.
- **ctop**은 사실상 **다른 평면(컨테이너)** 에 있어 좌표상 한 점으로 두지만 직접 경쟁이 아니다.

**핵심 통찰:** 경쟁자들은 모두 **가로축 왼쪽**(의미 인지 낮음)에 몰려 있다. ntop의 차별화는 이
**가로축(의미 인지)을 오른쪽으로 끌고 가면서 세로축(제어)도 높게 유지**한 데 있다. "더 편리"가 아니라
**축이 다르다.**

## 5. 시장 / 사용자 맥락 (C-5, 정성)

> ntop은 **무료 OSS 도구**이므로 매출 기반 TAM/SAM/SOM(시장 규모 금액)을 산정하지 않는다. 대신
> spec/playbook 지시에 따라 **"채택 가능 사용자 모집단/세그먼트"를 정성적으로** 추정한다.

### 5-1. 시장 흐름 — 왜 지금 이 도구가 의미 있나

1. **폴리글랏(polyglot, 한 프로젝트에 여러 언어) 로컬 개발의 일상화.** 프런트엔드는 Node(Next.js/Nuxt),
   백엔드는 Python(FastAPI/Django)·Java(Spring Boot)·.NET, 거기에 워커·큐·캐시까지 — 한 개발자가 **한
   머신에서 여러 런타임의 서버를 동시에** 띄우는 것이 표준이 됐다.
2. **한 머신에 동종 프로세스 더미.** 특히 `node` 프로세스는 dev 서버·번들러·LSP·MCP 서버·CLI 도구가
   모두 같은 이름으로 떠 있어, 범용 모니터에서는 **구분이 불가능한 익명 더미**가 된다.
3. **AI 도구·MCP 서버의 폭증.** npx로 띄우는 MCP/도구 프로세스가 늘면서, "건드리면 안 되는 도구
   프로세스 vs 죽여도 되는 dev 서버"를 가려야 할 필요가 커졌다. (ntop의 프로세스-로컬 전용 탐지 원칙은
   바로 이 오분류를 막기 위한 설계 — 도시에 §2.)
4. **터미널 우선·키보드 중심 워크플로의 부활.** TUI 모니터(htop·btop)의 인기가 보여주듯, 개발자는
   GUI보다 터미널에서 빠르게 점검·조치하길 원한다. ntop은 이 흐름 위에 **의미 인지**를 얹었다.

### 5-2. 채택 가능 사용자 모집단/세그먼트 (정성, 깔때기형)

```
[ 모집단 — 넓음 ]
 터미널을 쓰는 모든 개발자 (htop/top을 한 번이라도 써 본 사람)
        │  ↓ 좁힘: "한 머신에서 여러 런타임 서버를 띄운다"
[ 핵심 세그먼트 ]
 ① 폴리글랏 풀스택 개발자  — Node 프런트 + Python/Java 백 동시 구동
 ② 마이크로서비스 로컬 개발자 — 한 머신에서 여러 서비스 프로세스
 ③ AI/도구 개발자          — npx/MCP·CLI 프로세스를 다수 구동(오분류 회피가 중요)
        │  ↓ 좁힘: "익명 프로세스 더미 / 엉뚱한 kill에 데인 경험"
[ 1차 채택자 (early adopter) ]
 ④ Rust·TUI 선호 개발자    — 단일 바이너리·키보드 워크플로 애호
 ⑤ 다수 서비스 점검 운영자  — 빠른 식별·안전 종료가 잦은 작업
```

- **세그먼트 우선순위 근거:** ①·③이 "의미 격차" 통증이 가장 크고 빈도가 높다(매일·세션마다 발생).
  이들이 후속 S2 페르소나의 1순위 후보다.
- **모집단 추정의 성격:** 금액이 아니라 **"폴리글랏 로컬 개발 × 터미널 사용 × 다중 런타임"의 교집합**
  으로 좁힌 정성 추정이다. 수치 목표(Stars/다운로드 등)는 후속 `00-business-model`/`10-prd`에서
  **"가설"로** 다룬다(여기서 산정하지 않음).

  순환 밖에서 동일 사용자를 두고 경쟁한다(의미 인지 축 부재).

## 8. 갭 / 기회 (C-8 전반부)

### 8-1. ntop이 이기는 지점 (기회)

| 강점 | 근거(도시에) | 어느 경쟁자 대비 |
|---|---|---|
| **런타임 인지 분류**(8종) | §2 RUNTIME_RULES, Runtime enum | 범용 모니터 전부(RT=✗) |
| **프레임워크 식별** | §2 FRAMEWORK_RULES(Next/FastAPI/Spring Boot…) | **모든 경쟁자(FW=✗)** — ntop 독점 |
| **포트→프로세스 매핑 내장** | §4-1 NetworkInspector(LISTEN) | 범용 모니터(PORT=✗) |
| **안전한 종료 관리** | §5 KillSignal·graceful_kill·kill_tree | htop/btop/top/glances(△, 에스컬레이션·트리 약함) |
| **크로스플랫폼 + 단일 바이너리 + TUI/CLI 동시** | §7·§11, Cargo.toml | 어느 한 도구도 셋을 동시 충족 못 함 |
| **정확한 메모리(macOS phys_footprint)** | §4-2 (Activity Monitor 일치) | sysinfo RSS 과소보고 회피 |

### 8-2. ntop이 약한/비어 있는 지점 (한계)

| 약점 | 근거(도시에) | 우위 경쟁자 |
|---|---|---|
| **시스템 전역 그래프·실시간 이력 없음** | 프로세스 선별 중심, 이력 차트 모듈 없음 | btop·glances·gotop |
| **프로세스 감독/재시작/데몬화 없음** | ntop은 죽이는 도구(killer.rs), 살려두는 supervisor 아님 | **pm2·systemd**(자동 재시작) |
| **외부 명령 의존** | §12 lsof/netstat 없으면 포트 빈 결과 | (자체 구현 도구 대비 취약) |
| **Windows 시그널 제한** | §12 HUP/USR1/USR2 없음(Win 3종) | (Unix 환경에선 비문제) |
| **로그 캡처 한계** | §12 cwd glob + Linux `/proc/<pid>/fd/1` fallback만 | pm2(통합 로그 관리) |
| **프레임워크 규칙 공백** | §12 Express/Fastify/Koa/Hapi enum만 있고 규칙 미등록 | (로드맵 근거 — 40-backlog) |
| **컨테이너 비대상** | 호스트 프로세스 범위 | ctop(컨테이너) |
| **GUI 아님** | 터미널 전용 | Activity Monitor(GUI 선호자) |

> 이 한계들은 **로드맵 근거**이자(후속 `40-backlog`, 도시에 §12), 동시에 **포지셔닝의 선택**이다 —
> ntop은 "전역 모니터"가 아니라 "**서버 의미 인지 + 안전 종료**"에 집중한 도구다.

## 9. 차별화 모방난이도(해자/moat) 논거 (C-8 후반부)

> spec/playbook은 "더 편리/더 빠름" 같은 일반론을 **실패**로 규정한다. 따라서 차별화는 **실명 경쟁자
> 대비 명시 축 + 모방난이도 논거**로 제시한다.

**명시 비교 한 줄 정리:** *htop·btop·top·glances·gotop·Activity Monitor* 는 **RT·FW·PORT 축이 모두 ✗**
이고, *pm2·systemd·launchd* 는 **KILL은 ✓지만 RT/FW/PORT가 비고 범위가 좁다.* **세 축(RT·FW·PORT)이
동시에 ✓인 도구는 ntop뿐**이다(§3 매트릭스).

이 차별점이 **쉽게 복제되지 않는** 이유 — 모방난이도 3가지:

1. **데이터 주도 규칙 테이블 = 축적된 경험 사전.**
   `FRAMEWORK_RULES`/`RUNTIME_RULES`는 "각 프레임워크/런타임이 **프로세스 이름·커맨드라인에
   드러나는 방식**"을 수집한 사전이다 — `node_modules/.bin/next`, `-m flask`, `bin/rails`,
   `org.springframework`, macOS의 `comm` 16자 절단을 푸는 `normalize_name`(도시에 §2)… 이건 기능
   하나가 아니라 **현장에서 하나씩 검증해 쌓은 경험치**다. 범용 모니터가 이를 "한 번에 추가"할 수
   없는 이유다.

2. **프로세스-로컬 전용 탐지 원칙 = 비자명한 설계 결정.**
   ntop은 분류에 **파일시스템(`package.json`)을 절대 읽지 않는다**(도시에 §2). 순진하게 복제하려는
   쪽은 당연히 cwd의 `package.json`을 읽으려 하고, 그 순간 **전역 실행 프로세스(npx MCP 서버, CLI
   도구)를 상속된 cwd로 오분류**하는 함정에 빠진다. ntop의 정확성은 이 함정을 **의도적으로 피한**
   결과이며, 같은 통찰 없이는 재현하기 어렵다. (그래서 spec은 fs 기반 탐지 제안 자체를 금지한다.)

3. **확장 구조가 해자를 시간에 따라 키운다.**
   새 런타임/프레임워크는 **`Rule` 1개 + enum variant 1개**만 추가하면 되고 다른 코드 변경이
   불필요하다(도시에 §2). 즉 기여가 쌓일수록 사전이 **복리로** 커진다 — 후발 복제자는 매번 격차를
   따라잡아야 하는 움직이는 목표를 상대하게 된다.

**결론:** ntop의 해자는 "단일 기능"이 아니라 **① 경험이 응축된 데이터 사전 × ② 비자명한 탐지 설계
원칙 × ③ 복리로 커지는 확장 구조**의 결합이다. 이는 범용 모니터가 분류 계층 없이 흉내 낼 수 없고,
프로세스 매니저가 발견·분류 범위를 넓히지 않고는 도달할 수 없는 위치다.
